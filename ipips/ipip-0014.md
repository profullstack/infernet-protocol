## Abstract

Codifies five cross-cutting primitives every Infernet API endpoint and
every workload class MUST honor: idempotency keys on state-mutating
endpoints, weight-fingerprint scheduling, automatic job reassignment
on heartbeat staleness, per-provider circuit breakers, and Lamport
sequence numbers on cross-provider job events. These are the
distributed-systems safety nets that prior IPIPs implicitly assumed
but did not formalize. With this IPIP they become contract terms
referenced (not redefined) by every future endpoint.

## Motivation

Standard distributed-systems theory says that in an asynchronous
network, a sender cannot distinguish a lost message from a
delayed one — so every state-mutating receiver must be safe to call
more than once. Independent machines fail independently, so any
operation that hands work to a peer must have a recovery path when
the peer goes silent. Replicated state diverges if not reconciled,
so consumers must be able to detect when two peers' "copy" of the
same logical thing isn't actually the same.

Our existing IPIPs (0009-0013) define new endpoints, new workload
classes, new queue semantics — but each silently assumes its own
flavor of "messages won't get duplicated", "providers won't go
silent", "the model name uniquely identifies the weights." The
assumptions are wrong individually and re-implementing the safety
net per-endpoint duplicates work and creates inconsistent guarantees.

Five concrete bugs we either have today or will have once Class B/B.5/C
ship:

1. **Double-completion of jobs.** A daemon completes a job, its
   `POST /complete` succeeds at the server but the response is
   dropped. Daemon retries; server re-runs the post-complete logic
   (payment settlement, reputation update). No idempotency guard.
2. **Same-name, different-weights routing.** Two providers both
   advertise `qwen2.5:7b`. One has the official weights, one has
   a fine-tune republished under the same tag. The same prompt
   gets non-deterministic outputs depending on routing. Eval and
   RLHF use cases require reproducibility this design can't provide.
3. **Orphaned in-progress jobs.** A provider claims a job, goes
   silent (kernel panic, network partition). The scheduler holds
   the job at `in_progress` forever. No automatic reassignment.
4. **One bad provider poisons retries.** A flaky provider keeps
   getting routed to, burns through chunk retries (IPIP-0013), every
   downstream batch job pays the timeout cost. No per-provider
   circuit breaker.
5. **Cross-provider event interleaving.** Once Class B.5 and Class C
   ship, events from N providers per logical job land at the control
   plane in arbitrary wall-clock order. Reconstructing causal order
   ("peer B couldn't have started before peer A finished") needs a
   logical clock. Adding it later means migrating millions of rows.

## Specification

### 1. Idempotency keys

Every state-mutating endpoint MUST accept an `X-Idempotency-Key`
header carrying a client-generated unique token (UUID v4
recommended; any opaque string ≤128 bytes is acceptable).

```
POST /api/v1/node/jobs/{id}/complete
X-Idempotency-Key: 7f3c2a4b-...-deadbeef
```

Server behavior:
- On first receipt, process normally and persist
  `(idempotency_key, endpoint, request_hash, response_body, response_status, created_at)`
  in the `idempotency_records` table.
- On any subsequent request with the same key + endpoint:
  - If `request_hash` matches → return the cached response verbatim
    (same status, same body). Do NOT re-execute side effects.
  - If `request_hash` differs → return `409 Conflict` with
    `{"error":"idempotency_key_reuse_with_different_payload"}`.
- Records expire after 24 hours.

Endpoints that MUST honor this header (initial set):

| Endpoint | Side effect that MUST be exactly-once |
|---|---|
| `POST /api/v1/node/jobs/{id}/complete` | Payment settlement, reputation update |
| `POST /api/v1/node/jobs/{id}/events` | Token event persistence (per-batch) |
| `POST /api/v1/node/register` | Provider/aggregator/client row upsert |
| `POST /api/v1/user/nodes/{pubkey}/commands` | Queued command insertion |
| `POST /api/v1/jobs/batch` (IPIP-0013) | Batch job + chunk row creation |
| `POST /api/payments/webhook` | Payment ledger update |

Endpoints that read state (`GET ...`) are inherently idempotent and
do not require the header.

Schema:

```sql
CREATE TABLE idempotency_records (
  key             text NOT NULL,
  endpoint        text NOT NULL,
  request_hash    text NOT NULL,    -- sha256(method + path + sorted-headers + body)
  response_status int  NOT NULL,
  response_body   jsonb NOT NULL,
  created_at      timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (key, endpoint)
);
CREATE INDEX idx_idempotency_records_created
  ON idempotency_records (created_at);
```

A periodic janitor deletes records where `created_at < now() - interval '24 hours'`.

### 2. Model weight fingerprinting

Providers' heartbeats MUST advertise served models with a fingerprint
identifying the exact weights, not just the human-readable name.

Heartbeat schema additions in `specs.served_models[]`:

```json
{
  "served_models": [
    {
      "name": "qwen2.5:7b",
      "fingerprint": "sha256:b9c4f7...",   // sha256 of the weights manifest
      "quantization": "Q4_K_M",
      "context_length": 32768
    }
  ]
}
```

Fingerprint computation:
- For Ollama-served models, the fingerprint is the SHA-256 of the
  Modelfile manifest blob (Ollama exposes this via `ollama show
  --modelfile`).
- For vLLM-served models, the fingerprint is SHA-256 of the
  `config.json` + sorted `*.safetensors` file digest list as
  reported by HuggingFace Hub.
- For unknown engines, the fingerprint MAY be omitted (treated as
  `null` and excluded from determinism-required routing).

Scheduler routing rules:
- Default routing matches on `name` only (existing behavior).
- Jobs that declare `options.require_deterministic = true` MUST
  match providers whose `fingerprint` matches an explicitly trusted
  fingerprint registry (`models_canonical` table). Mismatches are
  excluded from candidate routing.
- Mismatches across providers serving the same `name` are surfaced
  in the dashboard as a warning.

### 3. Heartbeat-stale job reassignment

A scheduler janitor MUST run on a fixed cadence (default: 30 s) and
reassign jobs whose assigned provider has gone silent.

Behavior:

```
For each job J with status='in_progress':
  Let P = assigned_provider
  Let H = P.last_seen
  If now() - H > stale_threshold (default 90 s):
    1. Set J.status = 'queued'
    2. Increment J.reassign_count
    3. Clear J.assigned_provider (re-eligible for routing)
    4. If J.reassign_count > max_reassigns (default 3):
       Set J.status = 'failed' with reason='exhausted_reassigns'
    5. Emit a job_events row of type='reassigned'
```

`stale_threshold` MUST be greater than the heartbeat interval
(default 60 s) plus a network buffer. `max_reassigns` is per-job and
prevents pathological reassignment loops.

For batch chunks (IPIP-0013), the janitor operates at the
chunk-row level using the same logic.

### 4. Per-provider circuit breakers

The scheduler MUST track each provider's recent outcome rate and
exclude flaky providers from routing for a cooldown period.

State per provider, kept in `provider_circuit_state`:

```sql
CREATE TABLE provider_circuit_state (
  provider_id      uuid PRIMARY KEY REFERENCES providers(id),
  state            text NOT NULL CHECK (state IN ('closed','open','half_open')),
  consecutive_fail int  NOT NULL DEFAULT 0,
  opened_at        timestamptz,
  next_probe_at    timestamptz,
  failure_window   jsonb NOT NULL DEFAULT '[]'::jsonb -- ring buffer of last 20 outcomes
);
```

State machine:

```
closed → open      after N consecutive failures (default N=5)
                     OR failure rate ≥ 50% over last 20 jobs
open → half_open   after cooldown (default exponential: 30s, 2m, 8m, 30m, capped 60m)
half_open → closed after one successful job
half_open → open   after one failure
```

Routing rule: providers in state `open` MUST be excluded from job
candidate selection, except a single probe job per cooldown window
(transition to `half_open`).

A "failure" is: heartbeat-stale reassignment, job complete with
`reason='error'`, chunk retry exhaustion, or HTTP 5xx from the
provider's daemon. Cancellations and successful completions count
as successes.

### 5. Lamport sequence on job events

Every row in `job_events` MUST carry a logical clock counter,
maintained per Lamport's two implementation rules.

Schema addition:

```sql
ALTER TABLE job_events ADD COLUMN provider_seq bigint;
ALTER TABLE job_events ADD COLUMN provider_id  uuid REFERENCES providers(id);
CREATE INDEX idx_job_events_lamport
  ON job_events (job_id, provider_id, provider_seq);
```

Provider-side rules — **must implement BOTH IR1 and IR2**:

- **IR1 — local event increment.** The daemon maintains a monotonic
  counter per `(provider_id, job_id)` tuple. Increment between any
  two successive events. Persisted across daemon restarts at
  `~/.config/infernet/job_seq.json`.
- **IR2 — message-receive update.** When the daemon **receives** a
  message from another provider that carries a Lamport timestamp
  `Tm` (e.g. a Petals chain hidden-state message, an OpenDiLoCo
  delta from another worker), it MUST set its local counter to
  `max(local, Tm) + 1` BEFORE recording the receive event. Without
  this rule, cross-process events do not satisfy the Clock
  Condition (`a → b ⟹ C(a) < C(b)`) — Lamport, *Time, Clocks…*
  IR2(b) — and the seq numbers are useless across providers.

Cross-provider events that CARRY a Lamport timestamp (Class B.5
relay messages, Class C training deltas) MUST encode it in the
signed envelope as `lamport_ts`.

Total ordering across providers (when needed):

When two events have equal `provider_seq` values, ties break by
`provider_id` lexicographic comparison. This is the standard total-
order construction (Lamport, p. 561) — pick any total order on
processes, append it as the tiebreaker. The result is a consistent
total order that respects causality.

Distinguishing concurrent from happens-before:

The dashboard, replay tools, and consistency checks MUST distinguish
**concurrent** events (neither happens-before the other) from
**ordered** events. Expose via:

```ts
event_relation(a: Event, b: Event):
  "happens_before"   // C(a) < C(b) AND a's parent set transitively reaches b
  | "happens_after"  // mirror of above
  | "concurrent"     // C(a), C(b) in disjoint causal histories
```

Force-ordering concurrent events by wall-clock loses information
that matters for replay correctness (Class C training: were two
delta updates merged or sequential?) and for anomaly detection.

### 6. State-transition guards

Every endpoint that mutates a state machine (job lifecycle, payment
ledger, link status, etc.) MUST accept and validate an explicit
pre-state:

```
PUT /api/v1/node/jobs/{id}/status
{ "to": "running", "from": "queued" }
```

The server MUST reject the call with `409 Conflict` if the row's
current state ≠ `from`. This prevents out-of-order *different*
messages from corrupting state — idempotency keys (§1) prevent
double-processing of the *same* message but do nothing for the case
where a delayed `started` arrives after `completed` was already
persisted (different keys, both valid in isolation, system silently
regresses).

Endpoints that MUST honor pre-state:
- `PUT /api/v1/node/jobs/{id}/status`
- `POST /api/v1/node/jobs/{id}/complete` (implicitly: pre = "running")
- `POST /api/v1/jobs/batch/{id}/cancel` (pre ∈ {"queued", "running"})
- `POST /api/v1/user/nodes/{pubkey}/commands/{cmd_id}/ack` (pre = "pending")

### 7. Bounded-staleness reads in routing

Scheduler reads of provider state for routing decisions MUST be
performed at a single snapshot timestamp:

```sql
SELECT * FROM providers
WHERE last_seen >= now() - interval '60 seconds'
```

All candidates evaluated against the same temporal cutoff. The
documented contract: "you may be routed to a provider that died up
to 60s ago, never longer." This makes the AP guarantee on routing
explicit and bounded — the upper bound on stale-data damage is
60 seconds of routing to a dead peer plus one round of failure +
reassignment (§3).

### 8. Payment subsystem ACID boundary

All payment-affecting code paths MUST execute inside a Postgres
ACID transaction on the canonical control-plane database. No
payment write may depend on any AP system (DHT records, eventual-
consistency provider state, BullMQ job results, Lamport-ordered
event streams).

Concretely, the following operations MUST be wrapped in a single
transaction:
- Invoice creation
- Webhook receipt → ledger update
- Provider payout settlement on job complete
- Refund on cancel / not_implemented / exhausted_reassigns

This is a review gate. Any future endpoint or worker that touches
`payment_transactions`, `provider_payouts`, or `platform_wallets`
MUST be rejected at review unless it operates inside ACID.

### 9. Hinted-handoff outbox for offline writes

Provider daemons MUST persist outbound state-mutating events to a
local outbox before considering the operation "logged":

```
~/.config/infernet/outbox/<event_id>.json
```

The daemon's normal flow: write to outbox → POST to control plane
→ on 2xx, delete outbox entry. On 4xx, log + delete (irrecoverable).
On 5xx / network error, leave in outbox; retry with exponential
backoff.

Closes the partition-tolerance gap: events generated while the
control plane is unreachable (or while the daemon is between
heartbeats) are durable on disk and replay when connectivity
returns. Standard AP pattern; no new infrastructure.

Events that MUST go through the outbox:
- `POST /api/v1/node/jobs/{id}/complete`
- `POST /api/v1/node/jobs/{id}/events` (token / step batches)
- `POST /api/v1/node/heartbeat` (last_seen / status updates) —
  optional but recommended

### 10. External causal tokens for cross-channel ordering

When a workflow spans two independent message channels — e.g. a
payment webhook from CoinPayPortal and a job-complete from a
provider daemon — Lamport timestamps alone CANNOT capture the
causal relationship. This is Lamport's own external-observer
anomaly (*Time, Clocks…*, Anomalous Behavior section): events
caused via a channel outside our system can arrive in apparent-
contradiction order.

The fix is to require an **explicit external causal token** that
both channels carry. For payment+complete: the `payment_intent_id`
issued at invoice creation flows into both
- the CoinPayPortal webhook payload (preserved through the gateway), and
- the job's `payment_intent_id` column (set when the daemon
  acknowledges the invoice).

The settlement worker MUST refuse to credit a payout until BOTH
the webhook AND the matching job-complete have arrived for the
same `payment_intent_id`. The token is the causal anchor; arrival
order at the control plane is irrelevant.

The same pattern applies to any future cross-channel workflow
(SMS verification → user action; external eval signal → training
run). Each gets an explicit token threaded through both paths.

## Rationale

**Why exactly these five primitives and not others?** Each maps
directly to a class of bug we either have today or will have once
the in-flight IPIPs ship. Other possible primitives (vector clocks,
quorum reads, total-order broadcast) are more powerful but solve
problems we don't have at our scale; adding them now is premature
abstraction.

**Why MUST and not SHOULD?** Each of these has a silent failure
mode where the system *appears* to work but is actually wrong (double-
charged payments, non-deterministic eval results, jobs orphaned at
in-progress, bad providers earning reputation). Optional safety
nets get skipped under deadline pressure. MUST is the right strength.

**Why a single dedup table for idempotency, not per-endpoint?**
24-hour TTL means the table is bounded by recent traffic, not
historical. Single table is simpler to GC, easier to reason about,
and gives uniform behavior across endpoints. The
`(key, endpoint)` composite primary key prevents cross-endpoint
collisions if a client reuses a key by accident.

**Why per-provider Lamport rather than global?** Global Lamport
across N providers requires every event to carry every provider's
last-seen counter (vector clock), which scales poorly. Per-provider
is enough to answer "in what order did THIS provider's events
happen?" — and that's the only causal-order question the dashboard
or the control plane actually asks.

**Why fingerprint quality varies by engine?** Ollama exposes a
clean Modelfile manifest hash; vLLM does not have a single
manifest, so we hash a stable composite. Engines that don't
provide either get `null` and are excluded from determinism-
required workloads — operators wanting that traffic class run
Ollama or vLLM.

## Backwards compatibility

All five primitives are additive to existing wire formats:

- `X-Idempotency-Key` is optional; absence preserves current
  behavior. Endpoints in the table above SHOULD warn-log when the
  header is missing on a state-mutating call so we can phase clients
  toward compliance.
- `served_models[].fingerprint` is optional; missing fingerprints
  exclude the model from determinism-required routing only.
- Heartbeat-stale janitor adds new behavior but only fires on jobs
  that were already broken (silent provider). No existing
  successful flow changes.
- Circuit-breaker exclusion changes scheduler behavior. Until the
  state machine has accumulated enough history (cold start), every
  provider is treated as `closed` (the existing behavior).
- `provider_seq` defaults `null` for old `job_events` rows; new
  rows populate it. Dashboard timeline gracefully degrades for
  null-seq rows.

## Test plan

- **Unit:**
  - Idempotency middleware: same key + same body → cached; same key
    + different body → 409; expired record → re-execute; concurrent
    requests with the same key → only one executes.
  - Fingerprint helpers per engine: known-input → known-output.
  - Janitor reassignment: stale provider → job back to queued; max
    reassigns → failed.
  - Circuit-breaker state machine: 5 consecutive fails → open;
    cooldown elapses → half_open; success → closed; failure →
    re-open.
  - Lamport monotonicity: counter never decreases across daemon
    restart.

- **Integration:**
  - End-to-end batch job (IPIP-0013) where one chunk completion is
    delivered twice — exactly one payment settlement.
  - Two providers advertise the same model name with different
    fingerprints; deterministic-required job routes to the
    canonical-fingerprint provider only.
  - Provider goes silent mid-job; janitor reassigns; second
    provider completes; payment lands on second provider.
  - Flaky provider hits 5 consecutive failures; circuit opens;
    next 100 jobs route around it; cooldown probe re-tests.

- **Regression:**
  - Existing chat / register / heartbeat flows produce the same
    observable behavior with or without `X-Idempotency-Key`.
  - Old daemons (no provider_seq) continue to register and
    heartbeat without errors.

## Security considerations

- **Idempotency table abuse.** A malicious client could flood the
  table with unique keys to exhaust storage. Mitigation: per-user
  rate limit on key insertions (existing API rate-limit infrastructure
  applies); 24h TTL bounds steady-state size.
- **Fingerprint spoofing.** A provider could falsely advertise a
  trusted fingerprint to attract determinism-required jobs. Mitigation:
  the control plane periodically issues canary prompts whose
  outputs are stable for the canonical weights; mismatches expel
  the provider from the trusted set and drop reputation.
- **Circuit-breaker DoS.** An attacker could submit failing jobs
  to a competitor's provider to trip its breaker. Mitigation: only
  count failures from jobs the provider was scheduled by us, not
  user-direct calls; per-user failure attribution prevents
  amplification.
- **Lamport-clock manipulation.** A malicious provider could lie
  about its sequence numbers. Mitigation: this is a logical clock,
  not security state — it doesn't authorize anything; downstream
  consumers detect impossible orderings and surface as anomalies.
  Reputation handles the malicious case via the existing channel.

## Privacy considerations

None of these primitives change what data flows. Idempotency keys
are client-chosen and opaque; fingerprints describe public model
weights; reassignment, circuit state, and Lamport counters are all
operational metadata never tied to user content.

## Open questions

- **24-hour idempotency window — too long, too short?** Most clients
  retry within seconds, not hours. Shorter (e.g. 1 h) would bound
  storage tighter but risk legitimate retries falling outside the
  window. Punt to operator config?
- **Per-engine fingerprint registry.** Who curates `models_canonical`?
  We bootstrap with HuggingFace Hub manifest hashes for known
  open-weight families; long-term, providers can submit fingerprints
  for community review.
- **Cooldown schedule for circuit-breaker.** Exponential backoff
  with the proposed limits is a guess; production traffic will
  reveal whether 30s → 60m is the right curve.
- **Lamport seq persistence on identity rotation.** If an operator
  regenerates their Nostr key (new `provider_id`), the counter
  resets. Is this a problem? Probably not — it's a new identity
  from the consumer's perspective, but worth confirming.
