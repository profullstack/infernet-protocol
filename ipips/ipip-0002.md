## Abstract

Define a minimal chat layer for Infernet operators — encrypted direct
messages between two Nostr pubkeys, plus a small curated set of public
**default rooms** every node auto-joins on first run (Discord-server
style). All of it rides the public Nostr relay network we already use
for identity. No central server, no new database, no custom transport.
Think IRC `/msg` and IRC `#channel`, not Slack.

## Motivation

Operators have no way to coordinate today. There is no shared chat, no
inbox, no "hey are you up?" signal. Every existing communication path
(Discord, Telegram, Signal) requires onboarding to a separate identity
and trusting a separate provider.

Meanwhile, every Infernet node already holds a BIP-340 / secp256k1
Nostr keypair (shipped in IPIP-0001 frozen surfaces). Encrypted Nostr
DMs work today on hundreds of public relays. Adding chat is mostly
plumbing — one CLI command, one event handler, one filter.

This is also a forcing function for two pieces of decentralization
infrastructure we want anyway:

- A working **Nostr relay client** in the daemon, paving the way for
  IPIP-0001's deferred "P2P transport via Nostr relays" item.
- A **per-operator inbox model** — useful later for things like job
  bids, dispute escalation, and out-of-band peering negotiations.

> "person-to-person like irc, but much more basic. that would be sick"

That is the brief.

## Specification

### Identity

The same Nostr keypair the node uses for the signed-request envelope
(IPIP-0001) is the chat identity. No second key, no key rotation, no
extra registration.

A peer is identified by their 32-byte hex pubkey. Operators discover
each other's pubkeys via the public dashboard (`infernetprotocol.com`)
or by sharing them out-of-band.

### Wire format — direct messages

DMs are standard Nostr events using **NIP-44 v2 encryption** (modern
AEAD, replaces the deprecated NIP-04). Specifically:

- Kind `14` (NIP-17 / NIP-44 sealed direct message).
- Encrypted `content`, recipient in `["p", <hex pubkey>]` tag, sender
  in event `pubkey`.
- Tag `["client", "infernet:talk:v1"]` MUST be present so receivers
  can scope their filters to chat-from-other-operators rather than
  the entire Nostr firehose.
- Optional `["a", "infernet"]` application tag for cross-app interop.

### Wire format — rooms

Rooms are **plain-text** Nostr text notes (kind `1`), tagged with the
room topic. No encryption — rooms are public by design.

- Kind `1` text note.
- Topic tag `["t", "infernet:<room>"]` — e.g. `["t",
  "infernet:general"]`. Receiver MUST use this tag for subscription
  filters.
- Discriminator tag `["client", "infernet:room:v1"]` so non-Infernet
  Nostr text notes that happen to use a colliding topic don't appear
  in operator clients.
- `content` is UTF-8 markdown — minimal subset (paragraphs, inline
  code, fenced code blocks, links). Implementations MAY render or
  show raw.

Mention syntax for in-room replies: `@<8-hex-prefix>` is resolved
against the receiver's local alias map; an additional `["p", <full
hex>]` tag SHOULD be set for any pubkey mentioned, so clients can
highlight the mention.

### Default rooms

Every node auto-joins this set on first run (Discord parallel — when
you join a new server, you start subscribed to its baseline channels):

| Room        | Purpose                                                      | Auto-join |
| ----------- | ------------------------------------------------------------ | --------- |
| `#general`  | Anything goes. Default landing channel.                      | yes       |
| `#support`  | Help with running a node, troubleshooting, install issues.   | yes       |
| `#news`     | Protocol announcements. Read-mostly. Low traffic.            | yes       |
| `#offtopic` | Non-Infernet chat. Keeps the signal-to-noise of #general up. | yes       |
| `#dev`      | Protocol implementation discussion.                          | opt-in    |
| `#jobs`     | Marketplace coordination, "I have spare capacity" posts.     | opt-in    |

The list is intentionally short. Operators can join arbitrary topic
tags — `infernet talk #my-team` works and creates a de-facto room
just by virtue of two people subscribing. The "default" set is the
curated baseline; the larger room namespace is open.

Adding a new room to the defaults is a pull request that updates this
table. Removing one requires an IPIP because nodes pinned to an older
default set may keep listening on a removed room indefinitely.

Bumping the `infernet:room:vN` version (or `infernet:talk:vN`) is the
breaking-change escape hatch. Implementations MUST drop events whose
tag version they do not understand.

### Relays

The daemon connects to the same relay set used for identity broadcast
(today: `wss://relay.damus.io`, `wss://relay.nostr.info` —
configurable via `nostr.relays` in the CLI config). It MUST tolerate:

- Relays that go down (reconnect with backoff).
- Relays that drop events silently (publish to ≥ 2 distinct relays per
  message; consider a message delivered if any relay accepts it).
- Relays that censor by content (out of scope — operator picks their
  own relay set).

There is no Infernet-operated relay in v1 of this proposal. Run-your-
own relay is a follow-up that does not block this spec.

### CLI surface

A new top-level command, distinct from the inference-related
`infernet chat`:

```
# Direct messages (encrypted, 1-to-1)
infernet talk <pubkey>                  # interactive REPL
infernet talk <pubkey> "<message>"      # one-shot send

# Rooms (public, plain-text)
infernet talk #general                  # interactive REPL in #general
infernet talk #support "anyone running ROCm 6.0?"
infernet talk --rooms                   # list joined rooms
infernet talk --join #dev               # subscribe to a non-default room
infernet talk --leave #offtopic         # unsubscribe

# Inbox / listening
infernet talk --listen                  # tail all DMs + joined rooms
infernet talk --listen #general         # tail just one room
infernet talk --listen --since 1h       # backlog window then live
infernet talk --pubkey                  # print my chat pubkey
```

Message format on stdout (when listening):

```
[20:31:04] #general    <8f3a..b2d1> sup
[20:31:11] #support    <c4e9..a800> rocm worked once i set HSA_OVERRIDE_GFX_VERSION
[20:31:48] @8f3a..b2d1 → me           hey, are you the one running 4x 3090s?
```

The first 8 hex chars of each pubkey are shown by default; `--full`
prints the entire pubkey. Operators MAY install a local pubkey →
nickname alias file (e.g. `~/.config/infernet/talk-aliases.json`); the
alias resolution is purely client-side and not standardized here.

Auto-join state is persisted at
`~/.config/infernet/talk-rooms.json` and seeded with the default room
set on first `infernet init` (or first `infernet talk` invocation,
whichever comes first).

### Message lifecycle

1. Sender encrypts `content` for recipient using NIP-44.
2. Sender publishes the kind-14 event to all configured relays.
3. Each relay forwards to subscribers matching
   `{ "kinds": [14], "#p": [<my pubkey>] }`.
4. Receiver decrypts, displays, and MAY emit a kind-14 receipt back
   tagged `["e", <original event id>, "ack"]`.

Receipts are advisory — the absence of a receipt does not mean the
message was not delivered.

### Out of scope for v1 of this IPIP

- **Moderation tooling for rooms.** No `/kick`, `/ban`, `/mute` —
  rooms are public, anyone can post, anyone can mute locally via the
  block list. If a room becomes unusable, the community migrates to
  `#general-2` etc. Add real moderation in a later IPIP if needed.
- **Threading / replies-as-tree.** Flat timestamp order only.
  Mentions (`@<prefix>`) are supported; nested threads are not.
- **File / image attachments.** Plain UTF-8 text only.
- **Read receipts beyond the optional ack above.**
- **Retention guarantees.** Relays may drop history at any time;
  operators wanting durable history run their own relay.
- **Voice / video.** Out of scope forever for this proposal.
- **Inbox UI on the dashboard.** CLI-only in v1; web UI is a separate
  IPIP if/when it's worth doing.

## Rationale

**Why Nostr instead of a custom transport?** Identity, signing, and
relay infrastructure all already exist. Reinventing any of those
extends scope by months for zero gain.

**Why NIP-44 (kind 14) and not NIP-04 (kind 4)?** NIP-04 leaks message
length and uses crypto primitives that have known weaknesses. NIP-44
is the current Nostr standard; clients (Damus, Amethyst, nostr-tools)
have shipped it for years.

**Why interactive REPL instead of just one-shot?** IRC-shaped UX
matters; a back-and-forth conversation with two `infernet talk` calls
per turn is annoying. Listen + send in the same TTY is the IRC
experience.

**Why CLI-only and not a dashboard tab?** v1 of this proposal is for
operator coordination. Operators live in terminals. A web inbox is
strictly more work and adds a session-management story we don't need
yet.

**Why default rooms instead of just DMs?** Discord's onboarding
research is right: brand-new operators with no peers' pubkeys feel
like the chat is empty until somebody else shows up. Auto-joining
`#general` puts them in front of the existing community immediately,
the same way a fresh Discord server greets you with a few populated
channels. Without that, the chat layer feels broken on day one.

**Why these four defaults and not a bigger set?** The bigger the
default list, the more notification fatigue and signal-to-noise
problems on each individual channel. Four is the smallest set that
covers (a) general chat, (b) help, (c) announcements, (d) the relief
valve so #general doesn't fill with off-topic posts. Add `#dev` and
`#jobs` as opt-in because they have a self-selecting audience.

**Why topic tags (kind 1 + `["t", "infernet:..."]`) and not NIP-28
(kind 40/41/42)?** NIP-28 requires a creator event (kind 40) whose
event ID becomes the channel identifier. That introduces a "who
creates the official #general?" trust problem (whoever's event ID we
pin is the de-facto owner of the room). Topic tags side-step it
entirely: there is no owner, the protocol is the owner, and old
clients with stale tag versions degrade gracefully.

## Backwards compatibility

None — this is a new, additive feature. A node that does not implement
IPIP-0002 simply ignores incoming kind-14 events tagged for it.

The CLI's `infernet chat` command (local model inference) is unchanged.
The new command is `infernet talk` — distinct verb, distinct file
(`apps/cli/commands/talk.js`).

## Reference implementation

Not yet implemented. Suggested landing path:

1. `packages/nostr-client/` — relay pool client (connect, subscribe,
   publish, reconnect-with-backoff). Reuses `@infernetprotocol/auth`
   for the Nostr keypair.
2. `packages/talk/` — kind-14 / NIP-44 send + receive helpers, alias
   resolution, message formatting.
3. `apps/cli/commands/talk.js` — CLI entrypoint (interactive REPL +
   one-shot + `--listen`).
4. Memory: a 1-hour SQLite-backed message cache so reconnecting to
   relays surfaces the recent backlog. Optional; relays handle live
   delivery already.

Each step is its own PR, all referencing IPIP-0002.

## Test plan

1. `infernet talk --pubkey` prints the same hex string as the
   `register`-time pubkey. (Identity reuse.)
2. Two nodes A and B, both running `infernet talk --listen`, configured
   with overlapping relay sets. A runs `infernet talk <B-pubkey>
   "hello"` — B's listener prints the message within 2s.
3. Kill the relay A used. A retries on the next relay in its set.
   Message still arrives at B (publish to ≥ 2 relays per send).
4. NIP-44 round-trip: DM messages are unreadable on disk in a relay
   log, only readable by the recipient. Validated by inspecting raw
   event content.
5. Filter test: a Nostr DM from a non-Infernet sender (no
   `["client","infernet:talk:v1"]` tag) is dropped silently when the
   user runs `infernet talk --listen --strict`. Default mode shows it
   with a `[non-infernet]` prefix.
6. Default-room auto-join: `infernet init` on a fresh box leaves
   `~/.config/infernet/talk-rooms.json` populated with the four
   default rooms. `infernet talk --rooms` prints them.
7. Cross-room delivery: A posts in `#general`, B (subscribed to
   `#general`) receives it within 2s. C (subscribed only to `#dev`)
   does NOT receive it.
8. Auto-join idempotency: a node that has been offline for a week
   starts up and is still subscribed to the same default rooms. No
   resubscribe spam to relays.
9. Room name collisions: a non-Infernet Nostr note with `["t",
   "general"]` (no `infernet:` prefix) does NOT appear in
   `infernet talk #general`.

## Security considerations

- **End-to-end encryption is mandatory.** Plain-text mode (kind 1
  Nostr notes) is forbidden by this IPIP — leaks to every relay
  operator.
- **Replay**: NIP-44 events carry timestamps; clients SHOULD reject
  events older than 24 hours by default (configurable). Replays of
  recent events are observable but harmless given encryption.
- **Spam / harassment**: any pubkey can DM any other pubkey on a
  public Nostr relay. Implementations MUST support a local block list
  (`~/.config/infernet/talk-blocked.json`). Cross-operator block
  sharing is out of scope.
- **Pubkey reuse with the signed-request envelope is intentional** —
  it gives operators a single identity. The trade-off: deanonymizing a
  chat partner deanonymizes their compute identity. See "Privacy
  considerations".

## Privacy considerations

- **Relays see metadata.** Every public Nostr relay this node connects
  to learns that this pubkey is online and is talking to that pubkey,
  even though the content is encrypted. Operators concerned about
  network-level privacy MUST route their relay traffic through Tor or
  a SOCKS5 proxy (orthogonal to this IPIP; tracked under IPIP-0001's
  deferred "Tor / SOCKS5 outbound").
- **Pubkey linkability.** A node's chat pubkey is the same as its
  control-plane pubkey. A future IPIP can specify a separate chat
  pubkey if linkability becomes a problem; the wire format would
  trivially extend (use a different pubkey in the `pubkey` field of
  the kind-14 event).
- **Public dashboard.** If the dashboard ever publishes operator chat
  pubkeys (e.g. as a "ping me" button), it MUST display a clear
  warning that this links the chat identity to the listed compute
  identity.

## Open questions

- **Should we ship a default block list of known-spam pubkeys?** Lean
  no for v1 — that's a moderation policy, not a protocol detail.
- **Do we want a dashboard "ping me" button** that opens
  `infernet talk <pubkey>` from the operator's local install?
  Probably yes; deferred to a follow-up so it doesn't block this IPIP.
- **NIP-17 sealing vs. plain NIP-44.** NIP-17 wraps NIP-44 in an extra
  "seal" event to hide the recipient from intermediate observers. Add
  for v1 or defer? Defer — adds complexity, clients vary in support.
- **Should the daemon `start` command auto-listen** for incoming chat
  and surface unread counts via the IPC `stats` snapshot? Probably
  yes; tracked separately as the "chat inbox in stats" task.
- **`#announcements` vs `#news`.** Bikeshed: `#news` is the proposed
  default; `#announcements` is the conventional Discord name. Pick
  one before this IPIP goes Active so we don't rename.
- **Read-only `#news`.** Should non-maintainer pubkeys be soft-blocked
  from posting in `#news`? There is no enforceable answer at the
  Nostr layer — we can hide their posts client-side, but anyone can
  still publish. Probably accept the limitation and document it.
