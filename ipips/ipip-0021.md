## Abstract

Establishes Protocol Buffers as the canonical IDL for every
peer-to-peer wire contract in Infernet. Adds a `protocol/` directory
at the repo root holding versioned `.proto` files for handshake,
peer discovery, DHT, pubsub, distributed compute, payment references,
and remote method invocation (RMI). Each protocol package is
versioned (e.g. `infernet.handshake.v1`), backed by a markdown spec
with Mermaid diagrams under `protocol/docs/`, and codegen-ready for
multi-language SDKs (JS, Rust, Go, Python).

## Motivation

The current control-plane API has an OpenAPI 3.1 spec
(`packages/api-schema/openapi.yaml`) covering HTTP/REST. That works
for client → control-plane traffic. **It does not cover peer-to-peer
traffic** — daemon ↔ daemon dials, DHT records, gossip envelopes,
RMI calls between providers. As we ship IPIP-0016 (DHT), IPIP-0018
(causal broadcast), and Class B.5 / C workloads (IPIP-0010 to 0012),
peers will be exchanging structured messages directly without going
through HTTP.

Three pain points without a formal IDL layer:

1. **No source of truth for peer wire formats.** Today's daemon
   encodes Nostr-signed envelopes ad-hoc in JS; a future Rust or Go
   client can't depend on those shapes.
2. **No version negotiation.** Two daemons on different protocol
   versions can't gracefully detect the mismatch — they discover it
   by silent decode failure.
3. **No multi-language SDKs.** Operators wanting to run an Infernet
   provider in Rust (for the GPU layer) or Python (for ML tooling
   integration) currently have to reimplement the wire format.
   Codegen from a shared `.proto` source removes that.

### 2. IDL choice — Protocol Buffers v3

Why protobuf:

- Multi-language codegen battle-tested at scale (Google, gRPC, every
  major distributed-systems shop)
- libp2p uses protobuf natively for its handshake / kademlia / gossipsub
  wire formats — we align with the transport
- Forward-compatible by design (unknown fields preserved on decode)
- Compact wire footprint (~3-4× smaller than JSON for our shapes)
- Stable ABI across versions when authors follow the rules in §5

We considered Cap'n Proto (faster decode, less library support),
Thrift (legacy), FlatBuffers (zero-copy but harder for our use cases),
and WIT (WebAssembly Component Model — too early). Protobuf wins on
ecosystem maturity for v1.

### 3. Protocol packages — initial set

| Package | Purpose |
|---|---|
| `infernet.handshake.v1` | First-contact protocol — peer ID exchange, version negotiation, capability advertising |
| `infernet.peer.v1` | Discovery — find peers by namespace + protocol filter |
| `infernet.dht.v1` | Kademlia-style key-value lookup (per IPIP-0016) |
| `infernet.pubsub.v1` | Topic-based gossip (per IPIP-0018 causal broadcast) |
| `infernet.compute.v1` | Job submission to worker peers, status streaming |
| `infernet.payment.v1` | Payment-intent verification (NOT ledger replication) |
| `infernet.rmi.v1` | Remote method invocation on stateful objects |

Every package's wire types MUST cite the relevant IPIP in the proto
file's leading comment so cross-references survive code archaeology.

### 4. Remote Method Invocation (RMI)

RMI is the object-oriented sibling of plain RPC. Where RPC calls a
free-standing function, RMI calls a method on a specific object
instance identified by an `ObjectRef`. The object lives on a
particular peer and carries state across calls.

#### 4a. Why RMI in addition to plain RPC

Several Infernet workflows are inherently stateful and benefit from
object-shaped APIs:

- **Long-running compute jobs.** A `ComputeJob` object exposes
  `start()`, `getStatus()`, `cancel()`, `streamLogs()` — all
  targeting the same job instance.
- **Petals chain peer sessions** (IPIP-0012). A `PeerSession` object
  per active chain holds the shard + cache state.
- **Class C training coordinator role** (IPIP-0011 / IPIP-0020 §4b).
  The coordinator is an object whose ownership migrates on
  re-election.

These map naturally to ObjectRef-addressed methods rather than
free-standing RPC calls.

#### 4b. ObjectRef

```
ObjectRef { object_id, type_name, owner_peer_id }
```

`type_name` MUST include a version, e.g.
`infernet.compute.ComputeJob.v1`. Bumping the type version is a
breaking change exactly like bumping a protobuf package version.

#### 4c. Object migration

When ownership moves (e.g. a Petals shard re-balanced to another
peer), the old owner MUST respond with `OBJECT_MIGRATED` and the
new owner's `ObjectRef` populated. Clients update their local
proxy and retry against the new owner.

#### 4d. Object registry

Maps `object_id → owner_peer_id`. v1 implementation is DHT-backed
(IPIP-0016): the key is `sha256("rmi:" + object_id)` and the
value is a signed envelope carrying the current `ObjectRef` + a
TTL. Object owners republish on a 5-minute cadence.

### 5. Versioning rules

For every protobuf package and RMI type:

- **Never reuse field numbers.** Once a number is assigned, it's
  reserved forever.
- **Never change the meaning of an existing field.** Add a new
  field with a new number instead.
- **Adding optional fields is safe** — old clients ignore them.
- **Removing fields is breaking** — bump the package to v2.
- **Renaming wire-visible names is breaking** — bump the package.
- **The handshake MUST negotiate before any other protocol opens.**
  Two peers find their common protocol-version set during handshake
  and use only that overlap thereafter.

When a package goes to v2, both `v1` and `v2` packages live in the
tree simultaneously; `v1` stays maintained until a deprecation
window closes (default: 6 months from v2 release).

### 6. Generated SDKs

`protocol/generated/` holds language-specific output. CI (the
codegen step is reference-implementation, not specified here)
regenerates all four targets and fails if the committed output
drifts from the source `.proto` files.

Each SDK MUST expose:

- Message encoders / decoders for every type in every package
- Client stubs for every `service` definition
- Server interface types
- Wire-compatibility test fixtures
- Version metadata (the protobuf package version + git commit)

### 7. Wire compatibility test fixtures

`protocol/tests/fixtures/` holds binary-encoded sample messages
generated by one SDK and decoded by every other. CI runs the
cross-decode suite to catch divergence early.

Naming convention: `<package>-<message>-v<n>.bin`, e.g.
`handshake-request-v1.bin`, `compute-jobstatus-v1.bin`.

### 8. Security envelope

Every wire message MUST be carried inside a Nostr-signed envelope
(per IPIP-0003 / IPIP-0014). The protobuf types in this IPIP
describe the *payload* shape; the signature wrapping is uniform
across protocols. Adapters MUST verify the signature before
decoding the payload.

Specifically forbidden:

- Decoding-then-verifying (allows malformed-payload exploits to
  reach decode logic with insufficient validation)
- Trusting `peer_id` claimed inside a payload without verifying
  the signing pubkey matches
- Accepting envelopes with timestamps outside the 60-second replay
  window (IPIP-0014 §1)

### 9. CLI surface

```
infernet protocol validate         # check all .proto files parse + lint
infernet protocol generate         # regenerate SDKs into protocol/generated/
infernet protocol test             # conformance + wire-compatibility tests
infernet protocol diff <v1> <v2>   # report breaking changes between versions
infernet protocol docs             # rebuild markdown docs index
```

Reference implementation lands as a subcommand of the existing
`infernet` CLI (per IPIP-0009 surface).

## Rationale

**Why not just keep using JSON over HTTP?** HTTP/JSON is fine for
the control-plane API (operator ↔ central web app); it's the wrong
fit for peer-to-peer where (a) the wire footprint matters at gossip
volume, (b) version mismatch needs to be detected programmatically,
and (c) we want to support languages without first-class JSON
schemas (Rust / Go especially).

**Why a separate IPIP from the existing OpenAPI?** They cover
different layers. OpenAPI 3.1 (`packages/api-schema`) describes
HTTP endpoints on `infernetprotocol.com`; the protobuf IDL describes
peer-to-peer wire formats. Both stay maintained; they do not overlap.

**Why is `payment.v1` a "verify only" protocol, not a full ledger?**
Per IPIP-0014 §8, the payment subsystem is strict CP and never
gets P2P-replicated. The proto package exists so peers can *verify*
a payment intent via signed reference, not so they can settle one
peer-to-peer. Future decentralized escrow would be a separate IPIP
with a heavier consensus protocol (per IPIP-0020 §4c).

**Why include RMI as a peer to handshake / dht / etc. instead of
folding it under compute?** RMI is workload-agnostic — it'll be
used by `ComputeJob`, `PeerSession`, the Class C coordinator, and
likely future objects we haven't named yet. Keeping it as its own
package makes the abstraction explicit.

## Backwards compatibility

Fully additive. No existing endpoint, daemon, or API contract
changes. The `protocol/` tree is new; `packages/api-schema/` (the
HTTP OpenAPI spec) stays untouched.

The first peer that actually speaks `handshake.v1` will be a new
implementation; existing daemons today don't open peer-to-peer
streams (they outbound-poll the control plane), so there's no
back-compat burden until we ship the first peer-side
implementation.

## Reference implementation

This IPIP lands the spec + the proto files + the architecture doc.
Subsequent commits land per-protocol docs, codegen tooling, and
the first peer-side implementation. Phasing per the PRD:

- **Milestone 1 (this commit)**: Protocol skeleton + this IPIP +
  architecture doc + 7 `.proto` files + per-protocol docs.
- **Milestone 2**: Codegen pipeline → `protocol/generated/js/` first.
- **Milestone 3**: Wire-compatibility test suite.
- **Milestone 4**: Rust / Go / Python codegen + cross-SDK conformance.
- **Milestone 5**: `infernet protocol diff` + breaking-change
  detection in CI.

## Test plan

- **Schema validation**: every `.proto` file parses with `protoc`
  and passes Buf's lint rules (when codegen lands).
- **Wire compatibility**: cross-decode test for every fixture under
  `protocol/tests/fixtures/`; every SDK decodes every other SDK's
  output identically.
- **Backwards compatibility**: add a new optional field; verify v1
  clients ignore it cleanly; verify v2 clients receive default
  values when reading v1 messages.
- **Version negotiation**: peers offering disjoint protocol-version
  sets fail handshake with a useful `reason`.

## Security considerations

See §8 — every wire message is wrapped in a Nostr-signed envelope
verified before decode. Beyond that:

- Generated decoders MUST refuse messages exceeding a configurable
  size cap (default 1 MiB) to prevent decode-bomb DoS.
- Untrusted peer input MUST never be passed to language-specific
  reflection or `eval`-equivalent paths.
- The replay window from IPIP-0014 §1 applies uniformly.

## Privacy considerations

`PeerInfo` records expose multiaddrs (operator IPs) on the wire.
This is the same exposure as IPIP-0016 §8 already documents for
DHT discovery. Operators with privacy needs run with
`--no-advertise` (no PeerInfo published) and accept the loss of
direct-dial features.

## Open questions

- **Should `service` definitions in the proto files imply gRPC, or
  should we use libp2p stream-multiplexing with framed protobuf
  messages?** v1 implementation uses the latter (matches the libp2p
  ecosystem); the `service` keyword is documentation-only for now.
- **How do we sign RMI requests when `caller_signature` is over
  serialized fields that include the signature itself?** Standard
  fix: signature covers a deterministic encoding excluding the
  signature field. Spec the encoding in a follow-up IPIP if it
  proves contentious.
- **DHT record format — protobuf or CBOR?** libp2p uses CBOR for
  some records, protobuf for others. Picking protobuf for
  consistency with the rest of this IPIP, but worth revisiting if
  libp2p moves the ecosystem default.
