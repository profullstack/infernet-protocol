## Abstract

Specifies how Infernet adds a libp2p Kademlia DHT alongside the
existing centralized control plane for peer discovery, content-
addressed model lookup, and decentralized job announcements. The
control plane stays canonical for payment ledger, reputation, and
authoritative state; the DHT supplements it for discovery so the
network keeps working when infernetprotocol.com is unreachable
and operators can find providers by exact model fingerprint
without trusting a central registry.

## Motivation

The README has stated since v1 that "discovery bootstraps from
infernetprotocol.com, then peers gossip via libp2p Kademlia +
Nostr relays." Today that's design intent, not implementation.
Concretely:

1. **Single point of failure for discovery.** If Supabase or the
   Next.js host has an outage, no new providers can register, no
   clients can find existing providers. Operators with running
   daemons keep serving but the network stops accepting
   newcomers.
2. **Trust assumption for model integrity.** IPIP-0014 §2
   introduces weight fingerprints, but the *registry* of trusted
   fingerprints is still centralized. Anyone who can write to
   `models_canonical` controls deterministic-required routing.
3. **Censorship resistance.** A government or hosting provider
   pressuring infernetprotocol.com to deny service shouldn't kill
   the network.

A DHT solves all three: peers find each other without a central
broker, model fingerprints become content-addresses (the hash
*is* the lookup key), and shutting down our domain doesn't shut
down the network.

## Specification

### 1. Topology

Infernet uses **libp2p Kademlia** with parameters:

| Parameter | Value | Notes |
|---|---|---|
| Hash function | SHA-256 | matches IPIP-0014 fingerprint hash |
| Key size | 256 bits | derived from SHA-256 |
| `k` (bucket size) | 20 | libp2p default |
| `α` (concurrency) | 3 | libp2p default |
| Transport | TCP + QUIC | both SHOULD be enabled |
| Peer ID | Nostr pubkey (BIP-340) | reuses provider identity |
| Stream multiplex | yamux | libp2p standard |

Each daemon that opts into the DHT (default: enabled for
providers / aggregators; disabled for outbound-only nodes)
becomes a Kademlia node. The control plane runs 3-5 trusted
"seed" nodes which serve as initial bootstrap peers.

### 2. Bootstrap

A new daemon joins the DHT by dialing one or more bootstrap
multiaddrs. Initial seed list:

```
/dnsaddr/seed1.infernetprotocol.com/p2p/<pubkey1>
/dnsaddr/seed2.infernetprotocol.com/p2p/<pubkey2>
/dnsaddr/seed3.infernetprotocol.com/p2p/<pubkey3>
```

Operators MAY override with `--dht-bootstrap <multiaddr>...` or
`config.dht.bootstrap = [...]`. Community-run bootstrap nodes are
welcome and SHOULD be added to the published list once they meet
uptime requirements (TBD; revisit after first 90 days).

### 3. Record types

The DHT carries three types of records, each a Nostr-signed
envelope (same format as our HTTP API per IPIP-0003) so any
peer can verify authenticity without trusting the relay.

#### 3a. Provider announcement

```
key   = sha256("provider:" + pubkey)
value = signed-envelope({
    pubkey,
    role: "provider" | "aggregator" | "client",
    address: "host:port",
    served_models: [{ name, fingerprint, quantization }, ...],
    workload_classes: ["A", "B", "B.5", ...],
    last_announced_at: <unix-seconds>,
    expires_at: <unix-seconds>
})
```

Republished by the provider every 10 minutes. Records older than
1 hour are considered stale and ignored.

#### 3b. Model fingerprint mapping

```
key   = sha256("model:" + fingerprint)
value = signed-envelope({
    fingerprint,
    canonical_name: "Qwen/Qwen2.5-7B-Instruct",
    aliases: ["qwen2.5:7b", ...],
    providers: [pubkey1, pubkey2, ...]   // recent observers
})
```

Anyone can publish (it's a fingerprint of public weights), but
the consumer dereferences the `providers` list back through
provider announcements (3a) to verify those providers actually
serve the model with that fingerprint. This is content-addressed:
the lookup key IS the fingerprint, so there's no central
registry to compromise.

#### 3c. Job announcement (future, optional)

```
key   = sha256("job:" + workload_class + ":" + model_fingerprint)
value = signed-envelope({
    job_id,
    poster_pubkey,
    model_fingerprint,
    workload_class,
    payment_offer_usd,
    deadline,
    expires_at
})
```

Lets clients announce jobs directly to the DHT (bypassing the
control plane's `POST /api/v1/jobs`). Providers subscribe to
topic keys for the model fingerprints they serve, pull jobs,
and contact the poster directly. Reserved for a later phase —
not in v1 because escrow + reputation still flow through the
central ledger and we don't have peer-to-peer escrow primitives
yet.

### 4. DHT vs control plane

The DHT supplements; it does not replace.

| Concern | Authority |
|---|---|
| Provider exists, is reachable | DHT (announcements) OR control plane heartbeats |
| Model X has fingerprint Y | DHT content-address (3b) — the hash *is* the key |
| Provider has reputation R | Control plane (`providers.cpr_score`) — DHT cannot host this without a coin-style consensus we're not building |
| Payment owed to provider P | Control plane (Postgres ACID, IPIP-0014 §8) |
| Job state machine | Control plane |

When an operator queries "who serves fingerprint X right now?",
the daemon SHOULD merge results from both: control plane heartbeats
(authoritative when reachable) + DHT announcements (fallback when
not). Conflicts resolve by recency.

### 5. Authentication

Every DHT record value MUST be a Nostr-signed envelope (BIP-340
Schnorr) where the signature is over the canonical JSON of the
record fields. Receivers MUST verify the signature before
trusting any field; records with invalid signatures MUST be
discarded and the publisher's reputation tracked downward.

The `pubkey` field of the record MUST match the public key
deriving the signature. The DHT layer enforces this at insert
time on writers it controls; readers verify on retrieval since
DHT relays are untrusted.

### 6. TTL and republish

| Record | TTL | Republish cadence |
|---|---|---|
| Provider announcement | 1 hour | 10 minutes |
| Model fingerprint mapping | 24 hours | 6 hours |
| Job announcement | min(deadline, 1 hour) | 5 minutes |

A node that goes offline stops republishing; its records age out
within one TTL cycle. This is the AP guarantee on DHT — stale
data is bounded by TTL, not by any global consistency mechanism.

### 7. Partition merge — last-write-wins by Lamport

When a network partition heals and two peers' DHT views
diverge on the same key, the merge rule is **last-write-wins
by Lamport timestamp** (using the IPIP-0014 §5 / IPIP-0015 §1
clock):

```
keep value with max(lamport_ts)
ties broken by signing pubkey lex order  (matches IPIP-0014 §5)
```

This is a documented AP choice — it can drop a write that lost
a race. CRDTs would be correct-er but expensive to implement.
For our record types (provider state, fingerprint registry, job
postings), LWW is acceptable because:
- Provider announcements are republished on a fixed cadence;
  any lost write recovers within one cycle.
- Fingerprint mappings are append-mostly (no real updates).
- Job announcements expire fast; lost writes age out.

If a future record type needs CRDT semantics, that future IPIP
specifies its own merge rule.

### 8. Privacy considerations

Publishing provider announcements to a public DHT exposes:

- Operator's Nostr pubkey (already public via control-plane heartbeat)
- Operator's IP address (already exposed unless `--no-advertise`)
- The list of models the operator serves (already in heartbeat,
  but heartbeats only go to the control plane; DHT records go to
  every peer)
- Workload-class capabilities

The new exposure from DHT participation is the **model list +
fingerprint list to every peer in the network**, not just the
control plane. Operators who need anonymity SHOULD continue
running with `--no-advertise --no-dht` (the latter disables DHT
participation entirely; outbound polling against the control
plane still works).

## Rationale

**Why Kademlia and not GossipSub or pubsub?** Kademlia is a
content-addressed DHT — exactly what we need for fingerprint-
keyed lookups (§3b). GossipSub is a pubsub overlay; it broadcasts
to subscribers rather than supporting key-based lookup. We MAY
add GossipSub for job-announce flooding (§3c) in a later IPIP if
job latency from the announcement primitive proves too slow.

**Why libp2p?** Battle-tested (IPFS, Filecoin, Polkadot,
Ethereum), MIT-licensed, JS implementation works in our Node-only
stack. Reusing libp2p means we don't reimplement the transport,
peer-store, NAT traversal, or security handshake.

**Why is reputation centralized when discovery is decentralized?**
Reputation requires consensus on who behaved well — that's the
hard problem cryptocurrencies solve with proof-of-work / stake
(expensive, slow). For v1, the control plane is the single source
of truth for reputation; if we want decentralized reputation, that
gets its own IPIP and likely involves a CPR-style proof-of-correct-
work protocol we control.

**Why expire records by Lamport timestamp instead of wall-clock?**
Wall-clock requires synchronized clocks (the very thing this
infrastructure can't assume per Lamport's paper). Lamport
timestamps are monotonic per provider and survive clock skew.
Expiration as a server-side check at retrieval time uses the
publishing provider's claimed time but bounds it via the
republish cadence (a peer that keeps republishing fresh
timestamps stays live; one that stops ages out).

## Backwards compatibility

Fully additive. Daemons without DHT support continue to register
and heartbeat through the existing control-plane endpoints. The
control plane treats DHT-only providers as first-class once it
has consumed at least one signed announcement (proves identity);
DHT participation does NOT replace the existing
`POST /api/v1/node/register` flow until we have decentralized
reputation.

## Reference implementation

To land:
- `packages/dht/` — new package, wraps `@libp2p/kad-dht` + our
  signed-envelope helpers from `@infernetprotocol/auth`.
- `apps/cli/lib/dht.js` — daemon-side DHT lifecycle: bootstrap,
  publish, subscribe.
- `apps/cli/commands/init.js` — adds `--no-dht` flag, persists
  `config.dht.{enabled, bootstrap[]}`.
- `apps/web/app/api/seed/dht-status/route.js` — operator-facing
  view of seed-node health.
- `apps/web/lib/scheduler/discover.js` — merges control-plane
  heartbeats + DHT announcements when picking candidates.
- `seed/` — Docker image + bootstrap config for community-run
  seed nodes.
- DNS: `seed1.infernetprotocol.com` … `seed3.infernetprotocol.com`
  pointing at our hosted seed nodes.

## Test plan

- **Unit:** record encode/decode + signature verify; record
  expiry; LWW merge correctness.
- **Integration:** spin up 5 in-process DHT nodes; provider
  announces; client retrieves and dials successfully.
- **Partition test:** split a 5-node DHT into two halves;
  verify both halves accept writes; merge correctly when
  partition heals (Lamport LWW respected).
- **Censorship test:** kill the control plane; verify DHT-only
  discovery + dial still works for jobs that don't require
  payment escrow.
- **Authentication test:** publish a record with a forged
  signature; verify all consumers reject it.

## Security considerations

- **Sybil attacks.** A single attacker spinning up many DHT
  nodes can manipulate routing tables and eclipse honest peers.
  Standard mitigation: rate-limit per IP at seed nodes,
  reputation-weight DHT routing decisions (only accept records
  from peers with non-zero CPR), gossip cross-validation.
- **Content poisoning.** An attacker publishes a record claiming
  fingerprint X = malicious-weights. Consumers verify the
  publisher's signature, but the publisher could be lying about
  what they actually serve. Mitigation: canary prompts (per
  IPIP-0014 §2 security considerations) — the control plane
  periodically issues prompts whose outputs are stable for
  canonical weights, mismatches expel the provider.
- **DHT denial-of-service.** Flooding the DHT with junk records
  bloats peer storage. Mitigation: per-publisher record quotas
  enforced at storing peers, signed records only (anonymous
  spam impossible).
- **Eclipse attacks.** An attacker surrounds a target peer with
  malicious peers, controlling all its DHT view. Mitigation:
  multiple bootstrap peers (3-5), periodic re-bootstrap, refuse
  routing tables dominated by a single ASN.

## Privacy considerations

See §8. The privacy delta from DHT participation is the broader
broadcast of model-lineup + capability info that today only
heartbeats over HTTPS to the control plane. Operators who need
anonymity get `--no-advertise --no-dht` and continue functioning
in outbound-only mode.

## Open questions

- **Should we expose a JS / Python client SDK that talks the
  DHT directly** (no daemon required), or always require the
  daemon? Direct SDK is more decentralized but multiplies
  attack surface (every browser becomes a DHT participant).
  Punt to a follow-up IPIP.
- **Bootstrap node trust.** Three to five seed nodes are still
  a centralized choice. Community-run bootstrap eventually, but
  who curates the list?
- **NAT traversal.** libp2p has Circuit Relay v2 + AutoNAT;
  enabling them adds traffic but lets NATted providers
  participate. Default on or off?
- **GossipSub for job announcements** — see Rationale. Defer
  pending real-world latency data once §3a/§3b are live.
