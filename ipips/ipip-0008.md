## Abstract

Defines how Infernet provider nodes detect and advertise GPU-to-GPU and
host-to-host interconnect capabilities (NVLink, AMD xGMI / Infinity Fabric,
InfiniBand / RoCE, AWS EFA), and how clients can route distributed-training
or tensor-parallel inference jobs to providers that have the fabric to run
them efficiently. Also specifies the engine-spawn environment variables a
provider's daemon emits when those interconnects are present, so downstream
toolkits (NCCL, RCCL, libfabric, vLLM, Ollama) opt into the right transport
without per-node configuration.

## Motivation

The dominant inference workload today (single-GPU serving of 7B–70B models)
runs fine on commodity links — see the homepage `/#how-it-competes` for the
positioning argument. But Infernet also targets two coordination-sensitive
workloads: tensor-parallel inference of larger models across same-host
GPUs, and async/loosely-coupled distributed training (DiLoCo, federated,
gradient-compression). Both benefit from a fabric, and both are unusable
without one above a certain model size.

For matchmaking to work, the network needs three things:

1. A consistent way for providers to report what fabric they have, *without*
   leaking enough hardware fingerprint to deanonymize the operator.
2. A way for clients to express "this job needs NVLink" or "this job needs
   RDMA between hosts" and have the routing layer respect it.
3. A standard set of environment variables the daemon sets on the engine
   subprocess so the toolkits actually use the fabric. NCCL in particular
   silently falls back to TCP unless told which HCAs to use, which is the
   single most common 100× perf-loss footgun in distributed training.

## Specification

### 1. Detection

`@infernetprotocol/gpu` exposes `detectInterconnects(): Promise<Capability>`.
Implementations MUST be best-effort: any vendor command that is missing
or fails MUST be treated as "not detected" (no interconnect of that
type), MUST NOT throw, and MUST NOT block detection of the other types.

Returned shape (always defined; missing fields default to "not detected"):

```ts
type Capability = {
  nvlink: {
    available: boolean,
    topology: 'pair' | 'mesh' | 'all-to-all' | 'none' | 'unknown',
    links: Array<{ from: number, to: number, kind: string }>
  },
  xgmi: {
    available: boolean,
    topology: 'pair' | 'mesh' | 'all-to-all' | 'none' | 'unknown',
    links: Array<{ from: number, to: number, weight: number }>
  },
  infiniband: {
    available: boolean,
    devices: Array<{
      name: string,
      board_id: string | null,
      port: number | null,
      state: 'active' | 'down' | 'init' | string | null,
      rate: string | null,
      link_layer: 'InfiniBand' | 'Ethernet' | string | null
    }>
  },
  efa: {
    available: boolean,
    devices: Array<{ bdf: string | null, raw: string }>
  },
  rdma_capable: boolean
}
```

Detection sources:

- **NVLink**: parse `nvidia-smi topo -m`. Cells matching `/^NV\d+$/i` are
  NVLink edges; `SYS`, `NODE`, `PHB`, `PIX`, `PXB` are PCIe paths and
  MUST be ignored. The matrix is symmetric; implementations MUST emit at
  most one edge per unordered pair.
- **xGMI / Infinity Fabric**: parse the weight matrix from
  `rocm-smi --showtopo`. Edge weights ≤ 30 SHOULD be classified as xGMI;
  higher weights typically indicate cross-socket PCIe and SHOULD be ignored.
  Implementations MAY tune the threshold per ROCm version.
- **InfiniBand / RoCE**: enumerate `/sys/class/infiniband/*` (Linux only).
  RoCE NICs appear here with `link_layer = Ethernet`; both are reported as
  InfiniBand. A device is `available` iff at least one port is `active`.
- **AWS EFA**: `lspci -d 1d0f: -mm` filtered by `/EFA|Elastic Fabric/i`
  (Linux only). EFA also sometimes registers under `/sys/class/infiniband/`,
  in which case both InfiniBand detection and EFA detection MAY return
  signal — clients SHOULD treat them as additive, not duplicate.
- **Topology classification**: given `n` GPUs and `e` deduped edges,
  `pair` if `n=2`, `all-to-all` if `e ≥ C(n,2)`, otherwise `mesh`.
  Self-loops MUST be excluded.

The function `parseNvidiaTopo(text: string)` and `parseRocmTopo(text: string,
threshold?: number)` are exported for testing without spawning shell-outs.

### 2. Advertisement (registration payload)

`infernet register` MUST include a `specs.interconnects` field in the
Nostr-signed `POST /api/v1/node/register` body when role is `provider`.
The advertised shape is **coarsened** from the full detection shape — no
device names, board IDs, BDFs, or per-link kinds — to limit hardware
fingerprinting:

```json
{
  "specs": {
    "interconnects": {
      "nvlink":     { "available": bool, "topology": "...", "link_count": int },
      "xgmi":       { "available": bool, "topology": "...", "link_count": int },
      "infiniband": { "available": bool, "active_port_count": int },
      "efa":        { "available": bool, "adapter_count": int },
      "rdma_capable": bool
    }
  }
}
```

Servers MUST persist this in `providers.specs` (jsonb). Servers MUST NOT
require the field — a provider with no advertised interconnect block is
treated as "no fabric, single-GPU only," not as an error.

### 3. Engine-spawn environment

When the daemon spawns an engine subprocess for a job, it MUST merge the
output of `interconnectEnv(capability)` into the child's environment.
The function returns the union of:

| Condition                  | Env vars                                                |
| -------------------------- | ------------------------------------------------------- |
| `nvlink.available`         | `INFERNET_NVLINK=1`, `INFERNET_NVLINK_TOPOLOGY=<topo>`  |
| `xgmi.available`           | `INFERNET_XGMI=1`, `INFERNET_XGMI_TOPOLOGY=<topo>`      |
| ≥1 IB device with `state=active` and `name` set | `NCCL_IB_DISABLE=0`, `NCCL_IB_HCA=<name>:<port>,…` |
| `efa.available`            | `FI_PROVIDER=efa`, `NCCL_PROTO=simple`, `INFERNET_EFA=1` |
| `rdma_capable`             | `INFERNET_RDMA=1`                                       |

The `INFERNET_*` variables are advisory markers the daemon and engine
adapters can branch on. The `NCCL_*` and `FI_*` variables are the
established standard for NCCL / RCCL / libfabric and SHOULD be set
unconditionally when the corresponding fabric is present so that
downstream toolkits don't silently fall back to TCP.

The user MAY override any of these in the job's launch profile; daemons
MUST NOT clobber an explicit override from the caller.

### 4. Client-side capability requirements

A job submission MAY include a `requires.interconnects` block:

```json
{
  "requires": {
    "interconnects": {
      "rdma": true,
      "nvlink": "pair" | "mesh" | "all-to-all" | true,
      "min_active_ib_ports": 1
    }
  }
}
```

The matchmaking layer MUST filter providers whose advertised
`specs.interconnects` does not satisfy the requirements. A missing or
empty `requires.interconnects` block matches all providers.

## Rationale

**Why coarse advertisement.** Per IPIP-0001 the registration payload is
deliberately coarse (vendor + VRAM tier, not full GPU model). Adding
device names like `mlx5_0` would meaningfully fingerprint the operator's
hardware (model, vendor, sometimes a specific SKU). The coarsened shape
keeps the same privacy contract while still conveying enough capability
signal for matchmaking.

**Why advisory `INFERNET_*` markers.** NCCL / libfabric have well-defined
env var names that toolkits already honor, so a daemon that just sets
those gets correct behavior with no engine-side code. The `INFERNET_*`
markers are for *our* adapters that want to make routing or scheduling
decisions ("if `INFERNET_NVLINK=1` and topology is all-to-all, try
tensor-parallel"). Using a separate namespace makes the boundary clear.

**Why detection is best-effort, never throws.** Most operators won't have
all four fabrics. A failed `nvidia-smi` on an AMD-only box is the common
case, not an exception. Throwing would force every caller to wrap in
try/catch; silently treating it as "not detected" is the right default.

**Why InfiniBand state machine isn't normative.** The
`/sys/class/infiniband/*/state` file's exact contents have shifted across
kernel versions — `4: ACTIVE`, `ACTIVE`, `state=4`, etc. The spec mandates
the canonical values (`active`, `down`, `init`) but allows implementations
to forward whatever raw state string they observed. Routing decisions only
care about `active`.

**Alternatives considered:**
- **Full GPU UUIDs in registration**: rejected on privacy grounds.
- **Per-link bandwidth measurement**: rejected as too costly. Operators
  who care can opt into a benchmarking tool; the routing-time signal
  doesn't need it.
- **A new `interconnects` table joined to `providers`**: rejected for
  premature normalization. The data is small, slow-changing, and only
  read alongside the rest of the provider row. Jsonb is fine.

## Backwards compatibility

Providers running pre-IPIP-0008 daemons just don't include the field —
servers treat that as "no fabric advertised, single-GPU only." Servers
running pre-IPIP-0008 schemas accept the field as opaque jsonb without
needing a migration; consumers that don't read it ignore it. Wire format
is purely additive.

- `packages/gpu/src/interconnect.js` — detection + env emission.
- `apps/cli/commands/{setup,gpu,register}.js` — surface in CLI output and
  registration payload.

  across the user's providers.

## Test plan

- **Unit (vitest)**: 14 cases in `tests/interconnect.test.js` — parser
  round-trips for `nvidia-smi topo -m` and `rocm-smi --showtopo`, env
  construction for each fabric independently and in combination, edge
  cases (no fabric, IB up but inactive, sub-threshold xGMI weights).
- **Integration**: `infernet setup` and `infernet gpu` on hosts with
  known fabric — A100 + NVLink (4-GPU all-to-all), MI300 + xGMI, p4d
  with EFA, plain laptop (none of the above). All four exercised in CI
  if hardware is present; otherwise the relevant detector returns empty
  silently.
- **End-to-end**: register a provider with synthetic
  `specs.interconnects`, submit a job with `requires.interconnects.rdma=true`,
  verify the matchmaker filters correctly. (Routing wiring lands in the
  matchmaker IPIP, separately.)

## Security considerations

The advertised block is coarsened and signed under the same Nostr key as
the rest of the registration payload (per IPIP-0003). A malicious provider
could lie about having NVLink to win RDMA-required jobs and then run them
slowly; the same incentive exists for any reported capability. Mitigation
is reputation (CPR receipts per IPIP-0007): a provider that reliably
under-delivers on jobs they accepted gets routed away from. No new
authentication surface.

The engine-spawn env helper passes attacker-controlled-by-default-config
strings (e.g. NCCL_IB_HCA values) into a subprocess; HCA names come
exclusively from the local `/sys/class/infiniband/` enumeration, not from
network input, so there's no injection vector here.

## Privacy considerations

The advertised registration block deliberately *does not* include device
names, board IDs, PCI BDFs, GPU UUIDs, or vendor-specific identifiers.
Topology and counts (e.g. "3 active IB ports", "all-to-all NVLink across
4 GPUs") narrow the hardware identity, but match a sufficiently large
class of common configurations that they don't uniquely identify an
operator. Operators who consider even this signal too revealing can pass
`--no-advertise-interconnects` (proposed flag, not yet implemented) which
omits the block entirely.

The engine-spawn env vars are local to the subprocess and never leave
the node. They don't appear on the wire and aren't logged by the daemon.

## Open questions

- Should we add a per-job `--prefer-rdma` soft preference (cheaper job
  runs on RDMA-capable provider when one is available, else falls back)?
  Today the requirement is binary.
- Should the daemon proactively run a NCCL micro-benchmark on first boot
  to validate that the advertised fabric actually reaches the claimed
  bandwidth? Useful for sniffing out misconfigurations; expensive on
  every boot. Probably an opt-in `infernet doctor --bench-fabric`.
- For multi-GPU non-NVLink providers, should we still advertise
  tensor-parallel-capable but with a `pcie_only` topology marker? The
  current spec says no (PCIe-only providers shouldn't claim NVLink in
  any form), but downstream routing might want to know.
