## Abstract

This IPIP defines the scope and acceptance criteria for the **Infernet
Protocol v1.0** release. It enumerates what must be true on `master`
before the `v1.0.0` tag is cut: which protocol surfaces are frozen,
which integrations must be live, which release artifacts must publish,
and which work has been deferred to v1.x or later.

It is a living document for the duration of the pre-launch phase. As
items ship, they move from **In scope** to **Shipped** with a commit or
PR reference. Anything still in **In scope** when we propose to cut the
release is by definition a blocker.

## Motivation

Pre-launch work has been driven by issues, TODO.md, conversations, and
[`MEMORY.md`](../.claude-state/) entries. There is no single canonical
list of "what does v1.0 mean." This makes scope discussions ad-hoc and
makes it hard for new contributors to see how their work fits the
release.

This IPIP fixes that. After it is `Active`, every PR that closes a
launch-blocking item should reference IPIP-0001 in its description, and
the corresponding line in this document should move to **Shipped**.

## Specification

### What "v1.0" means

A self-hosted operator can:

1. Install the CLI from a published artifact (npm + Docker + Homebrew).
2. Run `infernet init` and `infernet start` against either the public
   `infernetprotocol.com` control plane or their own self-hosted Supabase +
   Next.js stack.
3. Have their node receive real chat-inference jobs, run them on a
   model the operator chose (via Ollama by default, per
   `@infernetprotocol/engine`), stream tokens back to the control
   plane, and get credited for the work.
4. Receive payment for completed jobs via at least one settlement path.

A client can:

1. Submit a chat job through the web dashboard or the SDK.
2. See the job assigned to a live provider node, watch tokens stream in
   real time, and receive the final result.
3. Pay for the job via the configured payment path (CoinPayPortal at
   minimum for v1.0).

The wire protocols listed under **Frozen surfaces** are stable. Breaking
changes after v1.0 require a new IPIP and a version bump.

### Frozen surfaces (v1.0 = locked)

| Surface | Source of truth | Version |
| ------- | --------------- | ------- |
| Signed-request envelope (`X-Infernet-Auth`) | `packages/auth/src/sign-request.js` | v1 |
| `/api/v1/node/*` route shapes | `apps/web/lib/data/node-api.js` | v1 |
| Engine IPC (Mojo backend, NDJSON over stdio) | `packages/engine/src/protocol.js` | v1 |
| CLI config schema (`~/.config/infernet/config.json`) | `apps/cli/lib/config.js` | v2 |
| CLI command surface (`infernet init`/`register`/`start`/...) | `apps/cli/commands/*.js` | v1 |

A bug fix that does not change wire format is fine. A field rename, a
required-field addition, or a status-code change requires a new IPIP.

### In scope (must ship before v1.0)

- [ ] **npm publishing un-blocked.** `NPM_TOKEN` regenerated as
      Automation token (since the original was issued under 2FA),
      `gh secret set NPM_TOKEN --repo profullstack/infernet-protocol`,
      release workflow successfully publishes `@infernetprotocol/cli`
      and other public packages.
- [ ] **Homebrew tap** un-blocks once npm does. Tap repo published with
      a working formula that pulls the published CLI.
- [ ] **`infernetprotocol.com` DNS** pointed at the v1 Next.js deployment so
      `infernet init` with no `--url` flag is meaningful.
- [ ] **Read-route auth.** `/api/overview`, `/api/nodes`,
      `/api/providers`, `/api/jobs` etc. accept the signed-request
      envelope and scope responses to the calling identity. (Extends
      the work in commit `b060a07`.)
- [ ] **Real chat runtime end-to-end.** ✅ scaffolded — a node with
      Ollama + a pulled model serves real chat jobs through the engine
      package shipped in commit `7d27273`. Remaining: prove this on a
      GPU box with a non-trivial model and one full client→node→client
      round-trip recorded in the test plan.
- [ ] **Job-payment loop.** A completed chat job credits the provider's
      payout balance and a payout request settles via the CoinPayPortal
      path. End-to-end run documented in the test plan.
- [ ] **Read-only public dashboard.** Anyone hitting `infernetprotocol.com`
      sees current node count, recent jobs (anonymized), aggregate
      throughput. No auth required for public stats.
- [ ] **Security review.** The signed-request layer, the Nostr identity
      handling, and the engine sidecar boundary get an explicit review
      pass — at minimum self-review via `/security-review`, ideally a
      second set of eyes on the auth path.
- [ ] **Dead-skeleton cleanup.** `packages/inference/src/distributed/*`
      currently looks up a `settings` / `node_roles` table that isn't
      in any migration. Either build the tables or delete the
      skeleton — keeping it as-is misleads new contributors.
- [ ] **`apps/cli/lib/runtime-config.js`** — drop the legacy
      `supabase: {}` block; the CLI hasn't used it since
      commit `05982b7`.
- [ ] **IPIP-0003 phases 2–5** — dashboard auth (Supabase Auth wired
      into Next.js), CLI device-code login (`infernet auth login`),
      pubkey-linking ceremony, **and** the self-hosted Nostr relay at
      `wss://relay.infernetprotocol.com` (`strfry` on a small VPS).
      Phase 1 (schema + `.env`) shipped in commit `a84d6de`.
- [ ] **IPIP-0004 phases 1–5** — supported-coin allow-list, CLI
      `infernet payout` extension, payouts step in
      `infernet setup` / `init`, **CoinPayPortal-provisioned custodial
      Lightning wallets (opt-in)**, **and** client-side payment
      routing on `/api/chat` so jobs can specify a payment coin.
      Every CoinPayPortal-supported coin must work on day one — the
      chain-agnostic story is the launch differentiator.
- [ ] **IPIP-0005 phases 1–4** — data access architecture: CI lint
      guard against direct DB clients in browser/CLI/SDK,
      `@supabase/ssr` cookie-based server-side auth (no direct
      browser → Supabase calls), self-hosted Supabase parity in CI.
      This is the rule that makes the whole "swap backend with .env,
      no client redeploy" story actually true.
- [ ] **P2P transport via Nostr relays.** Replace the signed-HTTP
      calls to the control plane with signed Nostr events broadcast
      on `wss://relay.infernetprotocol.com` plus public relays. The
      control plane stops being a routing chokepoint; nodes can find
      jobs and providers can advertise capability without hitting
      our HTTP endpoint. This is the "Bitcoin for AI" wire — without
      it, the protocol is still centrally-routed despite the rest of
      the decentralization work.
- [ ] **libp2p kad-dht node registry.** Persistent decentralized
      discovery so providers and clients can find each other when
      the control plane is offline / censored. Pairs naturally with
      the Nostr-relay transport above (Nostr for messages, DHT for
      durable peer addressing).
- [ ] **Separate payout identity.** Split `node.publicKey` (signs
      requests) from `node.payoutPublicKey` (receives funds) so
      deanonymizing payouts can't deanonymize compute. Privacy-first
      operator pitch falls apart without this — currently a single
      pubkey ties together the two surfaces a TLA most wants to
      correlate.
- [ ] **Tor / SOCKS5 outbound.** Operator can route every outbound
      connection (control plane, Nostr relays, peers) through Tor
      or a SOCKS5 proxy. Combined with `--no-advertise` (already
      shipped), gives full anonymity for nodes that want it. Without
      it, operator IPs are visible to relay operators / observers
      regardless of identity hygiene elsewhere.

### Shipped (already on master)

- [x] **Phase 1 decentralization.** Nodes hold a Nostr keypair instead
      of a Supabase service-role key; every control-plane call is
      Schnorr-signed. Replay cache, sanitized telemetry,
      `--no-advertise` outbound-only mode. Commits `7cf88ad`,
      `b060a07`, `05982b7`.
- [x] **v0.1.0 Docker image** published at
      `ghcr.io/profullstack/infernet-provider:0.1.0`.
- [x] **Pluggable inference engine.** `@infernetprotocol/engine` with
      `ollama` (default), `mojo` (experimental), `stub` (fallback)
      backends. Daemon imports the abstraction only — adding a new
      backend is one file. Commit `7d27273`.
- [x] **NIM fallback for chat** when no P2P providers are online —
      keeps the dashboard demo path working.
- [x] **CoinPayPortal invoice + webhook routes.** Crypto payment path
      live for client-side checkout.
- [x] **Default model preference: Qwen.** Documented in
      `packages/engine/README.md`. The engine itself is model-agnostic;
      this is purely a docs/example default.
- [x] **IPIP-0003 phase 1.** `pubkey_links` and `cli_sessions`
      schema migration plus `NEXT_PUBLIC_SUPABASE_*` and
      `INFERNET_CLI_SESSION_SECRET` env additions
      (commit `a84d6de`). Implementation phases 2–4 still in scope.

### Deferred to v1.x (explicitly out of scope for v1.0)

The scope for v1.0 has been deliberately expanded to "the full
thesis satisfied" — see the In scope list. Only items that don't
gate the headline pitch live below.

- **Direct on-chain settlement.** Noncustodial chain adapter, no
  CoinPayPortal dependency at all. v1 ships with CoinPayPortal-
  brokered multi-currency; on-chain-only settlement is its own
  multi-IPIP initiative (token economics, escrow contracts, gas
  models per chain).
- **Real Mojo+MAX inference backend.** The scaffold at
  `engine/mojo/` stays experimental in v1; Ollama is the recommended
  default and covers the v1 launch use case. Resurfacing this
  contradicts the engine-abstraction decision logged in
  `feedback_default_model_choice.md` — flag if you want it pulled
  back in.
- **Marketplace UI** (`/marketplace`, `/rentals/[id]`). The earlier
  call was "real runtime first for v1, marketplace as v1.x add-on."
  Flag if that's changed.

Each of the deferred items will get its own IPIP when the work starts.

### Timeline implications

This expanded scope means **v1.0 = "the full Bitcoin-for-AI thesis,
shippable end-to-end."** That's a multi-month sprint, not a 2-week
launch. Honest cost estimate (sequential, single-team):

| Workstream                              | Rough effort |
| --------------------------------------- | -----------: |
| IPIP-0003 phases 2–5 (auth + relay)     | 2–3 weeks    |
| IPIP-0004 phases 1–5 (multi-currency)   | 2–3 weeks    |
| P2P transport via Nostr relays          | 4–6 weeks    |
| libp2p kad-dht discovery                | 2–3 weeks    |
| Separate payout identity                | 1 week       |
| Tor / SOCKS5 outbound                   | 1 week       |
| Existing v1 blockers (read-route auth, security review, dashboard, npm/DNS) | 2–3 weeks |
| **Total (worst case, no parallelism)**  | **≈ 14–20 weeks** |

If parallelizable (auth and payments and transport are mostly
independent), the wall-clock could compress to ~8–10 weeks with
focus. Either way, the existing tagged releases (`v0.1.x`) are not
v1.0 — they're early-access milestones leading to it.

Suggested intermediate tags so the public has something to follow:

- **v0.2** — IPIP-0003 phases 2–4 shipped (auth working end-to-end)
- **v0.3** — IPIP-0004 phases 1–3 shipped (multi-currency payouts in CLI)
- **v0.5** — IPIP-0004 phases 4–5 + IPIP-0003 phase 5 (LN, payment routing, relay)
- **v0.8** — separate payout identity + Tor outbound
- **v0.9** — libp2p kad-dht + P2P Nostr transport (the headline)
- **v1.0** — security review passed, all v1 in-scope items shipped, npm/DNS un-blocked

## Rationale

**Why a Process IPIP and not a milestone in GitHub?** A milestone is a
list of issues. This document is a contract: the frozen surfaces, the
acceptance criteria, the explicitly-deferred items. A new contributor
reading IPIP-0001 should know in five minutes what v1.0 is and what
it isn't.

**Why "scaffolded but not GPU-verified" counts as in-scope, not
shipped.** The engine abstraction is on `master` and 38 tests pass, but
no node has actually served a real Qwen chat job to a real client yet.
Until that happens once on a GPU box, the integration risk is real.

**Why Ollama over a custom Mojo+MAX runtime for v1.0.** Ollama already
solves model download, multi-vendor GPU, and chat streaming. Custom
runtime work would gate the launch on multi-week implementation effort
for a feature operators don't need yet. The Mojo backend is kept as
optionality (`engine/mojo/`), not as a v1.0 dependency.

**Why no smart-contract settlement at v1.0.** CoinPayPortal works
today; building a noncustodial on-chain adapter is its own multi-IPIP
initiative and would slip the launch by months for a property no early
operator is asking for.

## Backwards compatibility

For pre-1.0 users (anyone running off a `0.x` Docker image or a
`master` checkout): the CLI's `migrate()` already strips
`supabase.serviceRoleKey` from old configs and promotes
`supabase.url` to `controlPlane.url` (commit `05982b7`). No further
migration is required to move to v1.0 from any post-Phase-1 build.

For implementers building against the wire protocol: every surface in
the **Frozen surfaces** table is at the listed version on v1.0. A v1.0
client and a v1.0 server will interoperate. Future protocol versions
will be additive and version-negotiated, or will live behind a new
endpoint path.

## Test plan

A v1.0 release is acceptable when **all** of the following pass on a
freshly-provisioned environment:

1. `pnpm install && pnpm vitest run` — all tests pass on a clean clone.
   Today: 38/38.
2. `infernet init` followed by `infernet register` succeeds against
   `https://infernetprotocol.com` with no manual configuration beyond
   `INFERNET_ENGINE_MODEL`.
3. A chat job submitted from the dashboard is picked up by the running
   node, streams real Qwen tokens back, and the dashboard records the
   completion. Captured as a screen recording in
   `docs/release/v1.0.0-e2e.mp4` (or equivalent).
4. The completed job credits a payout balance, and a manually-triggered
   payout settles via CoinPayPortal. Transaction id recorded in the
   release notes.
5. `docker pull ghcr.io/profullstack/infernet-provider:1.0.0` and
   `npm install -g @infernetprotocol/cli@1.0.0` both succeed from a
   clean machine.
6. `/security-review` on the v1.0.0 release branch returns no
   high-severity findings.

## Security considerations

The launch criteria above are the security-relevant items in summary
form:

- **Read-route auth** is in scope precisely because the read endpoints
  currently leak operator-scoped data without authentication. Shipping
  v1.0 without it is a known information-disclosure issue.
- **Security review** is its own line item rather than something
  implicit in code review. The signed-request envelope and the engine
  sidecar boundary are the two hot spots.
- **Sanitized telemetry** (heartbeat / register payloads dropped
  hostname / arch / cpu / ram and reduced GPUs to `{vendor, vram_tier,
  model}`) is already shipped — re-affirmed here so we don't
  accidentally re-expand the payload before launch.

## Privacy considerations

- The `--no-advertise` flag must continue to work end-to-end so a node
  can run outbound-only without leaking IP+port. Re-tested as part of
  the test plan above.
- Aggregate dashboard stats (test plan item: read-only public
  dashboard) MUST NOT expose per-node IPs, hostnames, payout
  addresses, or any field that links a job to a real-world identity.

## Open questions

- ~~**Control-plane address.**~~ Resolved 2026-04-26: root domain
  `https://infernetprotocol.com`, signed node API mounted at
  `/api/v1/node/*` (no separate `api.` subdomain).
- **Single-model vs multi-model nodes.** Today a node serves one model
  via `INFERNET_ENGINE_MODEL`. Routing assumes the control plane knows
  what each provider serves. Is per-job advertising in scope, or do we
  ship v1.0 with single-model nodes only and add multi-model in v1.x?
- **What is the minimum cost of running a launch-day node?** If it's
  >$200/mo for a hobbyist, that shapes the marketing positioning. We
  should pick a reference machine (e.g. RTX 3060 12GB, used) and write
  a "$X/month, $Y break-even" example for the launch announcement.
- **Are we publishing the Mobile and Desktop apps at v1.0, or only the
  CLI + web?** Affects scope by ~weeks; suggest deferring desktop /
  mobile to v1.x unless they're already close.
