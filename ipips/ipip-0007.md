## Abstract

Integrate the **CoinPay Reputation Protocol (CPR)** as Infernet's
reputation and Sybil-resistance layer. Every completed job writes a
signed Receipt to CPR; the control plane's job router consults CPR
Credentials when scoring providers; disputed outcomes flow through
CPR's existing dispute pipeline. Identities use the operator's
Nostr pubkey (`did:nostr:<hex64>`) and the platform's web identity
(`did:web:infernetprotocol.com`).

This is the missing trust layer that turns "any pubkey can claim to
be a provider" (per IPIP-0006 peer discovery) into "any pubkey can
*join*, but only those with proven track records get routed paying
work."

## Motivation

IPIP-0006 makes peer discovery permissionless. That's a feature —
no central gatekeeper — but it leaves a real Sybil problem: a fresh
malicious node can register, fail to serve, and burn client funds.
Without a trust signal, clients can't safely prefer reliable
providers, and operators can't compound a good track record into
better job placement.

CPR (already implemented in `~/src/coinpayportal/supabase/migrations/20260214000000_reputation_protocol.sql`)
solves this. It's:

- **DID-based** (so it works with any pubkey-shaped identity)
- **Receipt-driven** (signed, immutable record per task)
- **Already deployed** (CoinPay's running it; we register as an issuer)
- **Cross-platform** (other agentic-AI platforms can register too —
  reputation becomes portable across the ecosystem)

The integration cost is small — one issuer registration, one Receipt
write per job, one Credential read per routing decision. The
differentiator vs. competitors (Bittensor / io.net / Akash, none of
which have a portable cross-platform reputation) is large.

## Specification

### CPR concepts (verbatim from the existing schema)

| Concept | Schema | Role |
| --- | --- | --- |
| **Reputation Issuer** | `reputation_issuers` (did, name, domain, api_key_hash) | Registered platforms that submit Receipts. Infernet registers as one. |
| **Reputation Receipt** | `reputation_receipts` (receipt_id, task_id, agent_did, buyer_did, platform_did, amount, currency, sla, outcome, signatures) | Immutable record of one task. |
| **Reputation Credential** | `reputation_credentials` (agent_did, credential_type, data, window_start, window_end, signature) | Computed attestation, e.g. "95% acceptance, 30-day window". |
| **Merchant DID** | `merchant_dids` (merchant_id, did, public_key) | Every CoinPay merchant has one — Infernet operators automatically inherit one via their CoinPay payout setup. |

### DID conventions

Three DID method usages:

| Identity | DID format | Resolves to |
| --- | --- | --- |
| **Platform** | `did:web:infernetprotocol.com` | `https://infernetprotocol.com/.well-known/did.json` |
| **Self-hosted platform** | `did:web:<their-domain>` | `https://<their-domain>/.well-known/did.json` |
| **Operator (provider/client)** | `did:nostr:<hex64-pubkey>` | The Nostr pubkey itself; resolution = signature verification |

`did:nostr:` is a documented DID method that wraps a Nostr pubkey
and uses Schnorr signatures for verification. If method
standardization is incomplete at integration time, fall back to
`did:key:` with the same pubkey encoded per the multibase spec.

The control plane publishes its DID document at
`/.well-known/did.json`:

```json
{
  "id": "did:web:infernetprotocol.com",
  "verificationMethod": [{
    "id": "did:web:infernetprotocol.com#key-1",
    "type": "Ed25519VerificationKey2020",
    "controller": "did:web:infernetprotocol.com",
    "publicKeyMultibase": "<multibase-encoded-pubkey>"
  }],
  "service": [{
    "id": "did:web:infernetprotocol.com#cpr",
    "type": "CPRIssuer",
    "serviceEndpoint": "https://infernetprotocol.com/api/cpr"
  }]
}
```

### Issuer registration

A one-time setup step: Infernet registers as a Reputation Issuer
with CoinPay.

```jsonc
POST https://coinpayportal.com/api/cpr/issuers
{
  "did":     "did:web:infernetprotocol.com",
  "name":    "Infernet Protocol",
  "domain":  "infernetprotocol.com",
  "api_key": "<bearer-from-coinpay>"
}
→
{
  "issuer_id": "<uuid>",
  "active":    true
}
```

Each self-hosted control plane registers separately if they want
their receipts attested by CoinPay (most will; it's the network
effect). Self-host operators who want full sovereignty can either
issue their own credentials or skip CPR entirely and run unsigned.

### Receipt writing

On every job completion, the control plane writes a Receipt to CPR.
This happens server-side after the existing
`/api/v1/node/jobs/[id]/complete` flow finalizes:

```jsonc
POST https://coinpayportal.com/api/cpr/receipts
Authorization: Bearer <issuer-api-key>
{
  "receipt_id":    "<random-uuid>",
  "task_id":       "<infernet-job-uuid>",
  "agent_did":     "did:nostr:<provider-pubkey>",
  "buyer_did":     "did:nostr:<client-pubkey>",
  "platform_did":  "did:web:infernetprotocol.com",
  "escrow_tx":     "<coinpay-payment-tx-id>",
  "amount":        0.034,
  "currency":      "USDC",
  "category":      "inference:chat",
  "sla": {
    "model":               "qwen2.5:7b",
    "max_tokens":          512,
    "first_token_target":  1500,
    "first_token_actual":  890,
    "total_tokens":        421
  },
  "outcome":      "accepted",
  "dispute":      false,
  "artifact_hash": "<sha256-of-full-token-stream>",
  "signatures": {
    "provider": "<provider-schnorr-over-receipt>",
    "client":   "<client-schnorr-over-receipt>",
    "platform": "<platform-ed25519-over-receipt>"
  }
}
```

The receipt is signed three ways:

1. **Provider** signs with their Nostr privkey — proves they served the job.
2. **Client** signs (or the platform proxies on the client's behalf for unauthenticated jobs) — proves they accepted the result.
3. **Platform** signs with the issuer key — proves Infernet attested it.

Outcomes:

- `accepted` — client got the result, no objection. Default if no dispute filed within window.
- `rejected` — client refused the result (no payment, no reputation hit if rejection itself was unjustified — see disputes).
- `disputed` — formal dispute filed; resolution comes through CPR's dispute pipeline; reputation impact is held until resolved.

### Router integration

When the control plane's job router scores candidate providers, it
queries CPR for each candidate's current Credentials:

```jsonc
GET https://coinpayportal.com/api/cpr/credentials?agent_did=did:nostr:<pubkey>&category=inference:chat
→
{
  "credentials": [
    {
      "credential_type": "task_volume_30d",
      "data":           { "accepted": 4127, "rejected": 31, "disputed": 4 },
      "window_start":   "2026-03-26T00:00:00Z",
      "window_end":     "2026-04-26T00:00:00Z",
      "signature":      "..."
    },
    {
      "credential_type": "first_token_p95_30d",
      "data":           { "ms": 740 },
      "window_start":   "2026-03-26T00:00:00Z",
      "window_end":     "2026-04-26T00:00:00Z"
    }
  ]
}
```

The router scoring function (rough form):

```
score(provider) =
    base_score
  + w1 * sigmoid(accepted_30d / 100)        // recent volume
  + w2 * acceptance_rate_30d                // quality
  + w3 * (1 / (1 + first_token_p95))        // latency
  - w4 * dispute_rate_30d                   // trust prior
  + w5 * (1 if model_match else 0)
  + w6 * (1 if payment_match else 0)
```

New providers (no credentials yet) get a small bootstrap allocation
— enough to generate ~10 receipts so they can build a track record —
but capped by their stake / payment escrow until they have signal.

### Disputes

Clients filing a dispute hit the existing CPR dispute endpoint.
Resolution:

1. Client posts evidence (output didn't match prompt, model wasn't
   served, etc.) to `/api/cpr/disputes`.
2. CPR's dispute resolver (pluggable; default is a 7-day jury vote
   among reputation-weighted CPR participants) decides.
3. Provider's reputation is updated; payment is refunded or
   released accordingly (CoinPay holds escrow).

Infernet doesn't reimplement disputes — we just emit Receipts with
the right `dispute=true` flag and let CPR handle it.

### Self-hosted operator path

A control plane operator who clones Infernet and runs their own
deployment has three options:

1. **Plug into public CPR** — register their `did:web:<their-domain>`
   as a CPR issuer, write Receipts to coinpayportal.com. Their
   provider reputation flows back to the same network as ours.
2. **Self-host CPR alongside** — run their own CoinPay deployment;
   their reputation is local to their network. Compatible with
   IPIP-0005 (everything is `did:web` so resolution is uniform).
3. **Skip reputation entirely** — emit no Receipts. Their network
   has no Sybil resistance; tradeoff is theirs to make.

This IPIP doesn't require any of the three; it just specifies
options 1 and 2.

## Rationale

**Why CPR specifically and not, say, a custom on-chain reputation
contract?** CPR exists, is documented, and integrates cleanly with
DIDs we already plan to use. An on-chain alternative would cost
gas per receipt (millions of jobs → millions of fees). CPR's
off-chain receipts with cryptographic signatures are the right
trade-off for high-volume per-job reputation.

**Why three signatures (provider / client / platform)?** Each
attests to a different fact:

- Provider sig: "I performed this work."
- Client sig: "I accepted the result."
- Platform sig: "I observed both, time-stamped, and forwarded to CPR."

Removing any one weakens the system. Provider-only = clients can be
defrauded. Client-only = providers can be falsely flagged. Platform-
only = trust the platform fully. All three = each party stakes their
identity on the receipt.

**Why `did:web` for the platform and `did:nostr:` for operators?**
Platform identity benefits from HTTPS-anchored verification (anyone
can fetch the DID document and verify). Operator identity is
already a Nostr pubkey; reusing it avoids a second key the operator
has to manage.

**Why not just rebuild reputation in our own DB?** We could. But
the network effect of CPR is the differentiator: an operator's
reputation built on Infernet is portable to any other CPR-integrated
platform, and vice versa. That's much harder to bootstrap if every
agent platform has its own siloed reputation.

## Backwards compatibility

- New external dependency: CoinPay's CPR API. Failures (CoinPay
  down, network partition) MUST NOT block job completion — the
  control plane queues Receipts locally and retries on a back-off.
  Routing falls back to the existing scoring (load, last-seen,
  basic acceptance rate from local DB) if Credentials are
  unavailable.
- Existing `provider_payouts` / `payment_transactions` schemas
  unchanged. New columns added to `jobs` to track receipt-id and
  receipt-status (`pending`, `submitted`, `acked`, `disputed`).
- Operators who decline to register with CPR (option 3 above) lose
  reputation routing benefits but otherwise function normally.

## Reference implementation

Phased.

**Phase 1 — Issuer registration.**
- One-time setup: register `did:web:infernetprotocol.com` with
  CoinPay; store the issuer API key as `CPR_ISSUER_API_KEY` server
  env. Publish the DID document at `/.well-known/did.json`.

**Phase 2 — Receipt writing.**
- On `/api/v1/node/jobs/[id]/complete`, after persisting the
  job's terminal status, generate a Receipt, sign it (provider sig
  carried through from the daemon's complete payload; platform sig
  added server-side; client sig deferred to the SSE client's `accept`
  call), and POST to CPR with retry/back-off. Store the receipt-id
  on the `jobs` row.

**Phase 3 — Router integration.**
- The job router fetches Credentials for each candidate provider
  via CPR; combines with local signals to produce a score; weights
  configurable. Cache Credentials with a short TTL (60 s) to avoid
  per-request CPR calls.

**Phase 4 — Client receipt acceptance.**
- The dashboard / SDK / CLI emit a "I accept this result" signature
  after the SSE stream finishes (or after a configurable timeout if
  the client doesn't actively confirm). Update the receipt with the
  client signature.

**Phase 5 — Dispute UX.**
- Dashboard surface for filing a dispute on a specific job. Talks
  to CPR's dispute API; status pulled back into the dashboard for
  observability.

## Test plan

1. **Issuer registration**: control plane successfully registers as
   a CPR issuer; DID doc resolves at `/.well-known/did.json`;
   signature against the doc's verification key succeeds.
2. **Receipt round-trip**: a completed job produces a Receipt
   visible in CoinPay's `reputation_receipts` table within 5 s of
   completion; all three signatures verify.
3. **Outage tolerance**: CPR API hard-down → jobs still complete on
   Infernet; receipts queue locally; once CPR returns, queue drains
   in order without dupes.
4. **Router scoring**: with synthetic Credentials simulating
   providers A (high rep) and B (low rep) and equal everything else,
   the router prefers A on N=100 trials at p > 99%.
5. **Disputes**: filing a dispute on a job moves the receipt to
   `dispute=true`; resolution flows back into the provider's
   credentials window.
6. **Self-host parity**: a self-hosted control plane configured
   with `CPR_ISSUER_API_KEY` participates in the same reputation
   network; an operator linked to both the public Infernet and a
   self-host accumulates reputation across both.

## Security considerations

- **Issuer key compromise**: the `CPR_ISSUER_API_KEY` is the platform's
  ability to write Receipts. Compromise = an attacker can forge
  reputation events. Mitigation: rotate quarterly, store in
  Doppler / secret manager, audit-log every receipt write.
- **Receipt forgery via stolen client/provider key**: holding either
  side's Nostr privkey lets an attacker sign one half of the receipt
  in their favor. The platform signature is the third anchor — both
  sides have an audit trail, and CPR's dispute resolution adjudicates.
- **Replay**: `receipt_id` is unique-constrained at CoinPay. A
  duplicate POST gets 409 Conflict.
- **CPR endpoint impersonation**: control plane MUST verify the
  TLS cert chain for `coinpayportal.com` and pin the public key in
  CI as defense-in-depth.

## Privacy considerations

- **Receipt contents are persistent and visible to CoinPay.** Every
  Infernet job's amount, parties (DIDs only — no real-world names
  unless an operator self-publishes), and SLA outcome are recorded.
  Operators wanting privacy across this surface need separate
  Nostr keypairs per "persona" they want unlinked.
- **Aggregate-vs-individual leakage**: Credentials are aggregated
  attestations ("95% acceptance over 30 days"), but Receipts are
  per-task. A nation-state-level adversary could observe Receipts
  to deanonymize compute providers' patterns. Pair with Tor
  outbound (IPIP-0001 v1 in-scope) for full hygiene.
- **Cross-platform linkability**: the upside of CPR (reputation
  portable across platforms) is also a privacy edge — a node that
  works on both Infernet and another CPR-integrated platform is
  trivially linkable across both. Operators wanting unlinkability
  use distinct DIDs per platform.

## Open questions

- **Bootstrap allocation policy**: how much routing should a
  zero-credential provider receive? Too little and new operators
  starve; too much and Sybils exploit. Probably tied to a small
  payment escrow they post on registration (lost if they
  consistently underperform).
- **Default dispute window**: 24 h? 7 days? Long enough that
  legitimate complaints surface; short enough that providers
  aren't chronically under hold.
- **Credential cache TTL**: 60 s feels right for a high-volume
  router. Investigate if CPR can webhook us when Credentials change
  to avoid even that polling.
- **Should we expose Credentials in the dashboard publicly?** Pro:
  transparency, "look at our top providers." Con: doxxes whatever
  identity an operator wants kept quiet. Lean toward a per-operator
  visibility setting.
