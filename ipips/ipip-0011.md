## Abstract

Defines the pluggable training orchestration interface used by Infernet
nodes that opt into Class B / B.5 / C training jobs (IPIP-0010). Mirrors
the inference engine adapter (IPIP-0009) but for long-running training
runs: events for `step`, `checkpoint`, and `eval` instead of `token`.
Backends include DeepSpeed (Class B), OpenRLHF (Class B + vLLM), Petals
(Class B.5 fine-tunes), and OpenDiLoCo (Class C).

## Motivation

Training is fundamentally different from inference: jobs run for hours
to weeks, emit incremental progress (loss, learning rate, throughput),
checkpoint to durable storage, and may continue across daemon restarts.
A separate protocol with training-specific event types is cleaner than
overloading the inference protocol's `token` events.

We also need to be honest with operators about what's available today
versus what's scaffolded. Each backend's interface exists as a
placeholder so the scheduler, dashboard, and event store can be built
against a stable contract while Python integration work happens
incrementally.

## Specification

### Trainer factory

Implementations MUST expose `createTrainer(opts)` returning an object
with this shape:

```ts
interface Trainer {
  kind: "deepspeed" | "openrlhf" | "opendiloco" | "petals" | "stub";
  start(req: { id?: string; config: TrainingConfig }): {
    id: string;
    stream: AsyncIterable<TrainingEvent>;
    cancel(): void;
  };
  shutdown(): Promise<void>;
}
```

Auto-selection: `process.env.INFERNET_TRAINING_BACKEND` if set, else
`"stub"` until per-backend runtime detection lands.

### Training config

```ts
interface TrainingConfig {
  kind: "fine-tune" | "lora" | "rlhf-rollout" | "pretrain";
  base_model: string;                            // e.g. "Qwen/Qwen2.5-7B"
  dataset: { url: string; sha256?: string; format?: string; split?: string };
  hyperparams?: { lr?: number; batch_size?: number; epochs?: number; max_steps?: number };
  output: { destination_url: string; format?: string };
}
```

### Event protocol (v1)

NDJSON, every message carries `v: 1`, `type`, and (except engine-level
errors) `id`.

| Type | Fields | Meaning |
|---|---|---|
| `meta` | `id`, `started_at`, `backend`, `world_size?`, `base_model?`, `kind?`, `cluster?` / `swarm?` | First event; describes the run |
| `step` | `id`, `step`, `loss`, `lr?`, `throughput?`, `grad_norm?` | Per training step |
| `checkpoint` | `id`, `step`, `path`, `size_bytes?` | Persistent snapshot saved |
| `eval` | `id`, `step`, `metrics: object` | Evaluation pass results |
| `done` | `id`, `reason: "complete"\|"cancel"\|"error"\|"not_implemented"`, `final_step?`, `finished_at`, `note?` | Terminal |
| `error` | `id?`, `message` | Terminal failure |
| `log` | `level: "info"\|"warn"\|"error"`, `message` | Free-form log line |

`reason: "not_implemented"` is reserved for placeholder backends (the
state most backends are in today). The scheduler MUST treat
`not_implemented` as a non-retryable failure and refund any escrow.

### Step emission cadence

Backends SHOULD emit `step` events no more frequently than every ~5
seconds during long runs to keep storage bounded. Rapid bursts (e.g. a
warmup phase) MAY skip intermediate steps.

### Checkpoint storage

Checkpoints written by training jobs MUST be addressable via the URL
in the `path` field. Implementations MAY use any of: local filesystem
(`file://`), S3 (`s3://`), HuggingFace Hub (`hf://`), or IPFS
(`ipfs://`). The scheduler does not inspect checkpoint contents.

### Cancellation

`cancel()` MUST cause the in-flight run to terminate with a `done` event
of `reason: "cancel"`. Backends SHOULD attempt to write a final
checkpoint before terminating.

## Rationale

**Why separate from the engine protocol (IPIP-0009)?** Training jobs and
inference jobs have different lifecycles, different event shapes (step
vs. token), different storage needs (checkpoints), and run on different
schedules. Keeping the protocols separate prevents the union from
becoming a kitchen-sink superset.

**Why allow `reason: "not_implemented"`?** We're shipping the surface
ahead of the implementations to bound the integration work and let
downstream code (dashboards, schedulers) be built today against a stable
contract. A backend that returns `not_implemented` is a placeholder by
design; the scheduler treating it as a failure is the correct behavior.

**Why not auto-detect backends?** DeepSpeed/OpenRLHF/OpenDiLoCo/Petals
are Python libraries that require a Python venv with specific
dependencies installed. Detecting "is X usable" reliably means importing
the module, which is heavyweight. Explicit env-var selection is honest
and unambiguous.

## Backwards compatibility

None — new package, new protocol, no prior surface to maintain.

## Reference implementation

- `packages/training/src/protocol.js` — `TRAINING_PROTOCOL_VERSION`, `TMSG`
- `packages/training/src/index.js` — factory + auto-select
- `packages/training/src/backends/{stub,deepspeed,opendiloco,petals,openrlhf}.js`
  — initial set; only `stub` returns synthetic step events; the others
  return `meta` + `done(not_implemented)` until their Python integrations
  land.
- `tests/training-backends.test.js`

## Test plan

- Unit: each backend's factory returns `kind === "<backend>"` and
  `start()` produces the expected event sequence.
- Integration: a Class-B job submitted with `backend: "stub"` runs end
  to end and writes one synthetic checkpoint event.
- Real backends: each gets its own integration test once the Python
  layer is wired in.

## Security considerations

Training jobs may consume substantial compute (hours to weeks of GPU
time) and emit checkpoints to operator-controlled storage. Escrow
amounts MUST be sized to cover expected duration, and the scheduler
MUST refund on `not_implemented` / `cancel` outcomes.

Cross-provider training (Class C) introduces new attack surface: a
malicious peer could send poisoned gradient deltas. OpenDiLoCo's design
mitigates this with per-step robust aggregation, but any new training
backend SHOULD document its threat model in its own follow-up IPIP.

## Privacy considerations

Training datasets and model weights are typically the operator's primary
IP. The protocol carries dataset URLs (which may be public or private)
and checkpoint paths but never inspects contents. Operators SHOULD use
private storage URLs and sign signed-URL pre-flight when their dataset
is sensitive.

## Open questions

- Should `step` events include per-GPU breakdown (`world_step` vs.
  `local_step`) for distributed runs?
- How do we represent fine-tuning lineage (this run started from
  checkpoint X of base Y)? A separate `lineage` event, or a field in
  `meta`?
