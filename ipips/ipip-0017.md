## Abstract

Specifies how Infernet replicated state — DHT records, reputation
scores, fingerprint→provider observations, capability advertisements
— converges across replicas after concurrent writes and partition
heals. Replaces IPIP-0016's last-write-wins-by-Lamport rule (which
silently drops concurrent writes) with a per-record-type choice
between LWW (acceptable for single-writer state) and CRDT primitives
(required for multi-writer state). Adopts a small set of standard
CRDTs — G-Counter, PN-Counter, OR-Set, LWW-Register, LWW-Element-Set
— and maps each Infernet replicated state to the appropriate one.

## Motivation

In a distributed system without global time, two concurrent updates
to the same logical record can't be ordered. The system's choices
are:

1. **Refuse concurrent writes** (CP — what Postgres does inside the
   control plane).
2. **Pick one winner** (LWW — last-write-wins, breaks ties by
   timestamp + tiebreaker; loses the loser's update silently).
3. **Merge them** (CRDT — operations commute by construction so the
   merged state is the same regardless of arrival order).

IPIP-0016 §7 currently mandates LWW-by-Lamport for DHT partition-
heal merges. That's correct *only when one writer per record*. For
multi-writer records — reputation scores updated by multiple CPR
validators, fingerprint→provider observations published by every
peer that has seen a model, capability flags possibly toggled by
multiple admins — LWW silently drops valid updates.

The fix is well-known: CRDTs (Conflict-free Replicated Data Types,
Shapiro et al. '11) guarantee strong eventual consistency. Every
replica that receives the same set of operations converges to the
same state, regardless of the order or how many times they're
delivered. Operations are designed to commute, idempotent, and
associative.

We don't need every CRDT in the literature. We need a small,
well-chosen set that maps cleanly to our replicated-state shapes.

## Specification

### 1. Per-record-type convergence policy

Every record type stored in the DHT, in `pubkey_links`, in CPR
state, or in any other replicated table MUST declare which
convergence policy it uses:

| Policy | When applicable | What happens on concurrent writes |
|---|---|---|
| **LWW** | Single-writer or "writer's most-recent intent" semantics | Latest timestamp wins; older value lost |
| **CRDT** | Multi-writer, each contribution must be preserved | Operations merge by CRDT rules; all contributions counted |
| **CP** | Strict consistency required (payments, escrow) | Refuse concurrent writes; serialize via Postgres ACID |

Record schema MUST include a `convergence` field:

```json
{
  "convergence": "lww" | "crdt:<type>" | "cp",
  ...record fields...
}
```

Receivers reject records with missing/unknown `convergence`.

### 2. Adopted CRDT primitives

#### G-Counter (Grow-only counter)

State: `{[peer_id]: count}` per replica; replica increments only
its own slot. Merge: pairwise max of slots. Read: sum of slots.

Used for: monotonically-increasing counters that no replica ever
decrements (job-completed counts per provider, requests-served
counts, total-bytes-relayed for billing audit).

#### PN-Counter (Positive-Negative counter)

State: two G-Counters, one for increments, one for decrements.
Read: positive-G - negative-G.

Used for: reputation scores (CPR — IPIP-0007 — currently treats
reputation as a single number; with multiple validators voting up
or down, PN-Counter converges correctly).

#### OR-Set (Observed-Remove Set)

State: each element tagged with a unique add-tag at insert time;
remove records the set of tags currently observed. An element is
present iff it has any add-tags not matched by remove-tags.

Used for: fingerprint→provider observations ("which providers have
been seen serving fingerprint X?"). Multiple peers add the same
observation independently; concurrent removes don't conflict with
new adds.

#### LWW-Register

State: `{value, timestamp, tiebreaker}`. Merge: keep entry with
higher (timestamp, tiebreaker). Lamport timestamp + provider-id
tiebreak (per IPIP-0014 §5).

Used for: provider-owned single-writer state (a provider's own
served_models list, advertised price, public address). The
provider IS the only writer; concurrent writes only happen across
network partitions that produced two versions of the same
provider's intent — LWW picks the latest reasonably.

#### LWW-Element-Set

State: a set where each element carries an LWW-Register of
add/remove flag. An element is present iff its latest operation
was an "add". Merge per element via LWW-Register rule.

Used for: provider-controlled small sets (e.g. an operator's
list of trusted bootstrap peers). Same single-writer caveat as
LWW-Register.

### 3. Mapping Infernet replicated state → policy

| Replicated state | Policy | Reasoning |
|---|---|---|
| **DHT provider announcement** (IPIP-0016 §3a) | LWW-Register | Single writer (the provider). Republished every 10 min; LWW converges fine. |
| **DHT model fingerprint registry** (IPIP-0016 §3b) | OR-Set of (provider, fingerprint) | Multi-writer (every peer that has observed a model can publish). Concurrent observations of the same model from different peers must all count. |
| **DHT job announcement** (IPIP-0016 §3c) | LWW-Register | Single writer (the job poster). |
| **Reputation score (CPR)** | PN-Counter | Multi-writer (every validator can vote up or down). PN-Counter is precisely the CRDT for this shape. |
| **`served_models` per provider** | LWW-Register | Single writer (the provider). |
| **`pubkey_links` user↔node** | CP (Postgres ACID) | Authority is the control plane; never replicated. |
| **`payment_transactions`** | CP (Postgres ACID) | Strict — never CRDT under any circumstance. |
| **`provider_payouts`** | CP (Postgres ACID) | Same as above. |
| **Job state machine** | CP (Postgres ACID) | Strict. Out-of-order delivery handled by IPIP-0014 §6 state-transition guards. |
| **`job_events`** (IPIP-0014 §5 / IPIP-0015 §1) | Append-only with causal stamps | Not technically a CRDT — events are immutable; the append-only set IS naturally CRDT-like (G-Set semantics). |

### 4. Implementation contract

For each adopted CRDT, the SDK MUST expose two operations:

```ts
type CRDT<S, O> = {
  apply(state: S, op: O): S;          // local op → new state
  merge(a: S, b: S): S;               // partition-heal → converged state
};
```

Properties the implementation MUST satisfy (verified by tests):

- **Commutative**: `merge(a, b) = merge(b, a)`
- **Associative**: `merge(merge(a, b), c) = merge(a, merge(b, c))`
- **Idempotent**: `merge(a, a) = a`

These three properties together imply strong eventual consistency:
any two replicas that have observed the same set of operations
(in any order, with any duplication) converge to the same state.

### 5. Wire format

CRDT records carry their state in a normalized form. For each
type:

```
G-Counter:        { type: "g-counter",  slots: { [peer]: count } }
PN-Counter:       { type: "pn-counter", inc: G-Counter, dec: G-Counter }
OR-Set<T>:        { type: "or-set",     adds: [{tag, val}], removes: [tag] }
LWW-Register<T>:  { type: "lww-reg",    value: T, ts: lamport, tb: peer_id }
LWW-Element-Set:  { type: "lww-set",    elements: { [val]: LWW-Register<bool> } }
```

Records are Nostr-signed envelopes (per IPIP-0016 §5) wrapping
the JSON state. Receivers verify the envelope before applying
merge.

### 6. Garbage collection for OR-Set tombstones

OR-Set's "remove" leaves tombstone tags forever; over time they
bloat the state. Standard mitigation:

- Every replica periodically (default: every 24h) runs a GC pass:
  - Collect tags older than the GC horizon (default: 30 days)
  - If all replicas have observed those tags, drop them
  - Coordination via the DHT's "observed-up-to" gossip

Operators MAY tune `GC_HORIZON_DAYS` if their use case differs.

### 7. Backwards compatibility with IPIP-0016

This IPIP **amends** IPIP-0016 §7's blanket "LWW by Lamport"
rule. The amendment:

> Records SHOULD declare their convergence policy via the
> `convergence` field. For backwards compatibility, records that
> omit it default to `lww` — preserving IPIP-0016's original
> behavior. New record types (especially multi-writer ones) MUST
> declare `crdt:<type>` and use the corresponding merge rule.

Existing single-writer record types (provider announcement, job
announcement) keep LWW. The fingerprint registry — which I
originally specified as a single LWW record — gets upgraded to
OR-Set since it's genuinely multi-writer.

## Rationale

**Why not full Yjs / Automerge?** Both are mature, well-tested
CRDT libraries with rich data model support (text, lists, JSON
trees). Overkill for our use cases — we have a small fixed set
of CRDT shapes (counter, set, register), and rolling our own
keeps the wire format under our control and the dependency
surface small. Reconsider when we need rich-text collaboration
(e.g. operator-edited shared docs).

**Why call out the strict-CP cases?** Payments and job-state are
the things that must NEVER be CRDT. Without an explicit policy
field and review enforcement, future endpoints could
inadvertently CRDT-ify a payment record (e.g. someone "merging"
two concurrent settlement attempts into a sum). The policy field
is the review gate.

**Why is reputation a PN-Counter and not a more sophisticated
CRDT?** PN-Counter is the textbook fit for "any peer can
increment or decrement, all contributions count, eventual
consistency is fine." Reputation IS this shape. Sophistication
without a real need is just complexity.

**Why does `served_models` per provider stay LWW?** A provider's
own list of models served is a single-writer record — only that
provider updates it. Concurrent writes only arise across
partitions that produced two versions of the same provider's
intent; LWW with Lamport timestamps picks the latest, which is
what we want. CRDT-ifying a single-writer record would just
introduce overhead.

## Backwards compatibility

Additive at the wire-format level: existing records lacking
`convergence` default to `lww`, preserving IPIP-0016 behavior.
New record types declare `crdt:<type>` explicitly.

Schema migration is small: add a nullable `convergence` column
(or JSON field) to relevant tables. Default-fill `'lww'` for
existing rows.

The fingerprint registry record type is being upgraded mid-
flight from LWW-Register to OR-Set. Records published as LWW-
Register before this IPIP land continue to be readable; the
control plane re-publishes them as OR-Set seeds during the
transition.

## Reference implementation

To land:
- `packages/crdt/` — new package: G-Counter, PN-Counter, OR-Set,
  LWW-Register, LWW-Element-Set in TypeScript with property-based
  tests for commutativity / associativity / idempotence.
- `packages/dht/lib/merge.js` — replaces the LWW-only merge with
  policy-dispatched merge (calls `packages/crdt`).
- `apps/web/lib/cpr/score.js` — refactor reputation score from
  scalar to PN-Counter. Migration script to backfill existing
  scalar scores into PN-Counter form.
- `supabase/migrations/00xx_convergence_policy.sql` — adds
  `convergence` column to the relevant tables.
- `packages/sdk-js/src/crdt.ts` — opt-in SDK exports for
  external tools that want CRDT semantics directly.

## Test plan

- **Property-based unit tests** (using fast-check or similar):
  for each CRDT type, generate random sequences of operations
  and assert that all replicas converge to the same state
  regardless of operation interleaving / duplication.
- **Integration: DHT partition-heal.** Spin up 5 nodes; partition
  into two halves; both halves accept multi-writer operations
  (e.g. observations of fingerprint X); heal partition; assert
  every node sees every observation.
- **Migration test**: convert a synthetic scalar reputation
  score (say, 1273) to PN-Counter form and verify reads return
  the same number.
- **Negative test**: a record marked `crdt:pn-counter` whose
  state field has a malformed shape MUST be rejected at
  signature-verify time.

## Security considerations

- **Forged operations.** A malicious peer publishing fake CRDT
  ops (e.g. crediting itself reputation) is signature-verified
  per IPIP-0016 §5; the operation is recorded as the publisher's
  intent regardless. Mitigation: per-publisher rate limits,
  CPR-weighted operation acceptance (only count ops from
  validators with reputation > floor).
- **OR-Set tombstone DoS.** A malicious peer could spam removes
  + adds to bloat the OR-Set. Mitigation: per-publisher quota
  on operations per record per time window.
- **Counter overflow.** G-Counter slots are unbounded — never
  overflow in practice (64-bit integers), but spec the
  representation as bigint to avoid JS implicit-conversion
  surprises.

## Privacy considerations

CRDT records expose more *operation history* than LWW records
(an OR-Set carries every observed `add` tag forever, until GC).
For fingerprint→provider observations this is fine — observations
are public information. If we ever want a "private" multi-writer
record type we'd need encrypted OR-Set with capability-based
access; out of scope for v1.

## Open questions

- **Tombstone-free CRDTs.** δ-CRDTs and pure operation-based
  CRDTs avoid the tombstone bloat entirely but require reliable
  causal broadcast (IPIP-0018) underneath. Fold into v2 once
  IPIP-0018 ships.
- **CRDTs in payments.** We've explicitly excluded payments from
  CRDT-ification. Is there ANY payment-adjacent state that *could*
  be a CRDT? (Withdrawal-credit ledger across multiple validators?)
  Probably not in v1; revisit if we ever want decentralized escrow.
- **Total-order broadcast (TOB) for sequences.** Some applications
  want a strict total order on operations (e.g. operator chat
  history). CRDTs alone don't give total order; we'd layer in
  IPIP-0018's causal broadcast + a total-order sequencer. Not
  a v1 requirement.
