## Abstract

Specifies how Infernet captures and verifies causal relationships
between events that span multiple providers — the case Lamport's
1978 paper (*Time, Clocks, and the Ordering of Events in a
Distributed System*) addresses with logical clocks and the strong
clock condition. IPIP-0014 §5 already mandates per-provider Lamport
timestamps with the IR1+IR2 update rules. This IPIP layers on
**causal dependency stamps** (sparse vector clocks expressed as
parent-event-ID lists) for events whose ordering matters across
providers, plus an **anomaly detector** that flags impossible
orderings, and reserves the **distributed mutual exclusion**
primitive for future peer-to-peer coordination.

## Motivation

Per-provider Lamport seq (IPIP-0014 §5) gives a partial order
within a single provider's event stream. It does NOT answer
questions like "did peer B's start of layer 13 happen-after peer
A's finish of layer 12?" — those events come from different
providers and Lamport timestamps alone, even with IR2(b)
synchronization, only satisfy `a → b ⟹ C(a) < C(b)` (one
direction). The converse doesn't hold: two events with `C(a) <
C(b)` may be concurrent rather than causally ordered.

Three concrete places this matters:

1. **Class B.5 Petals chains.** Peer A finishes layer 12 → peer B
   starts layer 13 → peer C starts layer 14. Each provider's
   `provider_seq` is monotonic locally, but at the control plane
   the events arrive interleaved. We need to *prove* "B's layer-13
   start was caused by A's layer-12 finish" — for replay, for
   anomaly detection, and for assigning per-hop reputation when
   the chain fails.
2. **Class C training delta exchange.** OpenDiLoCo workers
   exchange model deltas; the control plane MUST be able to tell
   whether two deltas are sequential applications or concurrent
   merges. Lamport timestamps alone are ambiguous (concurrent
   events can have any relative timestamp ordering).
3. **Multi-batch chunk completion.** When IPIP-0013 batch chunks
   complete out of order, the manifest assembler needs to know
   which chunks were retried (causally dependent on a prior
   failure) versus which were independent first attempts.

Full vector clocks (one counter per peer, attached to every
event) are O(N) per message and don't scale to a 10k-provider
network. The pragmatic answer is sparse vector clocks: each
event explicitly names the IDs of events it causally depends
on, like Git commit parents. Most events have one parent;
merge events have several.

## Specification

### 1. Causal dependency stamps

Every cross-provider event added to `job_events` MUST carry an
explicit list of parent event IDs:

Schema additions (extending IPIP-0014 §5):

```sql
ALTER TABLE job_events ADD COLUMN parent_event_ids uuid[] DEFAULT '{}';
ALTER TABLE job_events ADD COLUMN lamport_ts bigint;       -- already added by 0014 §5
CREATE INDEX idx_job_events_parents
  ON job_events USING gin (parent_event_ids);
```

Each parent ID names an event that causally precedes this one.
Single-parent events (the common case — one local action followed
by another) carry one ID. Merge events (e.g. an aggregator
combining deltas from N workers) carry N IDs.

Required parent semantics:
- **Local sequential events**: parent = the last event this provider emitted on the same `(job_id, provider_id)` stream.
- **Cross-provider receive**: parent = the event that caused the
  message being received. The remote sender encodes its own
  emitting event ID in the message envelope; the receiver picks
  it up from there.
- **Merge events**: parents = all of the events whose results are
  combined.

### 2. Concurrent vs happens-before

The dashboard, replay tools, training-run inspectors, and any
consistency-check code MUST expose the relation:

```ts
type CausalRelation = "happens_before" | "happens_after" | "concurrent";

function eventRelation(a: Event, b: Event): CausalRelation;
```

Implementation: walk the `parent_event_ids` graph from `a`. If
`b` is reachable, `a happens_after b`. If `a` is reachable from
`b`, `a happens_before b`. Otherwise `concurrent`.

**Concurrent ≠ simultaneous.** Two events being "concurrent"
in this protocol does NOT mean they happened at the same
physical instant. It means the system is *unaffected by the
order in which they're processed* — the final state is
identical regardless. Force-ordering concurrent events by
wall-clock or by Lamport seq is a legitimate display choice
(operator wants a stable sort for the UI) but MUST NOT be
used by any correctness-critical code path. The relation
function returns the truth.

The practical implication: replay tools, training-run inspectors,
and CRDT merge code (IPIP-0017) treat concurrent operations as a
SET, not a sequence. They commute by definition.

### 2a. Sparse vs full vector clocks

The course-textbook approach is **full vector clocks** (Mattern
'88, Fidge '88): every process maintains a vector of N counters
(one per peer in the system); every message carries the sender's
full vector; on receive, the receiver increments its own counter
and takes the pairwise max of every other entry.

Full vector clocks have one elegant property our sparse stamps
don't: from any pair of clock vectors `Va` and `Vb`, you can
*directly* tell the relation by component-wise comparison:

```
Va < Vb  iff  ∀i: Va[i] ≤ Vb[i]  AND  ∃i: Va[i] < Vb[i]
```

If `Va < Vb`, then `a happens_before b`. If neither vector
strictly dominates the other, the events are concurrent. No
graph walk required.

The cost: O(N) space per event and per message, where N = peer
count. At our 10k-provider target this is ~80 KB per event,
prohibitive on the wire and in storage.

**Sparse stamps (this IPIP's choice)** trade O(N) space for
O(parent_count) space and a graph-walk to answer the same
question. Same expressive power, dramatically smaller payload.
The trade-off is fine because we only ask the
`a happens-before b?` question on demand (anomaly detector,
replay UI), not on every event-emission hot path.

Conversion is easy if we ever need it: a sparse stamp can be
projected to a full vector by walking the parent graph and
collecting per-provider max counters. We expose
`vectorClock(event)` as an opt-in projection in the SDK for
external tools that want vector-clock semantics directly.

### 3. External causal tokens (cross-channel)

For workflows that span channels outside Infernet's control
(payment webhooks, external eval signals, partner integrations),
Lamport timestamps and parent_event_ids are not enough — we don't
sign the external party's events.

Each cross-channel workflow MUST issue an explicit token at the
boundary point and require it on every subsequent message in the
workflow:

| Workflow | Token | Issued at | Required on |
|---|---|---|---|
| Pay-then-complete | `payment_intent_id` | invoice creation | webhook + job-complete |
| Verify-then-claim | `verification_token` | code-send | claim endpoint |
| Eval-then-train | `eval_run_id` | eval submission | training start + each step |

The token threads the causal chain across the system boundary
that Lamport's logical clocks can't cross. See IPIP-0014 §10
for the payment-specific implementation.

### 4. Impossible-ordering anomaly detector

A periodic janitor (default cadence: 5 minutes) MUST scan recent
`job_events` for impossible orderings:

```
For each event e with non-empty parent_event_ids:
  For each parent p in e.parent_event_ids:
    If p does not exist                      → emit anomaly("dangling_parent")
    If p.received_at > e.received_at + Δ     → emit anomaly("backward_causality")
    If p.lamport_ts >= e.lamport_ts          → emit anomaly("clock_inversion")
```

Where `Δ` is a small server-side processing tolerance (default 1
second; permits parent and child to land in same write batch).

Anomalies write to `system_anomalies(job_id, kind, details, detected_at)`.
Two consumers:
- **Bug detection**: our own scheduler / daemon / aggregation code
  may be assigning parent IDs wrong; anomalies surface internal
  bugs before they corrupt downstream state.
- **Malicious-provider detection**: a peer fabricating parent IDs
  (e.g. claiming a chain hop it didn't actually receive) trips
  the detector. Tripped providers lose reputation per IPIP-0007
  CPR.

### 5. Future work — distributed mutual exclusion

When peers compete for shared resources without going through the
control plane (a future capability — e.g. two providers both
trying to claim the right to cache a hot prompt range, two
trainers competing to be the parameter server for a swarm),
Lamport's distributed mutex algorithm (*Time, Clocks…*, p. 561)
is the textbook tool:

> 1. To request the resource, Pi sends `Tm:Pi requests resource`
>    to every other process and puts on its own queue.
> 2. On receiving, Pj puts on its queue and sends ACK.
> 3. To release, Pi sends `Pi releases resource` and removes from
>    its own queue.
> 4. On receiving release, Pj removes from queue.
> 5. Pi gets the resource iff its request is the queue head per
>    `⇒` AND it has received a message from every other process
>    timestamped later than `Tm`.

Required assumptions: FIFO message channels between any two
processes, eventual delivery, send-to-all. Our libp2p transport
gives FIFO per-connection; eventual delivery requires the outbox
pattern (IPIP-0014 §9) at minimum.

This is OUT OF SCOPE for v1 — we have a centralized scheduler
that handles claims. Reserved here so future P2P-claim work
references the algorithm rather than reinventing it.

### 6. Future work — physical clock synchronization

Lamport's strong clock condition addresses the external-observer
anomaly with synchronized physical clocks. The bound he proves:

> ε ≈ d(2κτ + ξ) for t ≥ t0 + τd

where `d` = graph diameter, `κ` = max clock drift rate, `τ` =
resync interval, `ξ` = unpredictable message delay.

Practical implication for Infernet: sub-second cross-provider
total ordering would require NTP stratum-1 clocks (κ ≈ 10⁻⁶) with
small graph diameter (d ≤ 10) and ξ < 100ms. We don't promise
this today and shouldn't. The bound is documented here so future
work can reason about feasibility.

For our actual use cases (per-job partial ordering, payment
auditing, anomaly detection), causal dependency stamps + external
tokens are sufficient.

## Rationale

**Why sparse vector clocks instead of full vector clocks?** Full
vector clocks cost O(N) per event where N = peer count; at 10k
providers that's 10k counters in every event envelope. Sparse
clocks cost O(parent_count) — typically 1, occasionally a handful
for merge events. Same expressive power for the questions we ask:
"is a happens-before b?" reduces to a graph reachability query in
either representation.

**Why explicit parent IDs and not implicit topological order?**
Implicit ordering (e.g. "events arriving from the same provider
on the same job are sequentially ordered") works for the simple
case but breaks for merges. Making parents explicit unifies the
two cases, catches bugs (impossible parent IDs), and lets the
anomaly detector run.

**Why a separate IPIP from 0014?** IPIP-0014 is the floor every
endpoint stands on — five primitives that apply to every state-
mutating call. This IPIP applies only to events that span multiple
providers, which is a smaller surface (Class B.5, Class C, batch
chunk retries). Splitting keeps each spec digestible and lets
0014 ship first while 0015 follows when Class B.5/C land for real.

**Why is the anomaly detector janitor periodic, not real-time?**
Real-time anomaly detection on every event write would block hot
paths. Periodic (5-minute) is good enough — anomalies surface
within minutes, not seconds. Operators investigating in real time
have the dashboard's `event_relation()` to debug interactively.

## Backwards compatibility

Additive. `parent_event_ids` defaults to empty array; existing
events lack parents (they predate this IPIP). The anomaly detector
ignores events without parents (no relations to check). Class A
single-provider chat is unaffected — single-provider streams use
implicit sequential parents, no envelope changes needed unless
the operator wants the dashboard's `concurrent` indicator.

## Test plan

- **Unit:** `eventRelation` correctness on synthetic graphs
  (chain, fork, merge, diamond). Anomaly detector flags each of
  the three impossible patterns (dangling, backward, inversion).
- **Integration:** simulated Class B.5 chain with parent stamps;
  verify replay reconstruction matches sender-ordered ground
  truth even under reordered arrival.
- **Negative:** malicious provider fabricating a parent ID for an
  event it never received → anomaly detector flags within one
  janitor cycle.
- **Performance:** event emission overhead (parent ID lookup +
  envelope inclusion) ≤ 1 ms p99 on commodity hardware; anomaly
  janitor scan completes within its cadence on 100k recent
  events.

## Security considerations

- **Parent-ID forgery.** A provider could claim parents that
  weren't truly causal. Mitigated by §4's anomaly detector: if
  the claimed parent doesn't exist, lamport_ts inversions, or
  arrival timing contradicts, the anomaly trips reputation.
- **Information leakage via parents.** A provider observing
  parent IDs can infer the topology of cross-provider chains.
  Mitigated by signing parents in the envelope and exposing them
  only to participants in the same job.
- **Causal-token capture.** External tokens (§3) authenticate
  via the existing API auth (Bearer / Nostr signature). A leaked
  token allows attaching false events to a real workflow;
  mitigation is the same as any leaked auth credential — rotate
  + audit.

## Privacy considerations

`parent_event_ids` exposes finer-grained causal topology than
plain Lamport timestamps. A consumer with read access to many
jobs' events can reconstruct provider mesh structure (who relays
to whom). Acceptable in v1 because the mesh is already public
information (heartbeats advertise provider addresses); revisit if
we add an "anonymous" relay mode.

## Open questions

- **Parent-set size cap.** Pathological merge events could carry
  thousands of parents. Cap at e.g. 64 per event; emit a "merge
  summary" parent instead for larger reductions?
- **Garbage collection.** Old events with broken parent chains
  (because their parents were GC'd) will trip "dangling_parent"
  anomalies. Need a janitor-aware GC that walks both directions.
- **Cross-job parents.** Can event in job X depend on event in
  job Y? Permit, but flag in dashboard? Forbid?
