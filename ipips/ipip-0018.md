## Abstract

Specifies a causal-broadcast delivery protocol for cross-provider
events in Class B.5 (federated inference) and Class C (distributed
training) workloads. Events are accepted into the receiver's local
view only when all of their causal predecessors have been observed
— guaranteeing that *no consumer ever sees an effect before its
cause*. Built on the per-event sparse causal stamps from IPIP-0015
§1; the wire format already carries everything the protocol needs.
Pairs with IPIP-0017 CRDT primitives to give us strong-eventual-
consistency-with-causal-delivery (the standard CBCAST + CRDT
combination).

## Motivation

IPIP-0014 §5 + IPIP-0015 §1 give us logical clocks and causal
dependency stamps at the *event level*. They tell us, given two
events, which one happened first. They don't say what to do when
events arrive at a consumer in the wrong order.

Concrete scenario: in a Petals chain (IPIP-0012), peer A finishes
layer 12 → sends hidden state to peer B → peer B starts layer 13.
The control plane subscribes to events from both. If B's
"started layer 13" event reaches the control plane before A's
"finished layer 12" event (because B's network path was faster),
the dashboard's job timeline shows B starting before A finished —
visually impossible, indistinguishable from a malicious peer
forging a fake parent ID.

Three ways to handle out-of-order arrival:

1. **Show events in arrival order** — what we do today. Wrong but
   simple.
2. **Sort by Lamport timestamp on display** — fixes display but
   correctness-critical code (anomaly detector, replay,
   reputation attribution) still sees the wrong order if it
   reads in arrival order.
3. **Causal delivery (CBCAST)**: the receiver buffers an event
   until all its causal predecessors have been delivered. Then
   ordered processing is automatic for any consumer.

Option 3 is the textbook fix — Birman-Schiper-Stephenson '91 for
the protocol with vector clocks; the same pattern works with our
sparse parent-stamps. This IPIP specifies the protocol.

## Specification

### 1. Causal-delivery contract

A receiver implementing causal broadcast accepts events as input
and produces a stream of "delivered" events. The contract:

> An event `e` is delivered only after every `p ∈ e.parent_event_ids`
> has been delivered.

Equivalently: the delivered stream respects the happens-before
partial order induced by parent stamps.

### 2. Receiver state

Each causal-broadcast receiver maintains:

```ts
interface CausalReceiver {
    delivered:  Set<EventID>;          // events already passed downstream
    holdback:   Map<EventID, Event>;   // received but not yet deliverable
}
```

### 3. Receive algorithm

```
on receive(e):
    if e.id ∈ delivered:                    // duplicate
        return

    holdback.set(e.id, e)
    drainHoldback()

drainHoldback():
    repeat:
        any_delivered = false
        for e in holdback.values():
            if all(p ∈ delivered for p in e.parent_event_ids):
                holdback.delete(e.id)
                delivered.add(e.id)
                emit_downstream(e)
                any_delivered = true
        if not any_delivered: break
```

After every receive, drain the holdback set as far as it can go.
Events whose parents are still missing stay buffered until those
parents arrive.

### 4. Holdback bounds

The holdback set is bounded by the number of in-flight events
that reference parents the receiver hasn't seen yet. Pathological
sequences (events with parents that will *never* arrive — e.g.
fabricated parent IDs from a malicious sender) bloat the set
unboundedly.

Mitigation:

- **Holdback timeout** (default 60 seconds): if an event has been
  in holdback for > timeout without its parents arriving, the
  receiver MAY:
  1. Forcibly deliver it as a "broken-causality" event marked
     `causal_status = "broken"`, AND
  2. Emit a `system_anomaly` per IPIP-0015 §4 against the
     publisher.
- **Holdback size cap** (default 10 000 events per job): events
  beyond the cap evict oldest-first (LRU).

### 5. Initial-state question (where do parents come from on join?)

When a receiver first joins (e.g. a fresh dashboard subscriber on
an in-progress job), it has empty `delivered` set, so events with
ANY parent ID reference would block forever waiting for parents
the receiver never saw.

Mitigation: receivers MAY bootstrap from a snapshot. The control
plane exposes `GET /api/v1/jobs/{id}/causal-snapshot` which
returns:

```json
{
  "delivered_at": "<RFC3339>",
  "delivered_event_ids": ["<id1>", "<id2>", ...]
}
```

The receiver pre-populates `delivered` from this list, then
subscribes for new events. Future events whose parents are all
in the snapshot deliver immediately; events that reference
even-earlier ancestors (if any escape into the live stream) hit
the holdback timeout path.

### 6. Interaction with anomaly detection (IPIP-0015 §4)

The IPIP-0015 anomaly detector runs on the *delivered* stream,
not the raw arrival stream. Its three patterns become:

| Pattern | Meaning under causal delivery |
|---|---|
| `dangling_parent` | A delivered event references a parent ID that's not in `delivered`. With CBCAST this should be rare — only happens if forced delivery via holdback timeout. |
| `backward_causality` | Parent event's `received_at` is later than child's. CBCAST guarantees parent is *delivered* before child but doesn't constrain server-side write timing. Still meaningful as a clock-skew signal. |
| `clock_inversion` | Parent's lamport_ts ≥ child's. A real bug somewhere — IR2 (IPIP-0014 §5) violation. |

Forced-delivery events from the holdback timeout (§4) are still
recorded but flagged `causal_status = "broken"` so consumers
know not to trust their position in the order.

### 7. Performance characteristics

The receive algorithm is O(holdback × max_parents) per drain
attempt. For typical workloads (single-parent linear chains in
Class B.5; small fan-out / fan-in trees in Class C), this is
effectively O(holdback). With the 10k-event cap (§4), each drain
is bounded.

A naive implementation re-checks every holdback entry on every
incoming event. Smarter implementations (out of scope for this
IPIP — leave to the implementation file): index the holdback by
"parent ID I'm waiting on" so an arrival of `e` directly wakes
only the events that reference it.

### 8. Pairing with CRDTs (IPIP-0017)

CRDTs guarantee strong eventual consistency under any delivery
order. They don't require causal delivery to converge; they're
strictly more lenient.

But many *applications* of CRDTs prefer causal delivery for
human-meaningful behavior. Example: a chat dialog rendered from
a CRDT log looks weird if a "Bob: who?" message renders before
the "Alice: did you see that?" it's responding to, even though
the text-CRDT state is correctly converged. CBCAST + CRDT is the
standard combination — strong eventual consistency *and*
causally-ordered observation.

For Infernet, the cases that benefit:
- Class C training event timeline (rendering deltas in causal
  order is more comprehensible than arrival order)
- Operator chat history if/when we ship it
- Job-event audit trail in the dashboard

For pure background convergence (the DHT registry merging during
partition heal), CBCAST is unnecessary overhead — go directly
through CRDT merge per IPIP-0017.

## Rationale

**Why CBCAST and not total order?** Total order broadcast (TOB)
guarantees every replica delivers events in the *same* order; we
don't need that. We need every replica to deliver an event after
its parents — partial-order, not total-order. CBCAST is cheaper
(no consensus required) and matches our semantics.

**Why holdback at the receiver, not gap-detection at the sender?**
Gap-detection requires the sender to know which receivers have
seen which events — adds state and round-trips. Receiver
holdback is local and stateless from the sender's perspective.

**Why a 60-second holdback timeout?** Above the typical p99
network delay across the public internet (~500 ms intra-region,
~3 s cross-region), well below human-visible "this dashboard is
broken" timescale. 60 s lets transient routing problems heal
without forcing broken-causality deliveries.

**Why bootstrap-snapshot for initial state?** Without a snapshot,
a fresh subscriber on a long-running job can't deliver anything
because every event's parents are pre-subscription history. The
snapshot is small (a list of UUIDs), cheap to serve, and makes
the join experience instant.

## Backwards compatibility

Additive. Receivers that don't implement CBCAST (existing
dashboard timeline component as of today) continue to render in
arrival order; nothing breaks. The control plane's anomaly
detector becomes more accurate as more consumers adopt CBCAST.

## Test plan

- **Unit:** receive sequences with permuted arrival orders;
  assert delivered stream always respects parent → child.
- **Holdback timeout:** event with non-existent parent → forced
  delivery after timeout, anomaly emitted, `causal_status =
  "broken"`.
- **Holdback cap:** flood with 20k events referencing missing
  parents; assert eviction caps memory at 10k.
- **Bootstrap:** late-joining subscriber catches up via snapshot
  + live stream, never hangs.
- **Integration:** Class B.5 chain simulation with reordered
  arrivals; CBCAST output matches sender-ordered ground truth.

## Security considerations

- **Holdback exhaustion attack.** Malicious sender publishes
  events with fabricated parent IDs to balloon receiver memory.
  Mitigated by §4's 10k cap + LRU eviction + per-publisher rate
  limit.
- **Forced delivery hides real causality bugs.** If broken-
  causality events are silently merged into the timeline,
  operators may not notice. Mitigation: surface anomalies
  prominently (system_anomaly UI panel) and tie them to
  reputation per IPIP-0007 CPR.

## Privacy considerations

Same as IPIP-0015 §Privacy: parent_event_ids reveal mesh
topology. CBCAST doesn't add new privacy exposure beyond what
the parent stamps already have.

## Open questions

- **Per-job vs global receivers.** Each subscriber currently has
  one CausalReceiver per job. Aggregating across jobs would let
  one receiver shared across all dashboards reduce memory, but
  hurt isolation. Stick with per-job for v1.
- **Snapshot freshness.** §5's snapshot is point-in-time at
  request; an event delivered between snapshot generation and
  subscriber consumption is "newer than the snapshot but older
  than live". Standard fix: include the live-stream sequence
  number in the snapshot response and have the subscriber
  resume from there.
