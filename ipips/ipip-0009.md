## Abstract

Defines the pluggable inference engine interface used by Infernet provider
nodes. A node MAY run inference via Ollama, vLLM, a Mojo+MAX binary, or a
canned-stub backend. All four speak the same v1 NDJSON protocol so the
daemon, scheduler, and chat surfaces are backend-agnostic. This IPIP is
the contract.

## Motivation

Operators run wildly different hardware: Apple Silicon laptops, single-GPU
desktops, multi-GPU servers, hyperscale clusters. No single inference
runtime fits everyone. Ollama is the easy default (any GPU + CPU + Apple
Silicon, no driver dance), vLLM is the throughput choice for serious
NVIDIA hardware (PagedAttention, in-cluster TP+PP), Mojo+MAX is the
experimental path for the eventual Modular runtime, and the stub keeps
the network functional on hardware that has none of the above.

Without a common adapter contract, every consumer of inference (the chat
playground, the dashboard's "push model", the daemon's job loop) has to
special-case each engine. The contract collapses that into one async
iterator of typed events.

## Specification

### Engine factory

Implementations MUST expose `createEngine(opts)` returning an engine
object with this shape:

```ts
interface Engine {
  kind: "ollama" | "vllm" | "llamacpp" | "mojo" | "stub";
  generate(req: GenerateRequest): { id: string; stream: AsyncIterable<Event>; cancel(): void };
  shutdown(): Promise<void>;
}
```

`opts.backend` MAY be passed explicitly. When absent, auto-selection
precedence is:

1. `process.env.INFERNET_ENGINE_BACKEND` — explicit override
2. `process.env.INFERNET_ENGINE_BIN` set → `"mojo"`
3. vLLM reachable on `VLLM_HOST` → `"vllm"`
4. llama.cpp reachable on `LLAMACPP_HOST` → `"llamacpp"`
5. Ollama reachable on `OLLAMA_HOST` → `"ollama"`
6. `"stub"`

vLLM ranks ahead of Ollama because installing vLLM is a deliberate
operator choice (heavy CUDA + Ray install, NVIDIA-only) — a running
vLLM server is a strong signal of intent.

### Generate request

```ts
interface GenerateRequest {
  messages: Array<{ role: "system" | "user" | "assistant"; content: string }>;
  id?: string;          // generated if absent
  model?: string;       // overrides defaultModel
  max_tokens?: number;
  temperature?: number;
}
```

### Event protocol (v1)

NDJSON, one JSON object per line, each terminated by `\n`. Every message
carries `v: 1` and `type`. Generation-scoped messages also carry `id`.

| Type | Fields | When |
|---|---|---|
| `meta` | `id`, `model?`, `started_at` | First event of a generation |
| `token` | `id`, `text` | Each streamed token chunk |
| `done` | `id`, `reason: "stop"\|"length"\|"cancel"`, `text`, `finished_at` | Terminal success |
| `error` | `id?`, `message` | Terminal failure (no `id` → engine-level) |

`cancel()` MUST cause the in-flight generation to terminate with a `done`
event of `reason: "cancel"`, NOT an `error`.

### Backend reachability probes

Each backend that does network probing MUST export an `is{Backend}Reachable(host?, timeoutMs?)`
function that returns `Promise<boolean>` and never throws. Defaults:
`PROBE_TIMEOUT_MS = 500`.

## Rationale

**Why NDJSON over SSE between adapters and consumers?** SSE is chosen at
the HTTP boundary (the `/api/chat/stream/[jobId]` route) but adapters speak
to the daemon process directly — NDJSON is simpler and matches the
existing engine-process IPC.

**Why per-backend env vars** (`OLLAMA_HOST`, `VLLM_HOST`, `INFERNET_ENGINE_BIN`)
**instead of one unified `ENGINE_HOST`?** Operators routinely run more than
one engine on the same box — Ollama for chat, vLLM for batch — and the
daemon picks per-job. A single env var would force them to switch state
between jobs.

**Why does vLLM rank ahead of Ollama in auto-select?** Installing vLLM is
an explicit, heavy choice. If both engines are running, the operator's
last action was almost certainly to start vLLM.

## Backwards compatibility

None. The current `createEngine()` factory already speaks this contract;
this IPIP retroactively documents it as the standard, and fills in the
vLLM backend as the third compliant implementation.

## Security considerations

Engines run as subprocesses or external HTTP servers under the operator's
control. No new attack surface introduced by this IPIP itself; per-backend
auth (e.g., vLLM's `--api-key`) is handled by each backend's adapter via
its own env (`VLLM_API_KEY`).

## Privacy considerations

Engine choice is observable to anyone who can read the provider's
heartbeat (`specs.engine.kind`). No prompt content or generated tokens
leave the operator's machine through this protocol — those flow over
the existing signed `/api/v1/node/jobs/*` endpoints (IPIP-0003).

## Open questions

- Should backends advertise capabilities (max context, supported chat-template format) in `meta`? Probably yes once batching/embeddings land.
- LLM tool-calling format: pass through engine-native, or normalize at the adapter layer?
