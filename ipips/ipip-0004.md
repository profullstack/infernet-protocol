## Abstract

Surface the multi-currency payment rails that already live in the
control plane (`packages/payments`, `provider_payouts`,
`payment_transactions`, `platform_wallets`) through the CLI so
operators can configure payouts during `infernet setup` / `init`,
and clients can pay for jobs in any cryptocurrency CoinPayPortal
supports. Includes optional **Lightning wallet provisioning** via
CoinPayPortal's hosted custodial wallets for operators who don't
already run their own LN node.

The launch position is: **every CoinPayPortal-supported coin works
on day one. Bitcoin/LN is one of many, not the default.**

## Motivation

The chain-agnostic story is the differentiator. io.net pays in $IO,
Akash in $AKT, Render in $RNDR, Bittensor in $TAO â€” every
competitor forces operators to hold a volatile, project-specific
token. Infernet's existing CoinPayPortal integration already supports
BTC, BCH, ETH, SOL, POL, BNB, XRP, ADA, DOGE, USDT (multi-chain),
USDC (multi-chain), and Lightning. Operators can take payment in
whichever asset they actually want to hold, and clients can pay in
whichever asset they actually have. **No native token. No volatility
exposure forced on either side.**

That capability ships in the schema today. Two gaps:

1. **Operators have no CLI affordance to configure their payout
   addresses.** They have to write to the database directly.
2. **Operators without a Lightning node** can't accept LN payments
   even though the schema supports it. CoinPayPortal offers hosted
   custodial LN wallets via API; we haven't wired it.

Closing both gaps puts a multi-chain payment story on launch day
that none of the named competitors can match.

### Storage schema (already shipped)

`provider_payouts` keys an operator's payout address per
`(provider_id, coin, network)`:

```sql
-- existing in supabase/migrations/20260419000000_cli_support_and_payments.sql
create table public.provider_payouts (
  id          uuid primary key,
  provider_id uuid references providers(id),
  coin        text not null,
  network     text,
  address     text not null,
  -- ...
  unique (provider_id, coin, network)
);
```

`payment_transactions` is the ledger; `platform_wallets` are the
control plane's own deposit addresses; `jobs.payment_coin` is which
coin a given job is paid in. **None of this needs new schema for
IPIP-0004.** What's new is CLI exposure + LN wallet provisioning.

### CLI surface

#### Payouts: `infernet payout` (existing â€” extended)

```bash
infernet payout list                      # show all configured payouts
infernet payout set BTC mainnet <addr>    # set / update an address
infernet payout set BTC ln <bolt12>       # set a Lightning address
infernet payout remove BTC mainnet        # delete a row
```

The existing `infernet payout` already speaks the signed-request
protocol (`/api/v1/node/payouts/{list,set}`); IPIP-0004 just adds
the LN sub-flow and ensures the coin/network allow-list mirrors
the supported matrix above.

#### Setup-time onboarding

`infernet setup` and `infernet init` get an opt-in step that walks
the operator through configuring at least one payout address:

```
[N/M] Payouts
  Configure how you want to be paid for served jobs?
  (You can skip and run `infernet payout set` later.)  [Y/n]: y

  Which coins do you want to accept? (space-separated, or "all")
  > BTC SOL USDC

  BTC payout address: <paste>
  BTC network [mainnet/ln]: mainnet
  Want a Lightning channel auto-provisioned via CoinPayPortal? [y/N]: n

  SOL payout address: <paste>
  USDC network [ethereum/polygon/solana/base]: solana
  USDC payout address: <paste>

  âś“ 3 payout addresses configured
```

Skip with `--skip-payouts` (similar to existing `--skip-firewall`).

#### LN wallet provisioning via CoinPayPortal

For operators who select Lightning but don't have an existing LN
wallet, the CLI offers to provision one:

```bash
infernet payout set BTC ln --provision
# â†’  CoinPayPortal hosted custodial LN wallet creation
# â†’  CLI receives bolt12 / lnurl-pay address back, saves to provider_payouts
```

Server-side: a new route `POST /api/v1/node/payouts/provision-ln`
(signed) that:

1. Verifies the caller (Schnorr sig per IPIP-0001).
2. Calls CoinPayPortal API to create a custodial LN wallet bound
   to the operator's `provider_id`.
3. Saves the returned address into `provider_payouts` as
   `(coin=BTC, network=ln)`.
4. Returns the address to the CLI.

This path is **opt-in.** Custodial means CoinPayPortal holds funds
until the operator withdraws. Operators with their own LN nodes
should NOT use it; pass their own bolt12 instead.

### Client-side (job submission)

Payments attach to **paid jobs**, not to any specific endpoint. A job
is a unit of compute the client requests from the network and pays
for. Job types covered by this IPIP:

| CLI verb            | Job type    | Status |
| ------------------- | ----------- | ------ |
| `infernet chat`     | `inference` | Shipped (today via `/api/chat`, see endpoint note below) |
| `infernet train`    | `training`  | Planned |
| `infernet embed`    | `embedding` | Planned |
| `infernet image`    | `image-gen` | Planned |
| `infernet *` (future) | TBD       | Same payment plumbing |

Every paid job carries an optional `payment` block on submission:

```jsonc
// generic shape â€” same regardless of job type
POST <control-plane>/api/jobs/submit
{
  "type": "inference",
  "input": { /* type-specific payload, e.g. messages: [...] for inference */ },
  "payment": {
    "coin":    "USDC",
    "network": "solana",
    "max":     0.05      // upper bound the client agrees to spend
  }
}
```

The control plane's job router prefers providers who have a matching
`(coin, network)` payout configured. Missing `payment` block defaults
to "any" â€” the router picks the cheapest matching provider.

**Endpoint note.** The current `/api/chat` route is the inference-only
ancestor of this generic submit endpoint. v1 may keep it as a backward-
compat alias that internally constructs `{ type: "inference", input:
{ messages: payload.messages } }`. The canonical surface for any new
job type is `/api/jobs/submit`. Cleaning up the legacy alias is a
follow-up PR; not a blocker.

### Boundary with IPIP-0002 (P2P chat)

**Person-to-person chat (IPIP-0002 â€” `infernet talk`) is a
communication layer, not a payment-bearing surface.** Direct messages
and room messages between operators MUST NOT carry payment data,
trigger payouts, or be accounted in `payment_transactions`. They're
plain Nostr DMs / kind-1 events on relays, exactly as IPIP-0002 specs.

Naming reality: the CLI verb `infernet chat` (inference job
submission, this IPIP) and the protocol surface "P2P chat"
(IPIP-0002) share the word "chat" only because the public language
hasn't separated them yet. They're orthogonal:

| | `infernet chat "what is 2+2"` | `infernet talk @abc123` |
| --- | --- | --- |
| Surface | Paid job submission | Person-to-person message |
| IPIP | IPIP-0004 | IPIP-0002 |
| Pays? | Yes | No |
| Transport | Signed HTTP â†’ control plane (or P2P transport when shipped) | Nostr events on relays |
| Recipient | An anonymous provider node serving the right model | A specific human pubkey or topic-tagged room |

Implementations MUST NOT route paid-job tokens through IPIP-0002
relays, and MUST NOT charge for IPIP-0002 messages. Two distinct
features in two distinct codepaths.

## Rationale

**Why CoinPayPortal and not Stripe / a custom adapter?** Already in
the stack and working today; non-custodial / multi-chain by design;
the founder has control over the CoinPayPortal product itself.

**Why surface in CLI rather than just web?** Operators run on
servers. The friction of "ssh in to register, then go to a browser
to set payouts" is the kind of context switch that loses a sign-up.
Same reason the npm publish flow lives in `npm` not in the website.

**Why opt-in LN provisioning?** LN is great UX (instant,
near-zero-fee) but custodial LN means the network is holding funds
for the operator. Some operators want non-custodial; force-default-
custodial would be hostile. Asking is the right move.

**Why cover ALL coins on launch?** Branding. "Every coin
CoinPayPortal supports" is a stronger pitch than "BTC/ETH/USDC and
five more later." The work is one allow-list; the marketing payoff
is real.

**Why not a native Infernet token?** That's the IPIP-0001 thesis
re-stated â€” no rent extraction from operators or clients, no
volatile exposure forced on participants.

## Backwards compatibility

- Existing `provider_payouts` / `payment_transactions` /
  `platform_wallets` schema unchanged. No migrations needed for v1.
- Existing `infernet payout` command extended (additive); existing
  flags continue to work.
- Adding the payouts step to `infernet setup` is gated by a
  `--skip-payouts` flag; existing scripts that pipe through setup
  can opt out.
- Clients that don't specify `payment` on `/api/chat` continue to
  hit the existing default routing â€” no breaking change.

## Reference implementation

Phased; each phase is its own PR referencing IPIP-0004.

**Phase 1 â€” supported-coin allow-list.**
- `packages/payments/src/supported-coins.js` exports the canonical
  list of `(coin, networks[])` tuples.
- Both server and CLI import from this list (no string drift).

**Phase 2 â€” CLI payout extension.**
- `infernet payout list/set/remove` plumbed through the existing
  signed `/api/v1/node/payouts/{list,set}` routes, with a
  `--coin/--network` validator against the allow-list.

**Phase 3 â€” Setup integration.**
- Add Payouts step to `infernet setup` and `infernet init`.
- Walks operator through coin selection + address paste.
- Skip with `--skip-payouts`.

**Phase 4 â€” LN wallet provisioning.**
- `POST /api/v1/node/payouts/provision-ln` route.
- `infernet payout set BTC ln --provision` CLI flag.
- CoinPayPortal API client wired in; bolt12/lnurl-pay returned.

**Phase 5 â€” Client-side payment routing.**
- New canonical endpoint `/api/jobs/submit` accepts the generic
  `{ type, input, payment }` shape. `/api/chat` becomes a thin
  backward-compat alias that constructs `{ type: "inference",
  input: { messages } }` from the legacy body.
- Job router scores providers by `(coin, network)` match plus
  existing factors (load, latency, model).
- CLI: `infernet chat`, `infernet train`, `infernet embed`,
  `infernet image` all submit through the same shape, differing
  only in the `type` field.

## Security considerations

- **Address-confusion attack**: pasted addresses MUST validate per
  coin (BTC base58/bech32, ETH 0x-checksummed, SOL base58, etc.).
  CLI rejects malformed addresses before signing the
  `/api/v1/node/payouts/set` request. Server validates again as
  defense-in-depth.
- **Provider-row poisoning**: `provider_payouts` rows are
  pubkey-scoped (signed-request envelope ensures the calling pubkey
  matches `provider_id.public_key`). A third party can't set
  someone else's payout address.
- **CoinPayPortal compromise**: payment-transactions are written
  by webhooks signed with `COINPAYPORTAL_WEBHOOK_SECRET`. Compromise
  of that secret = ability to forge transaction confirmations.
  Mitigation: rotate the secret quarterly, store in a secret
  manager, never log webhook bodies.
- **Custodial LN**: if `--provision` is used, operator funds are
  held by CoinPayPortal until withdrawn. CLI MUST display this
  warning prominently before provisioning.

## Privacy considerations

- **Linkability**: payout addresses are linked to the operator's
  Nostr pubkey. Operators wanting payout-vs-compute unlinkability
  should use the "separate payout identity" deferred item from
  IPIP-0001, when it ships.
- **Address re-use**: operators can configure a fresh address per
  coin or reuse the same one across providers; we don't enforce.
  The dashboard MAY surface "this address has received N payments
  across M providers" as a privacy nudge.

## Open questions

- **Default coin if operator doesn't pick one in setup?** Probably
  USDC on Solana (low fees, stable, fast). Need to confirm
  CoinPayPortal supports per-tx fee defaults that work for
  micro-payments.
- **Per-job pricing in arbitrary coins**: should the price field on
  jobs be denominated in USD (with conversion at settlement) or in
  the chosen coin's native unit? Leaning USD-denominated, settled
  in coin â€” but this is its own design decision.
- **Withdrawals from custodial LN**: how often do we let operators
  pull funds out of CoinPayPortal LN wallets? CoinPayPortal API
  rate limits?
- **Lightning Address (lud16) vs bolt12 vs lnurl-pay**: pick one
  canonical format for the `network=ln` row, or accept any of the
  three? Probably accept all three, normalize on read.
