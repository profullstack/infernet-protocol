## Abstract

Adds two capabilities to the Infernet engine layer in response to
Canonical's 2025-2026 push to package GPU runtimes natively in Ubuntu:

1. **Host AI runtime detection** during install — when the operator's
   OS already provides CUDA / ROCm / OpenVINO (via apt, dnf, etc.),
   `install.sh` skips the redundant bundled install paths (Ollama's
   ~3 GB mlx_cuda_v13 libs, the apt CUDA toolchain dance, etc.) and
   wires the engine to the host's existing runtime instead.
2. **OpenVINO engine adapter** — a fifth backend alongside Ollama,
   vLLM, llama.cpp, Mojo, and stub (per IPIP-0009). Targets Intel
   iGPUs (Arc, Iris Xe), Intel discrete GPUs (Battlemage), and the
   Intel NPU on Meteor Lake / Lunar Lake / Panther Lake laptops —
   hardware our current adapter set ignores entirely.

## Motivation

Ubuntu 26.04 LTS (per Canonical's announced plans) ships the full
NVIDIA CUDA toolkit, AMD ROCm, and Intel OpenVINO via standard
`apt`. Older releases required multi-step manual setup that broke
driver configuration in subtle ways. Three concrete consequences
for Infernet:

1. **Ollama's bundled CUDA libs are increasingly redundant.** We
   already saw this hurt: Ollama's installer extracts ~3 GB of
   `mlx_cuda_v13` even when the host has CUDA installed via apt.
   On RunPod's 5 GB overlay (per the install.sh saga earlier) this
   blew up the install. If we detect host CUDA, we can tell Ollama
   to skip its bundle (or use llama.cpp / vLLM directly against
   the host's CUDA).

2. **Intel hardware is a real audience we don't cover.** Operators
   running Lunar Lake laptops have a 48-TOPS NPU sitting idle
   because our adapters only know about NVIDIA / AMD / Apple
   Silicon / CPU. OpenVINO is the only mainstream way to use that
   hardware; ignoring it leaves an entire vendor's footprint
   unreachable.

3. **ARM64 + RISC-V** are coming up in Canonical's CPU-variant
   build pipeline. Our daemon already runs on ARM64 (Apple Silicon
   confirms this), but we don't formally test or document it for
   non-Apple ARM (Snapdragon X / Ampere / etc.). Making ARM64 a
   first-class target follows naturally.

Reference: [The Pragmatic Engineer — "Ubuntu and AI"](https://newsletter.pragmaticengineer.com/p/ubuntu-and-ai).

## Specification

### 1. Host runtime detection

`install.sh` adds a `detect_host_runtime()` step before
`try_install_node_unattended` that probes for:

| Runtime | Detection | If present |
|---|---|---|
| **CUDA** (apt-installed) | `dpkg -s nvidia-cuda-toolkit` succeeds, OR `command -v nvcc && /usr/local/cuda/bin/nvcc --version` works | Set `INFERNET_CUDA_HOST=apt`. Pass through to engine adapters; Ollama install path skips its bundled CUDA via `OLLAMA_NO_CUDA_BUNDLE=1`. |
| **ROCm** | `dpkg -s rocm-libs` OR `command -v rocminfo` | Set `INFERNET_ROCM_HOST=apt`. Engine adapters use host ROCm. |
| **OpenVINO** | `dpkg -s openvino` OR `python3 -c "import openvino"` succeeds | Mark host as OpenVINO-capable. The OpenVINO engine adapter (§2) becomes a candidate during auto-select. |

Detection is non-fatal: if a runtime is missing, install.sh proceeds
with its current bundled-install behavior. Detection only *avoids
duplication*, never *requires* a host runtime.

### 2. OpenVINO engine adapter

New file: `packages/engine/src/backends/openvino.js`. Same shape as
the vLLM adapter (IPIP-0009) — exposes:

- `OPENVINO_DEFAULT_HOST` (default: `http://localhost:8090`)
- `isOpenvinoReachable(host, timeoutMs)` — probes
  `<host>/health` (OpenVINO Model Server's health endpoint)
- `createOpenvinoBackend({ host, defaultModel, ... })` — speaks
  OpenVINO Model Server's OpenAI-compatible chat completions
  endpoint at `/v3/chat/completions` (note: v3, not v1 — OVMS has
  its own version path)

Engine `kind` is `"openvino"`. Auto-select precedence (extending
IPIP-0009):

1. `process.env.INFERNET_ENGINE_BACKEND` — explicit override
2. `process.env.INFERNET_ENGINE_BIN` set → `"mojo"`
3. vLLM reachable → `"vllm"`
4. llama.cpp reachable → `"llamacpp"`
5. **OpenVINO reachable → `"openvino"`** ← new
6. Ollama reachable → `"ollama"`
7. `"stub"`

OpenVINO ranks after vLLM/llama.cpp/Ollama because (a) it's the
narrowest hardware fit, (b) operators running OVMS have made an
explicit setup choice; the auto-detect just confirms what they
intended.

Env vars:
- `OPENVINO_HOST` — server URL (default `http://localhost:8090`)
- `OPENVINO_API_KEY` — optional Bearer token for OVMS's auth mode
- `OPENVINO_MODEL` — default model name
- `OPENVINO_DEVICE` — `CPU`, `GPU`, `NPU`, `AUTO` (default `AUTO`
  → OVMS picks the best available)

### 3. install.sh OpenVINO branch

When `nvidia-smi` is absent AND `rocm-smi` is absent AND a CPU has
known Intel NPU support (Meteor Lake / Lunar Lake / Panther Lake —
detected via `/proc/cpuinfo` model name match) OR `dpkg -s openvino`
succeeds, install.sh prints:

```
==> Intel NPU / GPU detected — OpenVINO is available for inference.
    Run with INFERNET_ENGINE_BACKEND=openvino once OVMS is up:
        sudo apt install openvino-model-server
        ovms --rest_port 8090 --model_path /path/to/openvino-ir/model
    See docs/engine-openvino.md.
```

We do NOT auto-install OpenVINO the way we auto-install Ollama —
OpenVINO model conversion (PyTorch → IR) is operator-driven and
not a one-line dance. The detection prints a pointer; the operator
opts in.

### 4. ARM64 + RISC-V notes

- ARM64 already works (Apple Silicon proves it). Add explicit CI
  for Linux ARM64 (Ampere Altra runners on GitHub Actions).
- RISC-V is forward-looking. No code changes; track via this IPIP's
  open questions.

### 5. Forward-looking: snap-managed engine install

When Canonical's "inference snaps" land (per the Pragmatic Engineer
article — early-stage as of 2026), `install.sh` could detect a
matching snap and skip our bundled install:

```
if snap list | grep -q '^ollama '; then
    info "host has ollama via snap — using it"
    OLLAMA_HOST="${OLLAMA_HOST:-http://localhost:11434}"
    return 0
fi
```

NOT specified here — too speculative until the snaps exist.
Reserved as a future amendment.

## Rationale

**Why is OpenVINO important enough for an IPIP?** Intel ships ~70
million PCs/year. A meaningful fraction now have NPUs (every Lunar
Lake / Arrow Lake / Panther Lake laptop, plus Meteor Lake before
them). Operators with that hardware can't currently participate
without us doing nothing-special tricks like running Ollama on
the iGPU instead of the NPU — which underutilizes the silicon by
~10×. Adding the OpenVINO adapter unlocks that hardware tier.

**Why detect host runtimes instead of always bundling?** Bundling
is what blew up the RunPod overlay disk. Detection follows the
"don't reach for consensus when CRDTs suffice" principle from
IPIP-0020 — match the primitive to the actual requirement. If the
OS already provides CUDA, we don't need a 3 GB Ollama bundle.

**Why does OpenVINO rank below Ollama in auto-select?** Ollama is
the broad-hardware default; OpenVINO is narrower (Intel only).
An operator running OVMS has made an explicit setup choice — but
they probably also have Ollama running for the GPU side. If both
are up, the operator's `INFERNET_ENGINE_BACKEND` env var is the
answer; the auto-select default just picks whichever they're
serving from at the moment.

## Backwards compatibility

Fully additive. Existing engines (Ollama / vLLM / llama.cpp / Mojo
/ stub) unchanged. The detection step in install.sh is non-fatal —
on an OS without the runtimes, install.sh proceeds exactly as
today. Existing daemons continue to work.

The auto-select precedence change (§2) inserts OpenVINO between
llama.cpp and Ollama. Operators currently running Ollama keep
getting Ollama; operators running vLLM keep getting vLLM. The new
slot only applies on hosts where OVMS is running — which is an
explicit setup step the operator took.

## Security considerations

OpenVINO Model Server is an HTTP server — operators expose it on
localhost by default. The same caveats as our Ollama / vLLM /
llama.cpp adapters apply: never bind to 0.0.0.0 without auth.
The `OPENVINO_API_KEY` env var configures OVMS's Bearer-token mode
when external exposure is required.

Host-runtime detection reads from `dpkg`, `command -v`, and
`/proc/cpuinfo`. None of these escalate trust beyond what install.sh
already had — they're observational.

## Privacy considerations

Adding the OpenVINO engine to a heartbeat advertises Intel-NPU /
Intel-GPU presence in the operator's `specs` field. This is the
same disclosure surface as advertising NVIDIA / AMD GPUs today;
operators with privacy needs run with `--no-advertise` (per
IPIP-0016 §8) and skip the heartbeat advertisement entirely.

## Open questions

- **Should `INFERNET_CUDA_HOST=apt` cause us to use llama.cpp
  preferentially over Ollama?** llama.cpp links against host CUDA
  cleanly without a bundle; Ollama still bundles. Worth measuring
  before flipping the default precedence.
- **OVMS model conversion UX.** Operators have to convert PyTorch
  models to OpenVINO IR (`omz_converter` or `optimum-cli export
  openvino`). Should we ship a helper subcommand
  (`infernet model add --backend openvino qwen2.5:7b`) that wraps
  this? Convenient but adds maintenance burden.
- **RISC-V binary availability.** Canonical's CPU-variant pipeline
  is forward-looking. No action today; revisit when a Linux RISC-V
  laptop with a usable inference accelerator actually exists.
- **Snap-managed engine install.** Reserved (§5). Amend when the
  snaps ship and stabilize.
