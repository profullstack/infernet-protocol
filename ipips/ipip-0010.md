## Abstract

Defines four workload classes for jobs scheduled on the Infernet network,
based on where parallelism happens and what latency tolerance the job
implies. The classes drive scheduler routing, advertised provider
capabilities, and the matrix of which engine + training backends apply
to each.

## Motivation

"Run AI on a P2P network" is too broad to design for. The math diverges
sharply between (a) one chat request to one GPU, (b) one big model
sharded across one provider's cluster, (c) one big model sharded across
multiple providers, and (d) training a model jointly with peers. Some
configurations are physically realizable on the public internet; others
are not.

We need a shared vocabulary so:
- the scheduler can advertise which classes a provider supports,
- the routing layer can match jobs to providers correctly,
- the docs/marketing pages can be honest about what works today,
- future IPIPs can reference classes without re-defining them.

## Specification

### Classes

| Class | Pattern | Realtime? | Engines | Training backends | CAP profile |
|---|---|---|---|---|---|
| **A** | One model fits one GPU; one request → one provider | Yes (<100 ms TTFT typical) | `ollama`, `vllm` (single-GPU), `mojo`, `stub` | n/a | AP within session, CP across sessions |
| **B** | One model fits one provider's cluster; sharded via TP+PP intra-LAN | Yes (<200 ms TTFT typical) | `vllm` with `--tensor-parallel-size`, `--pipeline-parallel-size` | `deepspeed`, `openrlhf` | CP inside cluster |
| **B.5** | One model spans multiple providers; pipeline-parallel over WAN | **No** — batch only (1+ s per token) | `petals` (pipeline-parallel inference) | `petals` (fine-tuning) | **CP** — full-coverage routing or `409 no_route` |
| **C** | Distributed training across providers; async delta exchange | n/a (long-running) | n/a | `opendiloco`, future `hivemind` | **AP** — DiLoCo-style delta exchange tolerates partitions by design |
| **Payments** | Escrow + settlement ledger (cross-cutting) | n/a | n/a | n/a | **Strict CP** — Postgres ACID; refuse settlement under partition |

**CAP profile semantics.** Every Infernet subsystem MUST declare which two
of (Consistency, Availability, Partition-tolerance) it provides; the third
is forfeit under partition. The `Payments` row is intentionally separate
from any class — payment writes pass through ACID transactions on the
canonical Postgres regardless of which workload they're settling, so
they're CP even when the workload itself is AP.

### Provider capability advertisement

Providers MUST advertise supported classes in their heartbeat under
`specs.workload_classes` as an array of class IDs (`"A"`, `"B"`, `"B.5"`, `"C"`).

A provider serving on `vllm` with one 80 GB GPU advertises `["A"]`. A
provider with a 4×H100 NVLink box and `vllm --tensor-parallel-size 4`
running advertises `["A", "B"]`. A node joined to a Petals swarm holding
some layers of a 405B model advertises `["B.5"]` (and `["A"]` only if it
also has spare VRAM for self-contained jobs).

### Job class declaration

Jobs MUST declare their target class in `jobs.workload_class`. The
scheduler MUST NOT route a class-X job to a provider whose
`specs.workload_classes` does not include X.

### Latency expectations the routing layer MUST honor

- **Class A and B** are eligible for SSE-streamed real-time chat.
- **Class B.5** jobs MUST be queued as batch (no SSE; result delivered
  on completion via webhook or `jobs/poll`).
- **Class C** jobs are long-running; status surfaces via the training
  protocol (IPIP-0011) `step` / `eval` / `checkpoint` events.

### Out of scope (explicitly NOT supported)

- Tensor-parallel sharding of one model across multiple providers over
  the public internet. The all-reduce frequency makes this physically
  impossible with current networking; pretending otherwise misleads
  operators. We support pipeline-parallel cross-provider (Class B.5) only.

## Rationale

**Why split B and B.5?** Both involve sharding one model. But B uses TP
which needs fast intra-LAN; B.5 uses PP which tolerates WAN. The latency
profiles, eligible engines, and operator intent differ so much they
shouldn't share a class.

**Why is real-time Class B.5 forbidden?** With 50 layers pipelined across
peers at ~50 ms per WAN hop, every token costs ≥2.5 s — no human will
stream tokens at that rate. Allowing class-B.5 to advertise as real-time
would let bad actors collect routed chat jobs they can't actually serve.

**Why bother with classes at all instead of free-form capability flags?**
Free-form leads to drift — provider A says "supports_70b_model=true",
provider B says "vllm_70b_supported=1". Classes force a fixed schema the
scheduler can reason about deterministically.

## Backwards compatibility

Existing providers without `specs.workload_classes` are treated as
Class A only by the scheduler (single-GPU inference, the only thing the
network supports today). Heartbeat schema bump is additive; old daemons
keep working with their reduced capability surface.

## Reference implementation

To land:
- `apps/web/lib/scheduler/route.js` — gate routing by `workload_classes`.
- `apps/cli/lib/register.js` — emit `workload_classes` in heartbeat
  based on detected hardware + configured engines.
- DB migration adding `jobs.workload_class TEXT NOT NULL DEFAULT 'A'`.

## Test plan

- Unit: scheduler selects only Class-X-capable providers for Class-X jobs.
- Integration: a Class-B.5 job submitted while only Class-A providers are
  online queues without 503-ing, surfaces "no eligible providers" after
  the standard timeout.
- Regression: existing Class-A jobs route as before once the migration
  default backfills `'A'`.

## Security considerations

A provider could falsely advertise Class B/B.5/C support to attract jobs
it can't fulfill. The standard reputation mechanism (CPR, IPIP-0007)
penalizes failed jobs the same way regardless of class — bad advertisements
self-correct via dropped reputation and reduced routing weight.

## Privacy considerations

Class advertisement is coarser than current `specs.gpus` (already
public), so no new privacy surface.

## Open questions

- Should Class B providers also advertise the maximum model size their
  cluster can serve, or is "Class B + GPU count + per-GPU VRAM" enough
  for the scheduler to compute it?
- Does Class B.5 need a separate "Petals swarm membership" field so jobs
  for *that specific model* route only to peers in the right swarm?
