## Abstract

Specify how Infernet nodes find each other. Bootstrap from the
control plane (the easy onramp), then transition to peer-to-peer
discovery so the network keeps working when no central server is
reachable. Two complementary mechanisms:

1. **Nostr-based capability advertising** for "who is online and
   serving what" (NIP-78 kind-30078 replaceable events).
2. **libp2p Kademlia DHT** for direct peer-to-peer routing
   (`pubkey → multiaddr`) when relays are down or censored.

Explicitly NOT using BitTorrent mainline DHT, AT protocol, or any
other "easy free network" that doesn't match Infernet routing
semantics.

## Motivation

The control plane (`infernetprotocol.com` or any self-hosted clone)
is the easy onramp. It's also a single point of failure, censorship
target, and centralization wart. Without P2P discovery:

- Take down `infernetprotocol.com` → the network is dark
- A government / ISP can block one domain to kill the whole protocol
- Self-hosted control planes can't easily federate

The "Bitcoin for AI" thesis (per IPIP-0001) requires that **the
network keeps working without our hosted infrastructure**. Bitcoin
nodes find peers via DNS seeds + ongoing peer gossip; Infernet should
do the equivalent with our identity layer.

## Specification

### Bootstrap

A fresh node knows nothing. Its first connection comes from a known
URL. Two configurable seed sources:

```jsonc
// ~/.config/infernet/config.json
{
  "controlPlane": { "url": "https://infernetprotocol.com" },  // existing
  "discovery": {
    "seedNodes": [                                            // NEW
      "https://infernetprotocol.com",
      // operators MAY add other long-running peers here, e.g.:
      // "https://acme-corp-infernet.com",
      // "https://my-private-cp.local"
    ],
    "relays": [
      "wss://relay.infernetprotocol.com",
      "wss://relay.damus.io",
      "wss://relay.nostr.info"
    ]
  }
}
```

Bootstrap flow:

1. CLI starts. Has no live peer list yet.
2. `GET <seedNode>/api/peers?limit=20` for each configured seed
   (in order). Response shape:
   ```jsonc
   {
     "data": [
       {
         "pubkey":     "5d0de683...c84d",
         "multiaddr":  "/ip4/162.250.189.114/tcp/46337",
         "last_seen":  "2026-04-26T16:44:00Z",
         "served_models": ["qwen2.5:7b", "llama-3-8b"]
       },
       ...
     ]
   }
   ```
3. For each returned peer, attempt a libp2p dial. Successful dials
   populate the local Kademlia routing table.
4. Once the routing table has ≥ K peers (default K=8), the node is
   considered "bootstrapped." Further discovery proceeds via DHT
   FIND_NODE and Nostr capability subscriptions; the seed nodes are
   no longer required.

A control plane MAY refuse to serve `/api/peers` (rate limit,
geographic restriction, etc.) — clients fall through to the next
seed. If all seeds fail, the CLI exits with a clear "no bootstrap
peers reachable" error.

### Primary discovery: Nostr capability advertising (NIP-78)

Providers continuously advertise their availability and served
models via **NIP-78 application-specific replaceable events**
(kind 30078):

```jsonc
{
  "kind": 30078,
  "pubkey": "<provider-pubkey-hex>",
  "tags": [
    ["d", "infernet:capability:v1"],          // NIP-78 d-tag, replaces previous
    ["client", "infernet:discover:v1"],
    ["m", "qwen2.5:7b"],                      // models served
    ["m", "llama-3-8b"],
    ["addr", "/ip4/162.250.189.114/tcp/46337"],
    ["coin", "USDC", "solana"],               // payout coin/network
    ["coin", "BTC", "ln"],
    ["price", "0.001"],                       // per-1k-tokens, USD
    ["expires", "1714152000"]                 // unix ts; clients drop stale
  ],
  "content": "",
  "created_at": 1714152000,
  "sig": "<schnorr-sig>"
}
```

Replaceable events (kind 30000–39999, per NIP-33) mean each new
heartbeat overwrites the provider's previous one. Clients
subscribing to `{ "kinds": [30078], "#client": ["infernet:discover:v1"] }`
on configured relays receive the live capability map.

Providers re-publish every 60 seconds (heartbeat-rate-equivalent);
clients drop entries whose `expires` tag is in the past.

### Secondary discovery: libp2p Kademlia DHT

Used when:

- A relay is down / censored — Nostr advertising fails, but DHT
  routes still resolve.
- A client knows a target pubkey already and wants direct connection
  (e.g., re-connecting to a previously-used provider for session
  continuity) without going through any relay.
- Provider-to-provider gossip (peer recommendations, capability
  sharing).

Implementation: `@libp2p/kad-dht` in the daemon. Standard Kademlia
parameters (k=20, alpha=3). The DHT stores `pubkey → multiaddr`
mappings; FIND_NODE queries return the peers closest to a target.

Each node:

1. On startup: connects to bootstrap peers, populates routing table.
2. Periodically (every 5 min): runs FIND_NODE on its own pubkey to
   refresh routes.
3. On request from a client: returns its k-closest peers from the
   routing table.

Result: the network is fully discoverable from any single peer once
bootstrapped. Total dependency on `infernetprotocol.com` after
bootstrap = zero.

### What we are NOT doing

- **BitTorrent mainline DHT.** Wrong key space (infohashes), wrong
  semantics, noisy network. The DHT *algorithm* (Kademlia) is what
  we want; the BitTorrent network is not.
- **AT protocol / `did:plc`.** Bluesky's identity scheme is fine but
  introduces a new dependency. We have Nostr already.
- **mDNS / local-network discovery.** Operators are typically across
  WAN; LAN discovery doesn't help.
- **Centralized "node directory" service** beyond the bootstrap
  step. Once bootstrapped, no central lookup is required.

### Anti-Sybil note

Peer discovery alone does NOT prevent fake nodes from joining the
network and advertising capability they don't have. Sybil resistance
is a separate concern handled by the **CoinPay Reputation Protocol
integration (IPIP-0007)** — provider reputation, signed receipts,
and dispute outcomes determine whether the router prefers a peer.

DHT membership is permissionless by design; reputation is what makes
membership *valuable*.

## Rationale

**Why both Nostr and DHT?** They solve different problems. Nostr is
"who is online and what do they serve right now" (a broadcast
problem). DHT is "given a pubkey, find a route to it" (a lookup
problem). One isn't a substitute for the other.

**Why bootstrap from the control plane and not from a fixed list of
seed multiaddrs (Bitcoin-style)?** Two reasons:

1. The control plane already exists and is the friendly onramp;
   bootstrapping from it is one less thing to maintain.
2. We avoid hardcoded seed IPs in client binaries, which become
   stale and force releases when seeds change.

Operators can still configure `seedNodes` to bypass our control
plane entirely if they want.

**Why NIP-78 instead of inventing a new event kind?** NIP-78 is
specifically designed for application-specific replaceable events.
Reusing the standard means standard Nostr clients (Damus, Amethyst)
can debug / inspect Infernet capability events without us shipping a
custom protocol library.

**Why not just NDN or some research-paper-fancy thing?** Kademlia
has 20 years of production deployment, multiple implementations in
every relevant language, and is what every adjacent project we'd
care to interoperate with uses. Boring is good here.

## Backwards compatibility

- `infernet init` adds `discovery.seedNodes` and `discovery.relays`
  arrays to the config. Existing configs (no `discovery` block)
  default to `[controlPlane.url]` for seeds and a hardcoded relay
  list. No breaking change.
- The `/api/peers` route is a NEW endpoint on the control plane —
  no existing route is altered.
- The kind-30078 capability events are net-new on Nostr; subscribing
  is opt-in.

## Reference implementation

Phased — each phase is its own PR.

**Phase 1 — Bootstrap endpoint.**
`GET /api/peers` on the control plane returns a paginated list of
recently-heartbeat'd providers (with addr + served models). Public,
rate-limited.

**Phase 2 — CLI bootstrap.**
`infernet start` consumes `/api/peers` from configured `seedNodes`
on first launch, populates a local `peers.json` cache.

**Phase 3 — Nostr capability publish + subscribe.**
Providers publish kind-30078 events every 60s. Clients subscribe to
relays via `@nostr/tools` and maintain a live capability map.

**Phase 4 — libp2p Kademlia.**
Add `@libp2p/kad-dht` to the daemon. Bootstrap routing table from
the peer list; periodic refresh; expose FIND_NODE to other peers.

**Phase 5 — Router integration.**
The control plane's job router consumes the same Nostr capability
map (where available) and falls back to its own DB when relays are
down. Browsers see the same data via the existing `/api/overview`
without behavior changes.

## Test plan

1. **Bootstrap from control plane**: fresh CLI install with no
   peers cached, `infernet start` populates a routing table within
   30 seconds.
2. **Survives control-plane outage**: with the network warm,
   `infernetprotocol.com` going down does NOT prevent new clients
   from finding providers (via DHT lookup of known pubkeys) or
   providers from being discovered (via Nostr capability events).
3. **Cross-bootstrap**: a node configured with only an alternate
   seed (`acme-corp-infernet.com`) joins the same network and finds
   peers originally bootstrapped from `infernetprotocol.com`.
4. **Stale entry cleanup**: provider goes offline; its kind-30078
   event has `expires` in the past; clients stop attempting to
   route jobs to it within 60 s.
5. **DHT routing**: given a provider's pubkey, `FIND_NODE` returns
   a multiaddr that successfully dials.
6. **No BitTorrent**: lint check ensures we never accidentally
   import a mainline-DHT bridge.

## Security considerations

- **Eclipse attacks**: an attacker who controls bootstrap seeds
  could feed a node a poisoned peer list. Mitigation: configure
  multiple seeds; verify peer pubkeys match capability events
  fetched from independent relays before routing payments.
- **Signature verification**: every NIP-78 capability event is
  Schnorr-signed by the provider's pubkey. Clients MUST verify
  before adding to capability map.
- **Relay censorship**: a single relay can drop or alter events.
  Mitigation: subscribe to ≥ 2 distinct relays per consumer; trust
  events that appear on multiple relays.
- **DHT spam / DoS**: standard Kademlia rate limits. Optionally
  require a small proof-of-work on FIND_NODE if attacks emerge
  (deferred — most networks don't need it).

## Privacy considerations

- **Capability events broadcast served models + multiaddr publicly.**
  Providers wanting to serve only a specific allowlist of clients
  can use `--no-advertise` (already shipped in IPIP-0001) and
  share their pubkey out-of-band.
- **DHT routing tables can be enumerated.** Anyone with libp2p can
  walk the network and build a map. This is intrinsic to public
  DHTs. Operators wanting unmappable participation should pair with
  Tor outbound (IPIP-0001 v1 in-scope item) and `--no-advertise`.

## Open questions

- **Heartbeat cadence**: 60 s feels right for capability events.
  Faster = more relay traffic; slower = stale routing. Tune with
  real data.
- **Should `/api/peers` accept geographic / latency hints?** Useful
  for routing — clients may want close peers. Probably yes;
  open-ended for v1.x.
- **DHT bootstrap from Nostr events**: could we skip the
  `/api/peers` HTTP call entirely and bootstrap from kind-30078
  events on relays? Saves one HTTP dependency but means the relay
  list is the bootstrap surface. Lean: keep the HTTP path for
  simplicity; treat Nostr as the steady-state.
