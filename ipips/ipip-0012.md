## Abstract

Defines how Infernet routes a single inference request across multiple
providers when no single provider can host the full model — the Class B.5
case from IPIP-0010. Uses pipeline parallelism (each peer holds a contiguous
range of transformer layers; hidden states relay through the chain) via
the Petals protocol. Inference is batch-only at this scale; real-time SSE
chat is explicitly out of scope.

## Motivation

Open models keep getting bigger: Llama-3.1-405B is ~810 GB FP16, and the
trajectory is still upward. The largest commodity-rentable GPU is ~192 GB
(MI300X) or ~80 GB (H100). For models that don't fit on any single
provider's hardware, the protocol either supports federated hosting or
abandons the workload class entirely.

Pipeline-parallel sharding over the public internet has been demonstrated
end-to-end (Petals on BLOOM-176B). Tensor-parallel sharding has not, and
will not — the all-reduce frequency requires <1 ms hops that the public
internet can't provide. So the design is forced: pipeline-parallel for
B.5, tensor-parallel restricted to Class B (intra-provider).

## Specification

### Provider role: layer holder

A provider opting into Class B.5 runs a Petals worker that joins a swarm
identified by the model name. The worker advertises a contiguous range
of transformer layers it has loaded into VRAM. Layer ranges are
inclusive on both ends; e.g. `layers: [12, 23]` means layers 12–23.

Heartbeat additions for B.5 providers:

```json
{
  "specs": {
    "workload_classes": ["A", "B.5"],
    "petals_swarms": [
      {
        "model": "meta-llama/Llama-3.1-405B",
        "layers": [12, 23],
        "swarm_id": "public",
        "initial_peers": "/dnsaddr/bootstrap1.petals.dev/p2p/Qm..."
      }
    ]
  }
}
```

### Job role: relay coordinator

A Class-B.5 inference job carries `model: "<repo/name>"` and the
scheduler MUST NOT route it to a single provider. Instead the scheduler:

1. Finds providers in the named swarm whose collective layer ranges
   cover `[0, num_layers - 1]` of the requested model.
2. If full coverage exists, returns a chain ordering (`layer_owners[]`)
   plus the bootstrap peers.
3. The client (or a dedicated relay node) orchestrates the chain via
   Petals' standard libp2p protocol. Tokens stream through the chain;
   the protocol takes care of fault tolerance via redundant routes.

If full coverage does not exist, the scheduler MUST return `409 Conflict`
with body `{ error: "no_route", missing_layers: [...] }` — never a
partial chain.

### Real-time inference is not supported

The scheduler MUST reject Class B.5 chat-style requests that ask for
SSE streaming (`/api/chat/stream/*`). Class B.5 jobs go through the
batch path: enqueue → poll → result delivered on completion. This is
the only physically achievable mode at WAN latency.

### Fine-tuning over Petals

Petals supports LoRA fine-tuning and full-parameter fine-tuning that
backpropagates gradients through the same chain. The training adapter
(IPIP-0011) `petals` backend uses this path; protocol-level rules are
identical to inference (batch only, full-coverage required).

## Rationale

**Why Petals specifically?** It's the only mature, MIT-licensed
implementation of trustless pipeline-parallel inference for LLMs.
Demonstrated at 176B params, public swarm running, integrates with
HuggingFace Transformers as a drop-in client. Reinventing the chain
protocol when an audited one exists would be a waste.

**Why pipeline-parallel and not tensor-parallel?** Forward-pass through
a transformer layer is hundreds of GB·s of internal compute; the
hidden state crossing layer boundaries is megabytes. So the bandwidth
requirement is millions of times lower at layer boundaries than at
tensor boundaries. Pipeline parallelism is the only sharding shape
where the math survives the public internet.

**Why batch-only?** At even 50 ms per WAN hop and a 50-layer pipeline,
each token costs 2.5 s minimum. SSE chat at that rate is worse than
not having it. Batch jobs (eval, embeddings, async chat, sample
generation for RLHF) tolerate this just fine.

**Why require full-coverage routing instead of partial chains with
fallback?** Partial-chain inference would let the scheduler hand out
"best-effort" jobs that silently fall back to garbage outputs when a
peer drops. Forcing full coverage at job-start time means failures are
loud at the right place (the scheduler) rather than silent at the
worst place (the user's response).

## Backwards compatibility

Additive. Providers that don't opt into B.5 continue advertising only
Class A (and possibly B). Existing Class-A jobs route as before. The
heartbeat schema bump is additive — new optional `specs.petals_swarms`
field — old daemons don't emit it and the scheduler treats them as not
B.5-capable.

## Test plan

- Unit: scheduler full-coverage check rejects partial coverage with
  `409 no_route` and `missing_layers`.
- Integration: spin up three providers with overlapping layer ranges
  for a small (e.g. 4-layer) test model, route a batch job, verify it
  completes through the chain.
- Negative: confirm SSE chat to a B.5 model is rejected at the API
  boundary (`/api/chat/stream/*`).

## Security considerations

A peer in the chain can:
- Drop the request (dealt with by Petals' redundant-route fault tolerance).
- Return garbage logits (mitigated by checksumming at swarm protocol level
  and by reputation; not perfectly mitigated — a malicious provider can
  always return wrong outputs, same threat model as Class A).
- Observe the prompt and intermediate hidden states (REAL privacy concern;
  see Privacy Considerations).

The scheduler is not on the data path — once a chain is selected, the
client and providers communicate directly over libp2p. The control plane
sees only the routing decision and the final result, not the per-hop
traffic.

## Privacy considerations

**This is the biggest open issue with Class B.5.** Pipeline-parallel
inference exposes intermediate hidden states to every peer in the chain.
Hidden states leak more about the prompt than just "compute pass-through"
suggests — they're high-dimensional dense representations of the input.
Research (e.g., DeepLeak, DistilledRecovery) has shown prompt
reconstruction from intermediate activations.

Operators MUST be informed that B.5 jobs leak prompt content to the
chain peers. The dashboard SHOULD display a "Visible to N relay peers"
warning on B.5 job submissions. Future work: investigate whether
homomorphic-style noise injection (DistilledRecovery countermeasures)
can be applied without destroying inference quality, and whether
trusted-execution-environment (TEE) attestation by chain peers is
worth the operator complexity.

For privacy-sensitive prompts, B.5 is the wrong tool — Class A or B with
a single trusted provider is the right tool. The scheduler SHOULD
expose a `privacy: "high"` job hint that disables B.5 routing entirely.

## Open questions

- Is there a smaller-than-Petals protocol worth standardizing for our
  exact subset of needs (no DHT, just scheduler-supplied chain)?
- Do we want to support Petals' "private swarm" mode where a single
  operator with multi-machine LAN runs their own swarm? That's
  technically Class B + B.5 hybrid.
- TEE-attested chain participation as a v2 feature for `privacy: "high"`
  Class B.5 jobs?
