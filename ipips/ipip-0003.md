## Abstract

Define a **two-tier authentication model** for Infernet Protocol that
keeps the daemon/protocol layer Nostr-signed (no central credential)
while giving humans a familiar email/passkey login for the
dashboard/billing/payouts UI. A **pubkey-linking ceremony** bridges
the two so a logged-in user can claim ownership of one or more node
identities and see them all under one dashboard.

This IPIP also locks in the **Nostr relay strategy**: when relays
become wire-load-bearing (per IPIP-0001 deferred items, IPIP-0002
chat), a self-hosted relay at `wss://relay.infernetprotocol.com`
joins a configurable list of public relays — never replacing them.

## Motivation

Two real problems show up the moment we try to onboard non-developer
operators:

1. **Humans need a dashboard.** Operators want to see their nodes,
   request payouts, configure billing, view invoices, and submit
   client jobs. None of that fits in a "sign every request with a
   raw secp256k1 keypair" model. They need a session, a password
   manager, and a recovery flow.
2. **Daemons should NOT use sessions.** Cookie-based session auth
   on machine traffic is a bad idea: cookies leak, rotate, get
   stolen, and break the "no-credential-on-the-node" guarantee
   IPIP-0001 worked hard to ship.

The fix is the same shape every protocol-plus-platform business has
landed on (Stripe API keys vs. dashboard logins, GitHub PAT vs. user
sessions, AWS IAM vs. console SSO): **two tiers, glued together by
an explicit ownership claim.**

## Specification

### Two tiers, one boundary

| Tier | Identity | Auth mechanism | Used by |
| --- | --- | --- | --- |
| **Protocol** | Nostr pubkey (BIP-340) | `X-Infernet-Auth` Schnorr signature (IPIP-0001) | daemons, CLI machine-to-machine, automation |
| **Dashboard** | Supabase Auth user (`auth.users.id`) | Magic-link email / passkey / OAuth, session JWT | humans logging into `infernetprotocol.com` |

**Hard rule:** every API route picks exactly one tier. No route accepts
both. A `/api/v1/node/*` route MUST verify a Schnorr signature; an
`/api/admin/*` route MUST verify a Supabase session JWT. Mixing the
two is a design error.

Public read routes (`/api/overview`, `/chat` POST) are explicitly
"none / IP-rate-limited" and form a third, narrower category.

### URL namespaces

```
/api/v1/node/*       Protocol tier      (Nostr-signed, machines)
/api/admin/*         Dashboard tier     (session JWT, humans)
/api/overview        Public             (rate-limited, no auth)
/api/chat            Public             (rate-limited, no auth — until IPIP-0003 phase 2)
/api/auth/cli/*      CLI device-code    (ephemeral, public; bridges to dashboard tier)
```

### Pubkey-linking ceremony

A `pubkey_links` table relates `auth.users.id` (the dashboard human)
to one or more `pubkey` values (the daemon machines). Linking a
pubkey to an account proves the human controls that pubkey, and lets
the dashboard show "your nodes."

```sql
create table public.pubkey_links (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  pubkey text not null check (length(pubkey) = 64),
  role text not null check (role in ('provider', 'aggregator', 'client')),
  label text,
  created_at timestamptz not null default now(),
  unique (pubkey, role)
);
```

A pubkey can be linked to **at most one** user per role. Re-linking
to a different account requires the previous owner to unlink first
(or fails — the originating user retains the row until they release
it).

#### Wire flow

To link pubkey `P` to user `U`:

1. Logged-in dashboard issues a **link-challenge**:

   ```
   POST /api/admin/pubkey-links/challenge
   { "pubkey": "<hex64>", "role": "provider" }
   →
   { "challenge": "<random-32-bytes-hex>", "expires_at": "<iso8601>" }
   ```

   `challenge` is signed-payload material. Expires in 5 minutes.
   Stored server-side keyed by `(user_id, pubkey, role, challenge)`.

2. The dashboard hands `challenge` to the operator (e.g. via a CLI
   command or copy-to-clipboard widget). The operator runs:

   ```bash
   infernet auth link --challenge <hex> --role provider
   ```

   The CLI uses the local Nostr privkey to produce a Schnorr
   signature over the bytes:

   ```
   "infernet:link:v1\n" + role + "\n" + challenge
   ```

   and POSTs back to the dashboard:

   ```
   POST /api/admin/pubkey-links/confirm
   { "challenge": "<hex>", "signature": "<schnorr-hex>" }
   ```

   Server verifies the Schnorr sig over the canonical bytes against
   the pubkey it issued the challenge for. On success: insert into
   `pubkey_links`, return the row. On failure: 403.

This ceremony has the right properties:

- The pubkey owner proves possession (Schnorr sig over a server-
  chosen challenge — replay-resistant, third-party-resistant).
- The dashboard user proves session (the challenge issuance is
  bearer-token gated — only logged-in users can issue challenges).
- The two are bound: the same `(user_id, pubkey, role)` triple
  has to match on issue and confirm.
- Nothing crosses the protocol/dashboard boundary except this
  one explicit handshake.

### CLI device-code flow

The CLI is not a browser. It can't render a magic-link landing page.
It uses a **device-code flow** (analogous to GitHub's `gh auth
login --web`):

1. `infernet auth login` POSTs `/api/auth/cli/start { hint: "infernet@example.com"? }`.
   Server creates a row in `cli_sessions` with a random `code`,
   returns `{ code, verifyUrl }`. `verifyUrl` is e.g.
   `https://infernetprotocol.com/auth/cli/<code>`.

2. CLI prints the URL, opens it in the user's default browser
   (`xdg-open` / `open`), and starts polling
   `GET /api/auth/cli/<code>` every 2 s.

3. Browser lands on the page, which:
   - If user is signed in → server marks the `cli_sessions` row as
     authorized for that `auth.users.id`, returns a CLI-bound bearer
     token (a short-lived JWT signed with `INFERNET_CLI_SESSION_SECRET`).
   - If user is not signed in → standard Supabase Auth magic-link
     flow finishes, then the same authorization step runs.

4. CLI's poll returns `200 { token, user_id, email }`. CLI stores
   `auth.bearerToken`, `auth.userId`, `auth.email` in
   `~/.config/infernet/config.json` under a new `auth` section.

5. Subsequent calls to `/api/admin/*` from the CLI carry
   `Authorization: Bearer <token>`. Token expiry: 30 days; refresh
   on each successful call (rolling).

```sql
create table public.cli_sessions (
  code text primary key,
  user_id uuid references auth.users(id) on delete cascade,
  authorized_at timestamptz,
  consumed_at timestamptz,
  expires_at timestamptz not null,
  created_at timestamptz not null default now()
);
```

Codes expire 10 minutes after creation. Polled-once-then-deleted to
prevent replay.

### Server `.env` requirements

| Variable | Purpose | Required |
| --- | --- | --- |
| `SUPABASE_URL` | (existing) Supabase project URL | yes |
| `SUPABASE_SERVICE_ROLE_KEY` | (existing) server-side privileged key | yes |
| `NEXT_PUBLIC_SUPABASE_URL` | client-side Supabase URL (browser) | yes (new) |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | client-side anon key (browser) | yes (new) |
| `INFERNET_CLI_SESSION_SECRET` | HMAC secret for CLI device-code JWTs | yes (new) |
| `NEXT_PUBLIC_APP_URL` | (existing) base URL the dashboard serves on | yes |

Email magic-link delivery is configured at the **Supabase project
level**, not in the app `.env`:

- **Self-hosted Supabase**: `supabase/config.toml` `[auth.email]`
- **Supabase cloud**: dashboard → Authentication → Email Templates / SMTP

Nothing in the Infernet codebase wires SMTP directly; we delegate to
Supabase Auth's built-in email plumbing.

### Nostr relay strategy

When Nostr relays become load-bearing (IPIP-0002 chat ships, or the
deferred IPIP-0001 "P2P transport via Nostr relays" begins), the
client SHOULD publish to **multiple relays including a self-hosted
one** at `wss://relay.infernetprotocol.com`:

- Implementation: `strfry` on a $5–20/mo VPS. Migrations zero. No
  blockchain. Supabase / Postgres not required.
- Default relay list in CLI config:
  ```json
  "nostr": {
    "relays": [
      "wss://relay.infernetprotocol.com",
      "wss://relay.damus.io",
      "wss://relay.nostr.info"
    ]
  }
  ```
- Operators MAY override. Removing all relays disables Nostr-event
  features but doesn't break protocol-tier auth (which is local
  signing, not relay publishing).

This stays a design goal until IPIP-0002 ships. There's no rush to
stand up the relay before then — empty relay = no benefit and no harm.

## Rationale

**Why two tiers and not one unified scheme?** Tried-and-true pattern:
machine credentials and human credentials have fundamentally
different requirements. Machines want long-lived, programmatic, easily-
rotated; humans want short-lived, recovery-friendly, MFA-able. One
scheme can't serve both without compromising on one side.

**Why Supabase Auth for the dashboard tier and not a custom system?**
It's already in the stack (`@supabase/supabase-js`). It supports
magic-link, OAuth, passkeys, MFA out of the box. Building our own
auth is the textbook example of work that returns nothing.

**Why a Schnorr-signed challenge for pubkey linking instead of "post
your private key" or similar?** Posting the private key would be
catastrophic — the protocol tier requires that key be secret. Sig
over a server-chosen challenge proves possession without exposure.
Same pattern Nostr uses for NIP-42 (auth to relay), and SSH/PGP have
used for decades.

**Why does the device-code flow exist instead of just "paste a token
from the dashboard"?** Pasting tokens is the kind of UX paper-cut that
made the user push back on workarounds in this very project. The
device-code flow takes ~30 LOC server-side, polls every 2 seconds,
and feels native (`gh auth login --web` is the same).

**Why a self-hosted relay alongside public ones, not instead of?** A
single relay = single point of failure / censorship. Multiple public
relays = no control, can rate-limit or filter at any time. Both =
control plane plus redundancy. Strictly better than either pure
extreme.

## Backwards compatibility

- The protocol tier (`/api/v1/node/*`) is **unchanged** — no daemon
  needs to update.
- The CLI gains a new `auth` section in its config; existing configs
  load fine (the new fields are optional and only matter when a user
  runs `infernet auth login`).
- Routes that are currently unauthenticated reads
  (`/api/overview`, `/api/nodes`, `/api/jobs`) are NOT moved to the
  dashboard tier in this IPIP — that's a separate IPIP (the
  read-route auth blocker from IPIP-0001).
- New route prefixes (`/api/admin/*`, `/api/auth/cli/*`) don't
  conflict with anything existing.

## Reference implementation

Phased — each phase is its own PR(s) referencing IPIP-0003.

**Phase 1 — schema + env.** This IPIP merges with:
- `supabase/migrations/20260426000000_auth_and_pubkey_links.sql` (`pubkey_links`, `cli_sessions`)
- `sample.env` updates for `NEXT_PUBLIC_SUPABASE_*` and `INFERNET_CLI_SESSION_SECRET`

**Phase 2 — dashboard tier.**
- Supabase Auth wired into the Next.js app (magic-link sign-in)
- Session middleware on `/api/admin/*` routes
- Dashboard pages: `/auth/login`, `/auth/cli/<code>`, `/dashboard/*`

**Phase 3 — CLI auth.**
- `infernet auth login` (device-code flow)
- `infernet auth status` / `logout`
- `lib/bearer-fetch.js` parallel to `lib/node-client.js` (signedFetch)
- `auth.*` config persistence

**Phase 4 — pubkey-linking ceremony.**
- `/api/admin/pubkey-links/{challenge,confirm,list,delete}` routes
- `infernet auth link [--role provider]`
- Dashboard UI to issue challenges + see linked pubkeys

**Phase 5 — Nostr relay (deferred).**
- Stand up `wss://relay.infernetprotocol.com` running `strfry`.
- Default relay list in CLI config.
- Triggered by IPIP-0002 (chat) merging — not before.

## Test plan

Per phase:

1. **Schema migration**: `supabase db reset` applies cleanly; tables
   exist; `pubkey_links` unique constraint enforced; `cli_sessions`
   ttl trims via cron.
2. **Dashboard tier**: an unauthenticated `GET /api/admin/me` → 401;
   after magic-link login → 200 with user info; logout clears cookie
   → 401 again.
3. **CLI device flow**: end-to-end `infernet auth login` opens
   browser, finishes magic-link, returns to CLI, saves token; second
   run skips browser (already signed in).
4. **Pubkey linking**: challenge issued for pubkey P → CLI signs →
   confirm → row appears; same pubkey → second user → 403; sign with
   wrong privkey → 403.
5. **Boundary check** (negative): a Bearer token presented to
   `/api/v1/node/*` is rejected; a Schnorr signature on `/api/admin/*`
   is rejected.

## Security considerations

- **Bearer token storage**: stored mode 0600 in
  `~/.config/infernet/config.json`, alongside the Nostr privkey.
  Same compromise blast radius — the privkey was already on the box.
- **Challenge replay**: each challenge is single-use, server-deletes
  on confirm or expiry. No nonces needed beyond the 32-byte
  challenge itself.
- **Role-collusion**: a pubkey provider-linked to user A can NOT be
  client-linked to user B without A unlinking first. Prevents one
  human from masquerading as another's machine.
- **Token leakage**: bearer tokens are short-lived (30-day TTL,
  rotated on every call). A leaked token is invalidated by `infernet
  auth logout` from any device, which deletes the corresponding
  `cli_sessions` row.
- **Boundary discipline**: enforced by middleware. A separate test
  in CI asserts no route accepts both auth schemes (greps for the
  forbidden combination).

## Privacy considerations

- **Linking is permanent and visible to the dashboard server.** The
  control plane learns "user A controls pubkey P." Operators who
  want to keep their compute identity unlinked from their human
  identity should NOT use the dashboard tier — Nostr-signed daemon
  operations work without it. This is opt-in linkage, by design.
- **Email is the human identity in v1.** Future iterations MAY
  support pseudonymous human identities (passkeys without email,
  Nostr-pubkey-as-account). Not in scope for IPIP-0003.
- **Cross-pubkey linking** (the same human linking many provider
  pubkeys) IS observable to the server. That's a feature for
  audit/payout consolidation; operators wanting unlinked pubkeys
  simply don't link them.

## Open questions

- **Passkeys vs magic-link as the default sign-in?** Passkeys are
  better security; magic-link is universally compatible. Probably
  ship both, default to magic-link.
- **Should `infernet auth login` use a localhost listener** (capture
  the redirect locally, no polling) instead of polling? Cleaner UX
  but breaks if the user's localhost isn't reachable from their
  browser (containers, remote VMs). Polling is the universal path.
- **Token format**: Supabase issues its own JWTs already. Use those
  raw, or wrap in a CLI-specific JWT signed with
  `INFERNET_CLI_SESSION_SECRET`? Probably wrap — gives us a
  CLI-revocation mechanism independent of Supabase sessions.
- **Should `/api/v1/node/me` reveal whether the pubkey is linked to
  a human user?** Useful for the CLI to surface "this node is owned
  by you (logged in as anthony@…)". But it leaks linkage to anyone
  who can sign as the pubkey — which is fine, since they ARE the
  owner. Lean yes; reveal only the email of the linked user, only
  to the pubkey owner via the signed `me` call.
