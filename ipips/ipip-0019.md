## Abstract

Specifies first-class GPU cloud deployment via `infernet deploy` for
RunPod, TensorDock, and Lambda Cloud, plus pricing-aware presets
(`cheap` / `balanced` / `reliable` / `production`) that compare
providers and pick the best fit for the operator's price + reliability
constraints. Establishes a common `DeployProvider` base class so new
clouds slot in cleanly, normalizes GPU names across providers, warns
on model-fit issues before paying for hardware, and persists local
deployment state for `list` / `status` / `destroy` lifecycle commands.

## Motivation

Today's `infernet deploy` (per pre-0019) shipped per-provider commands
(`runpod`, `digitalocean`, `vast`) with auth handling but **no
provider comparison and no lifecycle management**. Operators wanting
"cheapest GPU running my model" had to manually list, compare, pick,
deploy, then track destroy commands themselves. That breaks at the
first oh-shit moment ("which RunPod pod was I running?").

Three concrete pain points:

1. **No price-aware selection.** Operator runs `infernet deploy runpod
   --gpu 4090` and gets *some* RunPod pod; whether it's the cheapest
   right now, or whether TensorDock would be 30% cheaper, is invisible.
2. **No lifecycle.** Operators end up with paid resources they can't
   easily inventory or stop. "did I leave a $30/hr pod running?"
   becomes a real question.
3. **No model-fit warnings.** `--model meta-llama/Llama-3.1-70B` on
   a single 24 GB GPU silently boots a pod that will OOM before the
   model loads — paid time, no work done.

## Specification

### 1. DeployProvider base class

Every provider adapter MUST extend `packages/deploy-providers/src/providers/base.js`
and implement (or stub) the methods:

```ts
class DeployProvider {
    constructor(config: { apiKey, env? })

    validateAuth(): Promise<{ ok, accountInfo? }>
    listGpuTypes(): Promise<GpuType[]>
    listRegions(): Promise<Region[]>
    findOffers(req: OfferRequest): Promise<Offer[]>

    createNode(req: CreateRequest): Promise<NodeRecord>
    waitUntilReady(node: NodeRecord, timeoutMs?): Promise<NodeRecord>
    bootstrapNode(node, req): Promise<{ ok, healthEndpoint? }>

    getNode(nodeId: string): Promise<NodeRecord>
    listNodes(): Promise<NodeRecord[]>
    stopNode(nodeId): Promise<{ ok }>
    startNode(nodeId): Promise<{ ok }>
    destroyNode(nodeId): Promise<{ ok }>
    logs(nodeId, opts?): Promise<{ stdout, stderr }>
}
```

Stubbed methods that aren't supported by a provider MUST throw a
`NotSupportedError` with a helpful message — never silently no-op.

### 2. Common types

```ts
interface OfferRequest {
    gpu?: string                 // normalized name, e.g. "4090"
    gpuCount?: number            // default 1
    vramMin?: number             // GB
    region?: string
    maxPricePerHour?: number     // USD
    cpuMin?: number
    ramMin?: number              // GB
    diskSizeGb?: number
}

interface Offer {
    providerId: string           // "runpod" / "tensordock" / "lambda"
    offerId: string              // provider-internal ID
    gpu: { name: string; count: number; vramGb: number }
    cpu?: { cores: number; model?: string }
    ramGb?: number
    diskGb?: number
    region?: string
    pricePerHour: number
    deployStyle: "docker" | "vm-ssh" | "kubernetes"
    available: boolean
    raw?: unknown                // provider-native record for debugging
}

interface NodeRecord {
    id: string                   // our local stable ID (infernet-node-<rand>)
    provider: string
    providerNodeId: string
    name: string
    gpu: string
    gpuCount: number
    vramGb: number
    region?: string
    hourlyPrice: number
    engine: "vllm" | "ollama" | "custom"
    model?: string
    status: "creating" | "running" | "stopped" | "destroyed" | "error"
    createdAt: string
    controlPlaneUrl: string
    endpointUrl?: string
    providerPublicKey?: string   // Nostr pubkey of the registered node
}
```

### 3. GPU name normalization

`packages/deploy-providers/src/gpu-normalize.js` MUST resolve common
aliases to a canonical key for cross-provider matching. Initial set:

```
3090, 4090, a4000, a5000, a6000, l4, l40, l40s,
a100, a100-40gb, a100-80gb, h100, h100-80gb, h200
```

Each canonical name maps to a list of provider-native aliases. The
`findOffers` call in each adapter accepts the canonical name and
internally translates.

### 4. Pricing-aware presets

```js
const PRESETS = {
    cheap:      { price: 0.70, reliability: 0.10, dx: 0.10, availability: 0.10 },
    balanced:   { price: 0.45, reliability: 0.25, dx: 0.20, availability: 0.10 },
    reliable:   { price: 0.25, reliability: 0.45, dx: 0.20, availability: 0.10 },
    production: { price: 0.15, reliability: 0.50, dx: 0.20, availability: 0.15 }
};
```

Provider scoring algorithm (`packages/deploy-providers/src/pricing.js`):

```
score = normalizedPrice    × preset.price
      + reliabilityScore   × preset.reliability
      + dxScore            × preset.dx
      + availabilityScore  × preset.availability
```

Where:
- `normalizedPrice = 1 - (offer.pricePerHour / max(offer prices in candidate set))`
  — lower price = higher score
- `reliabilityScore` = static per-provider base (0..1) curated from
  operator experience; updated based on observed deploy success rate
  over time. Initial values:
  - RunPod: 0.85 (mature, stable)
  - Lambda: 0.90 (enterprise-tier, fewer outages)
  - TensorDock: 0.75 (newer, more variability)
- `dxScore` = static per-provider score (0..1) reflecting API quality,
  Docker support, error-message clarity. Initial values:
  - RunPod: 0.85 (good API, Docker-native)
  - TensorDock: 0.70 (REST API, VM bootstrap)
  - Lambda: 0.80 (clean API, well-documented)
- `availabilityScore` = 1.0 if offer.available, else excluded from
  candidate set. (Effectively a hard filter — the weight only matters
  for partial-availability cases.)

### 5. Cost display

Before any paid resource is created, the CLI MUST print an estimated
cost block and prompt for confirmation (unless `--yes` is passed):

```
Selected provider: RunPod
GPU: RTX 4090 24GB
Region: US
Estimated hourly cost: $0.34/hr
Estimated daily cost:  $8.16/day
Estimated monthly cost: $248.20/month

Continue? [y/N]
```

`--dry-run` prints the comparison table + cost block but does not
create any resource and exits 0.

`--json` emits machine-readable output (no prompts; assumes `--yes`).

### 6. Model-fit warnings

`packages/deploy-providers/src/model-fit.js` MUST estimate VRAM
requirements from the requested model and warn if the chosen GPU
likely can't fit it without quantization:

```
Warning: meta-llama/Llama-3.1-70B-Instruct (~140 GB FP16) may not fit
on RTX 4090 24GB without quantization.

Recommended:
  --quantization awq                    (~35 GB, fits 1× a100-80gb)
  --gpu-count 2                          (sharding via tensor-parallel)
  --gpu a100-80gb                        (single GPU with VRAM headroom)
  --gpu h100                              (with safety margin)
```

VRAM estimates use a static table of known model sizes (Qwen, Llama,
Mistral families). Unknown models warn "VRAM requirement unknown".

Does NOT block deploy unless `--strict-model-fit` is passed.

### 7. Local deployment state

```
$HOME/.infernet/deployments/
  deployments.json     # index by id
  nodes/
    <id>.json          # full NodeRecord per deployment
```

State is updated atomically (write to `<id>.json.tmp`, fsync, rename).
File permissions: `0700` on dirs, `0600` on files. State NEVER
contains API keys.

### 8. Lifecycle commands

```
infernet deploy list
infernet deploy status <node-id>
infernet deploy logs <node-id>
infernet deploy stop <node-id>
infernet deploy start <node-id>
infernet deploy destroy <node-id>
```

`<node-id>` is the local stable ID (e.g. `infernet-node-a82f`),
not the provider's internal ID. The CLI looks up the local
`NodeRecord`, dispatches to the right adapter using
`record.provider`, and updates the local state on success.

`destroy` MUST confirm interactively unless `--yes` is passed. After
successful destroy, the local record is updated with
`status: "destroyed"` (kept for audit, not deleted).

### 9. vLLM bootstrap support

When `--engine vllm` is passed, the adapter's `bootstrapNode` MUST
configure the node to run a vLLM server with:

```
vllm serve $MODEL \
    --host 0.0.0.0 \
    --port $PORT \
    [--tensor-parallel-size $TP] \
    [--max-model-len $LEN] \
    [--gpu-memory-utilization $UTIL] \
    [--quantization $QUANT] \
    [--dtype $DTYPE] \
    [--served-model-name $NAME]
```

After bootstrap, the deployment workflow MUST poll
`<endpoint>/v1/models` until 200 OK or a 5-minute timeout. On success,
the local record is updated with the `endpointUrl` and the node
registers with the control plane (per IPIP-0009 engine adapter).

### 10. Error contract — never leave paid resources unreported

If `bootstrapNode` succeeds but health check fails, OR if
`createNode` succeeds but a later step throws, the CLI MUST:

1. Print the provider-side node ID prominently
2. Print the exact `infernet deploy destroy <id>` command to clean up
3. Save the local record with `status: "error"` and an `error_log`

```
Node was created but bootstrap failed.

Provider:    TensorDock
Provider ID: vm_abc123
Local ID:    infernet-node-a82f

To clean up:
    infernet deploy destroy infernet-node-a82f

To debug:
    infernet deploy logs infernet-node-a82f
```

Never silently leave billable resources running without surfacing
both IDs and the destroy path.

### 11. Security

- API keys NEVER written to local state files.
- API keys NEVER printed in logs or error messages (mask all but
  last 4 chars when surfacing context).
- SSH private keys NEVER copied to remote machines.
- Generated provider keypairs (when applicable) stored under
  `$HOME/.infernet/keys/<provider>/` with `0700` dir / `0600` file.
- Operators on shared boxes can override storage location via
  `INFERNET_DEPLOY_STATE_DIR`.

## Rationale

**Why not roll all three new providers into one IPIP per provider?**
The interface, pricing engine, GPU normalization, model-fit, and
state layer are shared across providers. Splitting per-provider
would multiply the spec surface and make it impossible to reason
about "what does an Infernet deploy look like" without reading three
documents. One IPIP for the architecture; provider-specific quirks
go in the source comments where they actually matter.

**Why explicit `deployStyle` ("docker" | "vm-ssh" | "kubernetes")
instead of inferring from the provider?** RunPod has both Docker
pods *and* community VMs; some TensorDock instances support
cloud-init while others need SSH bootstrap. Per-offer style is more
honest than per-provider.

**Why `cheap` / `balanced` / `reliable` / `production` instead of
exposing weight tuning directly?** Most operators want a one-word
intent, not a vector. The four presets cover the meaningful range;
power users can override with `--price-weight`, `--reliability-weight`,
etc. (out of scope for v1).

**Why is the model-fit warning a warning and not a hard block?**
Quantization options vary fast (new community AWQ/GPTQ checkpoints
ship weekly); a strict block would be wrong as often as right. The
warning surfaces the risk; `--strict-model-fit` enforces blocking
for production / CI use.

## Backwards compatibility

- Existing `infernet deploy <provider>` commands keep working — new
  base class is additive; existing adapters get refactored to
  extend it without changing their public CLI surface.
- Existing local state (none today) is unaffected; lifecycle
  commands operate only on records the new code creates.
- New flags (`--max-price`, `--engine`, `--model`) on existing
  per-provider commands; backwards-compatible.

## Security considerations

API keys are the largest exposure surface. The state file MUST not
contain them; mask in logs; reject any code path that would persist
them. Standard practice already, codified here so it doesn't drift.

## Privacy considerations

Deploy records reveal which clouds the operator uses. Local-only
state, never published. If we ever sync state across devices that
becomes a privacy concern; out of scope for v1.

## Open questions

- **Reliability scores from where?** Initial values are guesses;
  ought to learn from operator-reported deploy success rates over
  time (telemetry opt-in).
- **vLLM-specific flags as positional vs nested object?** Today's
  CLI flat-flag approach (`--quantization`, `--max-model-len`) gets
  unwieldy at 10+ flags. Consider a `--engine-config '{...}'` JSON
  blob for power users.
- **Cost cap enforcement.** Should `infernet deploy` track lifetime
  cost per node and auto-destroy at a budget cap? Useful for "spin
  up for the weekend" experiments. Out of scope for v1 but
  important.
