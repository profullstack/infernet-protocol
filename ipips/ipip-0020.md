## Abstract

Specifies which subsystems of Infernet require **strict consensus**
(total order on any value, achieved across multiple replicas at
real-time-noticeable cost) versus **logical-clock ordering** (already
covered by IPIP-0014/0015) versus **eventually-consistent CRDT merge**
(covered by IPIP-0017). For the cases that genuinely require
consensus, this IPIP picks an algorithm family (Raft for crash-fault
trusted clusters, Tendermint/PBFT for byzantine-fault decentralized
peers) and constrains where each gets used. The default answer is
"don't use consensus" — it's the most expensive primitive in the
agreement spectrum and we get the same outcome cheaper for almost
every workload via the existing IPIPs.

## Motivation

Distributed-systems theory carves "agreement" into three buckets:

| Class | Value | Order | Cost | Where it fits in Infernet today |
|---|---|---|---|---|
| **Clock synchronization** | time / counter | partial / total via Lamport tiebreak | cheap | IPIP-0014 §5 logical clocks; IPIP-0015 sparse vector clocks |
| **Consensus** | any value | total | **expensive** | Implicit via centralized Postgres ACID; **no peer-to-peer consensus today** |
| **Consistency** | any value | flexible (eventual / causal / strong) | cost = f(guarantee) | IPIP-0017 CRDTs (eventual); IPIP-0018 causal broadcast |

We've shipped the cheap and the medium boxes. We have **no spec for
the expensive box** because, today, every place that would need it
is satisfied by Postgres ACID on a single control plane. That's a
correct choice for v1 — Postgres-as-coordinator scales to millions
of writes/sec, has decades of operational tooling, and we don't have
a reason to push consensus to peers.

But several roadmap items push that boundary:

1. **Multi-region control plane HA.** Today the control plane is one
   Postgres host. If we want write-availability during a region
   outage, we need either Postgres synchronous replication (CP-but-
   centralized) or a Raft-replicated state machine.
2. **Class C cross-provider training (IPIP-0011).** OpenDiLoCo /
   Hivemind workers periodically need to agree on which delta version
   becomes the next iterate — currently the coordinator decides. A
   consensus-based coordinator election would let the swarm tolerate
   coordinator failure without manual intervention.
3. **Decentralized reputation slashing.** If we ever want
   peer-to-peer "this provider misbehaved" voting (without a
   trusted central reputation authority), we need byzantine-fault-
   tolerant consensus among voters. CRDTs alone can't reject lies;
   they merge them.
4. **Cross-region payment ledger replication.** If/when payments go
   multi-master for HA, Postgres logical replication or a Raft-
   replicated ledger become the options. CRDT-replicated payment
   state is forbidden (IPIP-0014 §8).

This IPIP names which algorithm family fits each case, picks an
implementation per family, and most importantly **says "don't use
consensus" everywhere it doesn't apply** — premature consensus is
the easy way to make a system slow and brittle.

## Specification

### 1. Decision tree — when does a subsystem need consensus?

Before adding any new replicated state to Infernet, walk this tree:

```
Is the data single-writer (one provider, one user, one daemon)?
    YES → use LWW (IPIP-0017 §2 LWW-Register). Done.
    NO  → continue.

Are concurrent updates "merge correctly without coordination"?
    (counters, observed-add sets, last-write-wins on partition)
    YES → use a CRDT (IPIP-0017 §2). Done.
    NO  → continue.

Do we need a single canonical ordering of operations
    that every replica observes identically AND that decides
    cases where two operations are mutually exclusive
    (e.g. "only one of these two payments settles")?
    YES → consensus required. Continue.
    NO  → causal broadcast suffices (IPIP-0018). Done.

Is the threat model crash-faults only (trusted operators, no
    byzantine attackers)?
    YES → Raft (§2 below).
    NO  → byzantine-fault-tolerant consensus (§3 below).
```

Most replicated state hits the LWW or CRDT branch. The few branches
that fall through to consensus are listed in §4.

### 2. Crash-fault consensus — Raft

For trusted-cluster cases (multi-region control plane HA, Class C
swarm-coordinator election among providers in one operator's
cluster), Infernet uses **Raft**.

Implementation choice: **etcd v3** as the consensus substrate when
we need it. Reasons:
- Battle-tested (Kubernetes' core dependency, MIT licensed).
- Single binary, easy to operate alongside Postgres.
- Native gRPC client; fits our Node.js + Python stack via official
  clients.
- Linearizable reads + leases, the two primitives we'd actually use.

We do NOT roll our own Raft. The lib is mature; reinventing it is a
years-long detour with no business value.

#### 2a. Quorum sizing

Raft requires `2f+1` nodes to tolerate `f` simultaneous failures.

| Failure tolerance | Cluster size | Use case |
|---|---|---|
| 1 failure | 3 nodes | default for any Raft cluster we run |
| 2 failures | 5 nodes | multi-region HA control plane |
| Higher | not used | premature; revisit if real |

Even-numbered cluster sizes are forbidden — they double write cost
without improving fault tolerance.

#### 2b. Performance budget

Raft writes hit ~5-15 ms p50 for cross-region quorums. That's
acceptable for control-plane writes (job assignment, reputation
updates) but **forbidden for payment settlement** — payment writes
stay on the single Postgres ACID path until we have a concrete
multi-region requirement.

Read scaling: serializable reads can hit any node; linearizable reads
go through the leader (cost ~= one extra round-trip). Default to
serializable; opt into linearizable per-call when correctness
demands.

### 3. Byzantine-fault consensus — only if we ever need it

For peer-to-peer voting where a fraction of voters may lie or
collude (decentralized reputation slashing, on-chain-style payment
finality), Raft is insufficient — it assumes honest crash failures.

We do NOT specify a BFT algorithm in v1. Reason: we don't have a
real BFT use case yet. Naming an algorithm we won't build is
spec-rot waiting to happen.

When we do need it, the candidate set is:
- **Tendermint Core** (Apache 2.0, the engine under Cosmos) — well-
  understood, but heavy: each instance is its own validator network.
- **HotStuff** (LibraBFT lineage) — newer, simpler, lower latency
  than classical PBFT. Still no first-class Node.js implementation.
- **PBFT-style custom** — only if one of the above doesn't fit.

The branch point is "first IPIP that proposes a BFT use case" —
that IPIP picks the algorithm and justifies it, this one just
reserves the slot.

### 4. Where consensus IS specified for Infernet

Three concrete branches today:

#### 4a. Multi-region control plane HA (future, not v1)

When we deploy `infernetprotocol.com` in N regions for write
availability under regional outage:
- **Algorithm**: Raft via etcd
- **State machine**: a small slice of the schema — leader-of-record
  for each `provider_id`'s heartbeat, the assigned-job-to-provider
  pointer, the next-job-id sequence
- **NOT in Raft**: the bulk Postgres data (jobs, events, payments)
  — those use Postgres logical replication with a Raft-elected
  primary
- **Failure mode**: if Raft loses quorum, control plane goes
  read-only; existing chat jobs continue (daemons poll outbound)
  but no new jobs assign

#### 4b. Class C swarm coordinator election (when OpenDiLoCo lands)

When N operators jointly run an OpenDiLoCo training swarm:
- **Algorithm**: Raft via embedded library (hashicorp/raft has a
  Go reference; libraft for Python)
- **Elected**: the parameter-server / aggregation-coordinator role
  for the next epoch
- **Failure mode**: if the elected coordinator dies mid-epoch,
  surviving workers re-elect, lose at most one delta-exchange
  cycle (already AP-tolerated by DiLoCo math)

#### 4c. ANY other use case

Forbidden until a future IPIP proposes the use case AND justifies
why CRDTs (IPIP-0017) + causal broadcast (IPIP-0018) are
insufficient.

The default answer to "should I use consensus" is **no**. The
exceptions above are the *only* exceptions. Reviewers MUST reject
PRs that introduce new consensus dependencies without a referencing
IPIP.

### 5. Anti-patterns explicitly forbidden

- **Consensus on every chat job.** A consensus round per job would
  cap network throughput at hundreds of jobs/sec across all of
  Infernet — orders of magnitude below where we want to operate.
  Job assignment uses single-master Postgres ACID + outbound
  daemon polling.
- **Blockchain-style consensus.** Bitcoin / Ethereum-class consensus
  is ~7-30 tx/sec with multi-second finality. Wrong tool for
  inference matchmaking. We use them for nothing.
- **CRDT-ifying state that needs consensus.** If two payments race,
  one MUST lose. CRDT merge would silently keep both — IPIP-0014 §8
  forbids this. Payments stay strict CP.
- **DIY consensus.** Don't roll your own Raft. Don't roll your own
  Paxos. Don't roll your own clock synchronization protocol. The
  literature has decades of subtle bugs found and fixed; we
  consume, we don't reinvent.

## Rationale

**Why is consensus relegated to "future" use cases?** The course's
framing of "cost = f(guarantee)" is exactly right. Consensus is the
strongest agreement primitive AND the most expensive. Most workloads
get the agreement they need from cheaper primitives:
- Time agreement → Lamport (IPIP-0014/0015)
- Eventual agreement → CRDTs (IPIP-0017)
- Causal agreement → CBCAST (IPIP-0018)

Reaching for consensus when one of those would do is the easy way to
add 100x latency and operational complexity. The decision tree (§1)
forces the question.

**Why Raft and not Paxos?** Both work. Raft is simpler to reason
about, has more high-quality open-source implementations, and is
what the operations community actually runs (etcd, Consul, CockroachDB,
TiKV all use Raft variants). Paxos is mathematically more general
but practically equivalent for our needs.

**Why etcd specifically and not embedding a Raft library directly?**
Operating a Raft cluster is non-trivial — log compaction, snapshot
delivery, member changes, leader monitoring. etcd does this
production-quality. Embedding a library means we own all of that
plus the Raft state machine; etcd separates concerns.

**Why no BFT spec today?** We have no BFT use case. Reserving the
slot without filling it is honest about what's needed; filling it
prematurely produces spec that no implementation matches.

## Backwards compatibility

Fully additive. No existing subsystem changes. The decision tree is
documentation + review-time enforcement; the etcd dependency only
enters the system when 4a or 4b actually ships.

## Reference implementation

To land alongside the first concrete consensus use case (4a or 4b):

- `tooling/infra/etcd/` — Docker compose + production
  configuration for the etcd cluster.
- `packages/consensus/` — thin client wrapper over etcd for the
  specific operations we need (election, lease, watch).
- `apps/web/lib/scheduler/leader.js` — multi-region case (4a).
- `packages/training/src/coordinator-election.js` — Class C case (4b).

Tests for each: kill quorum → cluster goes read-only; restore
quorum → cluster resumes; leader fails → re-election within
heartbeat window.

## Test plan

When 4a or 4b ships:
- **Unit**: state machine transitions correct under simulated
  message loss / reorder.
- **Integration**: 3-node etcd cluster; partition into 1+2; the
  2-node partition retains write availability, the 1-node
  partition goes read-only; reunite; logs converge.
- **Chaos**: kill the leader mid-write; verify no committed write
  is lost; verify the new leader picks up where the old one stopped.
- **Performance**: measure p50/p99 write latency under target load;
  flag if it exceeds the budgets in §2b.

## Security considerations

- **Leader-election DoS.** A malicious node in the cluster can
  cause repeated elections by lying about leader liveness. Raft
  assumes crash faults; for byzantine actors switch to §3 BFT
  algorithms. For our trusted-operator clusters (4a, 4b), the
  threat model is "operators don't lie to themselves," which is
  acceptable.
- **Snapshot poisoning.** When new nodes join via snapshot,
  signature-verify the snapshot before installing. etcd handles
  this if cluster membership uses TLS client certs (which we'd
  require).
- **Quorum-size manipulation.** Adding/removing members goes
  through Raft itself; an attacker can't unilaterally shrink the
  cluster.

## Privacy considerations

Consensus state (who's leader, log positions) is operational
metadata; it doesn't contain user content. Standard ops-data
exposure level — same as Postgres replication state today.

## Open questions

- **When exactly do we need 4a (multi-region control plane HA)?**
  Today's single Postgres host is fine for our load; HA is
  premature until we have a concrete uptime requirement (a paying
  customer with an SLA). Track via roadmap, not by speculatively
  building the etcd dependency now.
- **Decentralized reputation slashing.** Real or hypothetical?
  Centralized CPR (IPIP-0007) works fine if operators trust the
  control plane. If they don't (censorship resistance argument),
  BFT consensus among reputation validators becomes the design
  question. Possibly justifies a future IPIP-0021.
- **Spectrum hybrid: Raft + CRDT.** etcd's `mvcc` model is
  Raft-replicated linearizable state; CRDTs are AP. Some systems
  layer them: Raft for control-plane decisions, CRDTs for the bulk
  data those decisions reference. Worth considering when 4a
  matures, not before.
