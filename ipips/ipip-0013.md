## Abstract

Defines a `POST /api/v1/jobs/batch` endpoint that accepts a single
logical batch job (embed N documents, classify N items, summarize N
texts, etc.) and decomposes it into K independent chunks, each routed
to a provider as an ordinary Class-A inference job. Chunks complete in
parallel across the provider pool; results are aggregated into a final
manifest. Backed by **BullMQ** (Redis-backed job queue) for retries,
backpressure, dead-letter, and scheduling. A Postgres-queue fallback
is defined for low-volume / no-Redis deployments behind the same
stable interface.

## Motivation

Chat is our latency-sensitive primary surface, but the largest
addressable inference market is *batch*: embeddings over millions of
documents, bulk classification, dataset cleaning, eval runs, OCR-to-LLM
pipelines, agent rollouts. These workloads share three properties that
make them the ideal P2P workload:

1. **Embarrassingly parallel.** Item N has no dependency on item N-1.
2. **Latency-tolerant.** A 10-second per-item latency that would kill
   chat is fine if 10,000 items finish in 30 minutes total.
3. **High aggregate value.** Per-item revenue is small; per-job
   revenue is large.

Today we route every job synchronously to one provider. A 10,000-doc
embedding submission would either hammer one provider for an hour or
require the operator to hand-shard requests at the client. Neither is
viable.

We need: server-side decomposition into chunks, fan-out across the
provider pool, automatic retries when individual chunks fail, and a
single completion signal when all chunks settle.

## Specification

### Submission endpoint

```
POST /api/v1/jobs/batch
Authorization: Bearer <bearer | nostr-signed envelope>
Content-Type: application/json

{
  "kind": "embed" | "chat" | "classify" | "summarize" | "extract",
  "model": "<model id>",
  "items": [ ... ],            // an array of input units
  "options": {
    "chunk_size": 100,         // optional override; server chooses default
    "max_concurrency": 32,     // optional cap on parallel chunks
    "deadline_at": "RFC3339",  // optional; chunks not done by then are failed
    "webhook_url": "https://...",        // optional completion callback
    "result_destination": "s3://bucket/key/" // optional; default is server-hosted
  }
}
```

The server MUST:
- Validate `items` is an array of length ≥ 1.
- Choose a default `chunk_size` if not provided (see Rationale).
- Insert one row in `jobs_batch` and N rows in `jobs_batch_chunks`
  (one per chunk).
- Enqueue each chunk into the implementation's queue.
- Return `202 Accepted` with `{ job_id, chunk_count, status_url }`.

### Status endpoint

```
GET /api/v1/jobs/batch/{job_id}/status

200 OK
{
  "job_id": "...",
  "kind": "embed",
  "status": "queued" | "in_progress" | "completed" | "failed",
  "chunks": { "total": 100, "queued": 32, "running": 8, "done": 58, "failed": 2 },
  "progress": 0.58,
  "created_at": "...",
  "started_at": "...",
  "completed_at": null,
  "result_manifest_url": null     // populated on completion
}
```

### Result manifest

When all chunks settle (success or failure-after-retries), the server:
1. Builds a manifest mapping each chunk index → its result URL + status.
2. Writes the manifest to `result_destination` (or the server's default bucket).
3. POSTs to `webhook_url` if set (HMAC-signed with the operator's API key).
4. Updates `jobs_batch.status` and `result_manifest_url`.

Manifest shape:

```json
{
  "v": 1,
  "job_id": "...",
  "kind": "embed",
  "model": "...",
  "chunks": [
    { "index": 0, "status": "done",   "result_url": "s3://...", "provider": "<pubkey>" },
    { "index": 1, "status": "failed", "error": "all retries exhausted",   "last_provider": "<pubkey>" }
  ],
  "summary": { "total": 100, "done": 99, "failed": 1, "completed_at": "..." }
}
```

### Per-chunk lifecycle

Each chunk is a row in `jobs_batch_chunks` and a queue job. State machine:

```
queued → running → done
    ↘     ↘
     ↘     failed (transient) → queued (retry)
      ↘                       ↘
       failed_terminal (max retries exhausted)
```

The implementation MUST:
- Cap retries at 3 by default (operator-overridable in `options`).
- Use exponential backoff between retries (1s, 5s, 25s).
- Route each retry to a *different* provider than the prior attempt
  when possible.
- Mark a chunk `failed_terminal` only after all retry attempts fail.
- Allow chunk-level cancellation when the parent batch is cancelled.

### Provider eligibility

Chunks route to providers exactly like Class-A jobs (IPIP-0010):
existing reputation, weight, and `specs.workload_classes` rules apply.
A batch job MAY pin to one provider via `options.provider_pubkey` for
privacy-sensitive corpora.

### Queue implementations

Two implementations MUST share the same `BatchQueue` interface:

```ts
interface BatchQueue {
  enqueue(chunk: ChunkRow): Promise<void>;
  dequeue(workerId: string): Promise<ChunkRow | null>;
  ack(chunkId: string, result: ChunkResult): Promise<void>;
  nack(chunkId: string, err: ChunkError): Promise<void>; // schedules retry
  cancel(jobId: string): Promise<void>;                   // bulk cancel
}
```

Implementations:

1. **BullMQ** (`packages/queue-bullmq`) — **default**. Redis-backed
   (Upstash, Railway Redis, self-hosted). Native support for retries
   with exponential backoff, dead-letter queues, rate limiting, job
   priorities, repeatable cron jobs, and a UI for inspection
   (BullMQ Board / Bull Dashboard). The retry / scheduling semantics
   we need for batch are exactly what BullMQ was built for; no point
   reimplementing them on top of Postgres.
2. **Postgres-queue** (`packages/queue-pg`) — fallback. Uses
   `SELECT … FOR UPDATE SKIP LOCKED` against `jobs_batch_chunks` for
   atomic dequeue plus LISTEN/NOTIFY for worker wake-up. No new
   infrastructure required, but lacks BullMQ's mature scheduling
   features. Intended for low-volume installs or operators who don't
   want to run Redis.

Operators choose at deploy time via `INFERNET_QUEUE_BACKEND=bullmq|pg`.
**Default is `bullmq`**. The control plane fails to start with a clear
"set REDIS_URL or INFERNET_QUEUE_BACKEND=pg" error if neither is
configured.

Required env for BullMQ:
```
REDIS_URL=redis://[user:pass@]host:port[/db]
```

### Database schema

```sql
CREATE TABLE jobs_batch (
  id              uuid PRIMARY KEY,
  user_id         uuid NOT NULL REFERENCES auth.users(id),
  kind            text NOT NULL,
  model           text NOT NULL,
  status          text NOT NULL CHECK (status IN ('queued','in_progress','completed','failed','cancelled')),
  chunk_count     int  NOT NULL,
  options         jsonb NOT NULL DEFAULT '{}'::jsonb,
  result_manifest_url text,
  created_at      timestamptz NOT NULL DEFAULT now(),
  started_at      timestamptz,
  completed_at    timestamptz,
  webhook_url     text,
  webhook_secret  text   -- HMAC key for the completion callback
);

CREATE TABLE jobs_batch_chunks (
  id            uuid PRIMARY KEY,
  job_id        uuid NOT NULL REFERENCES jobs_batch(id) ON DELETE CASCADE,
  chunk_index   int  NOT NULL,
  status        text NOT NULL CHECK (status IN ('queued','running','done','failed','failed_terminal')),
  attempts      int  NOT NULL DEFAULT 0,
  max_attempts  int  NOT NULL DEFAULT 3,
  payload       jsonb NOT NULL,            -- the items in this chunk
  result_url    text,                      -- where the chunk's output lives
  last_error    text,
  last_provider uuid,                      -- providers.id of last attempt
  worker_id     text,                      -- queue worker holding the lease
  visible_at    timestamptz NOT NULL,      -- delays retry visibility
  created_at    timestamptz NOT NULL DEFAULT now(),
  done_at       timestamptz,
  UNIQUE (job_id, chunk_index)
);

CREATE INDEX idx_jobs_batch_chunks_visible
  ON jobs_batch_chunks (status, visible_at)
  WHERE status IN ('queued','failed');
```

## Rationale

**Why decompose server-side, not client-side?** Optimal chunk size is
model-dependent (context window, batch-throughput sweet spot, the
provider's hardware tier). Operators shouldn't have to know "embed
this with chunks of 64 because Qwen has 32k context." Server picks
based on `model`'s known characteristics, with operator override for
expert use.

**Why BullMQ as the default instead of Postgres-queue?** Batch
inference is the workload where queue features actually matter:
exponential-backoff retries that survive worker restarts, rate
limiting per provider, dead-letter queues for forensics, scheduled
jobs (e.g. retry a failed chunk in 5 minutes during off-peak),
mature inspection UIs. BullMQ ships all of these; rolling our own
on top of Postgres re-implements the same surface with more bugs.
The Redis dependency is real ops cost, but hosted Redis (Upstash,
Railway Redis) is a one-click add and the cost is bounded —
unlike the cost of debugging a hand-rolled Postgres queue under
load.

The Postgres-queue fallback exists for low-volume self-hosters who
genuinely don't want Redis. Behind the same `BatchQueue` interface,
so operators can start on `pg` and migrate to `bullmq` when they
need the features.

**Why retry on a different provider?** A single bad provider would
otherwise poison every retry of the same chunk. Routing the retry
elsewhere both surfaces the bad provider's failure to reputation
tracking and gives the chunk a real chance to complete.

**Why webhook + manifest URL instead of inline results?** Manifests
can reference output blobs that are larger than any reasonable HTTP
response (a 10k-doc embedding result is ~30 MB of vectors). The blob
goes to object storage; the manifest is a small JSON pointer.

**Why not stream chunk completions?** The job is designed for
batch consumers — schedulers, ETL pipelines, agent loops — that
poll or wait on a webhook. SSE per-chunk would add complexity
without a real consumer.

## Backwards compatibility

Additive. No existing endpoint changes. New tables (`jobs_batch`,
`jobs_batch_chunks`) live alongside `jobs`. Existing chat/inference
path is unaffected.

The CLI MAY grow `infernet batch` subcommands (submit/status/cancel)
in a follow-up — out of scope for this IPIP.

## Test plan

- **Unit:** chunk decomposition produces N independent units of the
  right size; per-chunk retry counter increments on failure;
  visible_at backoff respects exponential schedule.
- **Integration:** submit a 100-item embed job, verify it lands as
  ~10 chunks, each routes to a provider via the existing scheduler,
  manifest assembles correctly when chunks complete out of order.
- **Failure:** simulated provider drop mid-chunk → chunk retries
  on a different provider; bad-actor provider that always fails
  → chunk hits `failed_terminal` after 3 attempts, batch completes
  with partial success.
- **Queue parity:** the same submission produces equivalent manifests
  regardless of `INFERNET_QUEUE_BACKEND=pg` vs `bullmq`.
- **Webhook:** completion callback fires once, HMAC-signed with the
  operator's secret; replay protection via the standard nonce window.

## Security considerations

A batch job by design exposes its content to multiple providers (one
chunk's worth of items per provider). The operator MUST be informed
of this at submit time:

- Dashboard SHOULD render "this batch will be visible to up to N
  providers" at submission.
- Operators with privacy-sensitive corpora (PII, trade secrets) SHOULD
  use `options.provider_pubkey` to pin to a single trusted provider —
  same threat model as Class A.
- Future work: TEE-attested chunk processing for `privacy: high`
  batches; out of scope for v1.

The batch job's authentication, escrow, and per-chunk reputation
accounting all reuse the existing Class-A primitives — no new
attack surface introduced by the queue or the decomposition itself.

## Privacy considerations

Chunk fan-out is the privacy-relevant new behavior. Two operators
submitting the same batch query would still hit different providers
unless explicitly pinned, which is the same property as Class A
chat — but the *aggregate* (one operator's whole corpus split across
many providers) is harder to reason about for the privacy-conscious
operator. The "Visible to N providers" UI hint is the minimum
mitigation; longer-term, batch-level TEE attestation closes the gap.

The result manifest URL is server-side only; providers see only
their assigned chunk, never the manifest. Aggregation happens at
the control plane.

## Open questions

- **Default chunk size table.** Need a per-model lookup of "good"
  chunk sizes. Punt on this and start with a single hardcoded default
  (e.g. 64 items) plus operator override?
- **Mid-job cancellation semantics.** When a batch is cancelled, do
  in-flight chunks get a kill signal (provider sees `cancel`) or
  do we just stop counting them toward the manifest? Latter is
  cheaper but wastes provider compute.
- **Cross-operator chunk batching.** Two operators submit identical
  embedding requests; could we deduplicate at the chunk level? Big
  efficiency win, real privacy concern (one operator's results
  could leak via cache hits).
- **Webhook delivery guarantees.** At-least-once with HMAC nonce
  for idempotency, or exactly-once with persistent state? At-least-
  once is simpler; consumers handle dedup.
- **Result destination authentication.** If `result_destination` is
  an S3 bucket the operator owns, how do we authenticate without
  storing the operator's S3 keys? Pre-signed URLs uploaded by the
  operator at submit time?
