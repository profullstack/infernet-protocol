## Abstract

Lock in the rule that **all data access goes through the control
plane API**. No direct database client in browsers. No direct
database client in the daemon. No carve-outs — including for
Supabase Auth, which is wrapped in our own server-side routes via
`@supabase/ssr` cookie-based session management.

This makes the storage / auth backend a deployment switch, not a
code branch: Supabase Cloud, self-hosted Supabase, Auth0, Clerk, raw
Postgres + auth shim — all are server-side replacements that the CLI,
SDK, and browser code never witness.

## Motivation

Three separate forces push toward "everything through one API":

1. **Backend portability.** Today we use Supabase Cloud for the
   official `infernetprotocol.com` deployment, and Supabase
   self-hosted is a first-class option for operators who want
   sovereignty. Tomorrow we may swap to Convex, raw Postgres + a
   custom auth layer, or split storage by concern. If browsers /
   daemons / SDK consumers have any direct database client baked in,
   each of those swaps requires a coordinated client redeploy. With
   strict API-only access, every backend change is a server-side
   PR with zero downstream blast radius.

2. **Audit boundary.** Every read and write that matters for
   security (auth events, payment ledger writes, job assignments)
   should flow through one set of routes the team controls and logs.
   Direct database access from clients fragments the audit trail
   across Supabase logs + our logs + client behavior.

3. **Query-shape leakage.** RLS protects rows. It doesn't protect
   query *patterns*. A browser able to issue arbitrary
   `SELECT * FROM …` reveals the schema, indexes, and probes for
   RLS gaps. API routes expose only the shapes we mean to expose.

## Specification

### Actors that use the control plane

The control plane (`infernetprotocol.com` or any self-hosted clone)
serves seven distinct actor types. Each uses exactly one auth tier.

| Actor | Examples | Auth | Access |
| --- | --- | --- | --- |
| **Anonymous visitor** | Random browser, signup page, public chat playground | None (IP-rate-limited) | Marketing pages, `/api/overview` public stats, `/api/chat` rate-limited public submissions |
| **Logged-in human** | Operator viewing their dashboard from the browser | Cookie session via `@supabase/ssr` (IPIP-0003 dashboard tier) | `/api/admin/*` scoped to their `auth.users.id` and linked pubkeys |
| **Daemon node** | `infernet start` on a GPU server polling for jobs | Nostr-signed `X-Infernet-Auth` envelope (IPIP-0001 protocol tier) | `/api/v1/node/*` scoped to their pubkey |
| **CLI client (job submission)** | `infernet chat "..."` / `infernet train ...` | Nostr-signed envelope when registered as a `client` role; otherwise anonymous + rate-limited | `/api/jobs/submit`, `/api/jobs/<id>/events` |
| **External SDK consumer** | App using `@infernetprotocol/sdk-js` programmatically | Bearer JWT from `infernet auth login` (IPIP-0003 phase 3) | `/api/admin/*` scoped to the bearer's user |
| **Webhook sender** | CoinPayPortal posting payment confirmations | HMAC signature with `COINPAYPORTAL_WEBHOOK_SECRET` | `/api/webhooks/coinpayportal` only |
| **Self-host operator** | Someone running their own `infernetprotocol.com` clone | The whole stack is theirs — auth model is identical | They are the platform; this IPIP applies inside their deployment too |

The platform team itself has no privileged client. Internal admin
work goes through the same `/api/admin/*` routes a logged-in
operator uses, scoped to platform-owned accounts.

### Hard rules

#### Rule 1 — No direct database client in browsers

Browser code MUST NOT import `@supabase/supabase-js` (or any future
storage client). Every read and write hits an API route.

This includes Supabase Auth. The `auth.signInWithOtp`,
`auth.exchangeCodeForSession`, `auth.signOut`, etc. methods are
called **server-side only**, behind routes like `/api/auth/login`,
`/api/auth/callback`, `/api/auth/logout`. The browser receives an
HTTP-only cookie set by our server; it never sees a Supabase URL.

The `@supabase/ssr` package is the supported tool for this — it's
designed for exactly this server-cookie-based pattern in Next.js
App Router. The package goes in server components and route
handlers only; no client component imports it.

#### Rule 2 — No direct database client in the daemon

The CLI / daemon code in `apps/cli/` and `apps/daemon/` MUST NOT
import any database client. Already true post-IPIP-0001 (the
`@supabase/supabase-js` dep was removed from `apps/cli/package.json`
in commit `05982b7`); this IPIP forbids re-introducing it.

#### Rule 3 — No direct database client in the SDK

`packages/sdk-js` is published to npm and runs in third-party
applications. It MUST NOT bundle a database client. SDK methods are
thin wrappers over `/api/admin/*` calls authenticated with a bearer
token.

#### Rule 4 — `auth.users` is the only table the auth tier touches directly

Server-side auth routes call Supabase Auth methods (which manipulate
`auth.users` and `auth.refresh_tokens` etc.). They MUST NOT also
write to application tables in the same request. Linking a
just-signed-up user to a profile row, payment-method row, etc., is
done in a separate explicit route call (`/api/admin/onboarding/...`).
This keeps the auth boundary cleanly auditable.

#### Rule 5 — Realtime / SSE goes through our routes

When a feature needs streaming (chat token streams, job event tails,
inbox updates), the streaming endpoint is on our API
(`/api/chat/stream/<jobId>`, `/api/jobs/<id>/events`). Browsers
never connect directly to a Supabase Realtime websocket. SSE over
Next.js is the supported transport; if a feature truly needs
WebSocket semantics, we proxy through a route handler with
`Upgrade` support, not by exposing the upstream URL.

### Self-hosted vs Supabase Cloud

Both are first-class. They are a `.env` choice, not a code branch.

| | Self-hosted | Supabase Cloud |
| --- | --- | --- |
| URL | `http://127.0.0.1:54321` | `https://<ref>.supabase.co` |
| Service-role key | from `supabase status` | from dashboard |
| Anon key (for SSR cookie helpers) | from `supabase status` | from dashboard |
| SMTP for auth emails | `supabase/config.toml` `[auth.email]` | dashboard → Authentication → SMTP |
| Cost | $5–50/mo VPS | $0 free tier → $25/mo Pro |

Code MUST work against both with no conditional branching. CI runs
the test suite against a self-hosted Supabase started by
`supabase start`, validating that the Supabase Cloud → self-hosted
migration is genuinely zero-effort.

### When P2P / Nostr replaces API access

The "P2P transport via Nostr relays" item in IPIP-0001's v1 scope
moves *some* reads off the API. The boundary:

| Surface | API or Nostr? |
| --- | --- |
| Sensitive per-user data (your jobs, your payouts) | API only — never Nostr |
| Public network state (online provider count, recent jobs anonymized) | Both — API for browsers, Nostr for daemon-to-daemon discovery |
| Live job tokens | API/SSE — Nostr is allowed but adds a hop and is not recommended for low-latency streaming |
| Operator chat (IPIP-0002) | Nostr only — never API; this is communication, not platform data |
| Provider availability / job offers | Nostr (via the P2P transport when shipped); API as a fallback / convenience layer |

Sensitive data NEVER moves to Nostr. The thesis is decentralized
*compute*, not "broadcast every operator's payout history to every
relay in the world."

## Rationale

**Why no carve-out for Supabase Auth?** The convenience argument
("browser-native magic-link redirects are easier") is real but
short-term. The cost is a permanent client coupling to one auth
provider. `@supabase/ssr` solves the convenience problem cleanly
with cookie-based server sessions — same DX, full backend
portability. Posting `/api/auth/login` from the browser is one
fetch call.

**Why allow Nostr to bypass the API for some reads?** Decentralization
is the thesis, not "API-everything dogma." Provider availability
and job offers are public-by-design and benefit from peer-to-peer
broadcast — that's exactly what Nostr is good at. But the line is
clear: anything that's per-user / per-account stays on the API.

**Why ban direct DB client in the SDK?** The SDK ships to npm and
gets installed in dozens of third-party apps. Every install that
bundles `@supabase/supabase-js` becomes a forever-coupling — once
even one customer ships a binary depending on Supabase URLs in the
SDK, our backend choice is locked.

**Why is the actor table in the spec?** Because the rules are
contract-level — every reviewer, every PR, every new endpoint should
know exactly which actor it serves and which auth tier applies. An
out-of-table actor is a design smell.

## Backwards compatibility

- Server-side code already follows Rule 1 (no direct DB in
  browsers — the existing `/api/*` routes wrap all DB calls).
  Verifying this is a CI check, not a code change.
- The CLI hasn't imported a DB client since IPIP-0001's
  Phase 1 work (commit `05982b7`). Already compliant with Rule 2.
- The SDK at `packages/sdk-js` has no DB dependency today. Already
  compliant with Rule 3.
- IPIP-0003's existing draft mentioned Supabase Auth as a possible
  carve-out for the dashboard tier. **This IPIP supersedes that
  carve-out**: dashboard auth is server-side cookie based via
  `@supabase/ssr`. No client-side `auth.signIn*` calls.

## Reference implementation

This IPIP is mostly a constraint document. Its enforcement lands as:

**Phase 1 — CI guard.**
A small lint script (`tooling/lint-no-direct-db.mjs`) that scans
client / SDK / CLI source trees for forbidden imports
(`@supabase/supabase-js`, future replacements). Fails the build if
any forbidden import is detected outside of allowed paths
(`apps/web/lib/supabase/server.js` etc.).

**Phase 2 — Refactor any drift.**
Spot-check current code against rules 1–5. Specifically: any
existing `/chat` page or component using Supabase Realtime directly
gets refactored to consume `/api/chat/stream/<jobId>` SSE.

**Phase 3 — `@supabase/ssr` cookie auth.**
Wire up `@supabase/ssr` for cookie-based session management as part
of IPIP-0003 phase 2. Browser POSTs `/api/auth/login`, redirect URL
is `/api/auth/callback`, session lives in an HTTP-only cookie.

**Phase 4 — Self-hosted vs cloud CI matrix.**
Add a CI lane that runs the test suite against a `supabase start`
self-hosted instance, in addition to the existing lane that uses
test fixtures. Validates the deployment-switch promise.

## Test plan

1. **Lint guard** (Phase 1): introducing `import { createClient } from
   "@supabase/supabase-js"` inside `apps/cli/`, `packages/sdk-js/`,
   or any client component (`"use client"`) fails CI with a clear
   pointer to IPIP-0005.
2. **Cookie auth round-trip** (Phase 3): `POST /api/auth/login`
   with an email triggers a magic-link email; clicking it lands on
   `/api/auth/callback`, which sets a cookie and redirects;
   subsequent `/api/admin/me` requests with that cookie return the
   user; `POST /api/auth/logout` clears the cookie.
3. **Self-hosted parity** (Phase 4): the entire test suite passes
   against `supabase start` with no code changes — only `.env`
   differs.
4. **Backend swap rehearsal**: a test deployment swaps
   `SUPABASE_URL` to a fresh project, runs migrations, walks the
   onboarding flow end-to-end. Browsers / daemons / SDK consumers
   pointed at the new control plane URL work without any client
   redeploy.

## Security considerations

- **Cookie configuration**: HTTP-only, `Secure`, `SameSite=Lax` (or
  `Strict` once we're confident no third-party redirect flows need
  cross-origin cookies). Cookie contents = a Supabase session
  reference, not raw user data.
- **CSRF**: cookie-based auth requires CSRF protection on
  state-changing routes. Use double-submit token pattern via Next.js
  middleware or `@supabase/ssr`'s built-in protections.
- **Rate limiting**: anonymous routes (`/api/chat`, `/api/overview`)
  are IP-rate-limited (already shipped); cookie-protected routes
  inherit user-scope limits via `auth.users.id`; signed-envelope
  routes inherit pubkey-scope limits.
- **Audit log**: every state-changing route writes a row to a
  forthcoming `audit_log` table (its own IPIP, deferred). Direct DB
  access from a non-server context would bypass audit; this
  architecture prevents that by construction.

## Privacy considerations

- **Hiding the backend choice from clients** is a real privacy /
  operational benefit: we can move sensitive data off Supabase to
  a different provider in the future without operators noticing,
  and operators who want their compute identity unlinkable from
  their human identity benefit from a single audit boundary.
- **Anonymous public reads** are fine on the API; they're not
  anonymous on the wire — IP shows up in our logs. Operators
  wanting full anonymity for `infernet chat` requests should pair
  with Tor outbound (IPIP-0001 v1 in-scope item).

## Open questions

- **Should we publish a typed REST schema** (OpenAPI, JSON
  Schema, or `packages/api-schema`'s existing format) so external
  developers can generate clients without depending on
  `@infernetprotocol/sdk-js`? Probably yes; it's a v1.x add-on.
- **Server-side caching layer**: now that everything goes through
  the API, we have one place to add Redis / in-memory caching for
  hot reads (provider availability, dashboard stats). Not in scope
  for this IPIP, but the architecture this spec locks in *enables*
  it.
- **Future backend swap candidate**: which non-Supabase backend
  would we actually consider? The answer informs whether
  `@supabase/ssr` is too tight a coupling. Reasonable candidates:
  Convex, Neon + Lucia, or raw Postgres + WorkOS. None require a
  decision now.
